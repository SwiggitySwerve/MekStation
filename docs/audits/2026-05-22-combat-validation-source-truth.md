# Combat Validation Source-Truth Cross-Check

## Verdict

The three-lane validation direction is still right: catalog contracts, behavior-class tests, and representative runner/interactive scenarios. The correction is that catalog presence is not rule truth. A row is only "integrated" when it has both local implementation evidence and an explicit authority boundary.

## Authority Model

- Rulebook authority: Total Warfare / BattleMech Manual style rules define the tabletop mechanic.
- MegaMek source authority: use local MegaMek as executable reference for exact behavior only after reading the relevant source.
- MekHQ behavior authority: use only for campaign/maintenance behavior, not tabletop combat effects.
- MekStation deviation: UI, networking, event ordering, target filters, objectives, and turn-rotation semantics are product contracts layered over tabletop outcomes.

## Gates To Keep

- Do not promote a requirement to integrated if any support reference is helper-only or unsupported.
- Do not use OpenSpec alone as external rule authority; it is the project contract.
- Do not use broad known-limitation filters to certify combat coverage.
- Do not claim non-BattleMech coverage in the BattleMech matrix. Vehicles, aerospace, infantry, battle armor, and ProtoMechs need separate matrices or explicit out-of-scope rows.
- For lifecycle claims, require command/intent/wire coverage, event emission, state mutation, target filtering, and terminal summary evidence.
- Feature support rows that put `source-backed` in their evidence must carry structured `sourceRefs` with pinned rulebook or MegaMek/MekHQ authority links; prose-only authority claims are not enough.
- Helper-only or unsupported combat classes must have a named authority row, an expected simulation scenario, and an explicit reason for exclusion from the BattleMech matrix.
- Physical legality rows must carry commit-pinned MegaMek `sourceRefs`; unsupported rows may stay unsupported, but they cannot stay source-ambiguous.
- Static `WEAPON_DATABASE` subset coverage, synthetic Medium Laser fallback rejection, and variable missile damage-string parsing are first-class validation-scope rows; future weapon PRs should reference those rows instead of hiding behind broad official-catalog scope.
- Validation-scope rows must stay row-source-pinned when knownLimitations bypasses, catalog filter-gate tests, fallback guards, variable-damage parsing, or non-BattleMech split contracts move.
- Non-BattleMech event-scope rows must stay row-source-pinned when vehicle, VTOL, battle armor, swarm, leg-attack, mimetic, or stealth event factories/helpers move; these rows are explicit exclusions from the BattleMech matrix, not BattleMech combat coverage.
- BattleMech event-stream rows must stay row-source-pinned when lifecycle, movement, attack, damage, heat, PSR, physical, objective, morale, retreat, ejection, or unsupported planning event factories, runner phases, session helpers, reducers, or tests move; these rows describe executable MekStation event contracts, not independent tabletop rule parity.
- Canonical SPA scope rows must stay row-source-pinned when the canonical SPA catalog, legacy alias bridge, MegaMek pilot option registry snapshot, or support fallback classifications move; integrated behavior refs alone are not enough to prove the canonical id is in scope.
- Invalid ranged attack no-side-effect guard rows now carry row-level MekStation source references for the exact invalidation exits and event/state mutation boundaries they must not reach: `AttackDeclared`, `AttackResolved`, heat/fired-state accounting, ammo consumption, and damage application. This keeps the guard catalog source-checkable without over-claiming MegaMek parity where MekStation intentionally owns the event-suppression contract.
- Evading-attacker ranged invalidation is source-backed by MegaMek `ComputeToHitIsImpossible` and sprinting-attacker ranged invalidation is source-backed by MegaMek `Compute`: explicit `isEvading: true` attackers emit `AttackInvalid { reason: 'AttackerEvading' }`, and explicit `sprintedThisTurn: true` attackers emit `AttackInvalid { reason: 'AttackerSprinted' }`, both before `AttackDeclared`, heat, ammo, fired-weapon, or damage side effects.
