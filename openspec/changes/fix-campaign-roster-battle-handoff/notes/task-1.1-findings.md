Real API exercise confirmed that `POST /api/forces` creates a Lance with four pre-created assignment rows, slots 1 through 4, all with `pilotId` and `unitId` initially null. Lance capacity is therefore exactly 4 in the exercised API path, matching the campaign wizard's four-unit cap. `PUT /api/forces/assignments/[id]` accepted a combined `{ unitId, pilotId }` body for two different assignment ids, and a subsequent `GET /api/forces/[id]` showed both fields persisted on both slots. No assignment-semantics surprises surfaced; the minor response-shape note is that assignment PUT returns `{ success: true, id }`, not just `{ success: true }`. Canonical unit index entries do not carry BV directly, so the representative-unit guard prewarms BV from the validation report before asserting positive, non-placeholder BV. The shared representative unit list lives at `src/lib/campaign/wizard/representativeUnits.ts` so the roster wizard and later backfill helper can import the same source without depending on component code.
