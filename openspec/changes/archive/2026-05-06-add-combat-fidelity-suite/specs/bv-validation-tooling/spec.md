# BV Validation Tooling (delta)

## ADDED Requirements

### Requirement: BV Validation Report Is the Source for Swarm Catalog Hydration

The combat-fidelity test pyramid SHALL source per-unit Battle Value from `validation-output/bv-validation-report.json` (generated by `scripts/validate-bv.ts`) for the Atlas-anchored hydration path. The report MUST cover Ōēź99% of catalog units (currently 4196 / 4225 = 99.3%). For units missing from the report, the prewarmer MAY fall back to `calculateUnitBV()` from `src/utils/construction/bvAdapter.ts` with the understanding that fallback BV is best-effort and may be 0 for unsupported chassis types (Tripods, ProtoMechs, vehicles).

#### Scenario: Atlas BV resolves from validation report

- **GIVEN** a swarm CLI run with the validation report present at `validation-output/bv-validation-report.json`
- **WHEN** the prewarmer hydrates the catalog
- **THEN** the Atlas AS7-D entry's `bv` field MUST equal the report's `calculatedBV` for `unitId: 'atlas-as7-d'`
- **AND** that value MUST match canonical MUL data within ┬▒1% (the report's >99.8% parity gate)

#### Scenario: Tripod chassis falls back gracefully

- **GIVEN** a Tripod unit not covered by the BV report
- **WHEN** the prewarmer attempts hydration
- **THEN** the Tier 2 fallback (`calculateUnitBV`) MUST be invoked
- **AND** if the fallback returns 0, the entry's `bv` MUST be 0
- **AND** the entry MUST be filtered out by `randomForceGenerator` as ineligible ŌĆö NOT crash the prewarmer

### Requirement: BV Cache Invalidation Tied to Catalog Version

The BV prewarm cache at `.cache/swarm-bv-cache.json` SHALL key on the catalog `index.json`'s `version` AND `totalUnits` fields. When either changes, the cache MUST be invalidated (re-read from validation report + fallback) automatically. Cache hits MUST NOT be returned when version or unit count differs from current catalog state.

#### Scenario: Catalog regen invalidates cache

- **GIVEN** an existing cache with `catalogVersion: '1.0.0'` and `catalogTotalUnits: 4225`
- **AND** the catalog index regenerates with `version: '1.1.0'`
- **WHEN** the prewarmer reads the cache
- **THEN** the cache MUST be treated as a miss
- **AND** rebuild MUST proceed from the BV report
- **AND** a new cache MUST be persisted with `catalogVersion: '1.1.0'`

#### Scenario: Mismatched unit count invalidates cache

- **GIVEN** an existing cache with `catalogTotalUnits: 4225`
- **AND** the catalog has been pruned to `totalUnits: 4200`
- **WHEN** the prewarmer reads the cache
- **THEN** the cache MUST NOT be returned
- **AND** rebuild MUST proceed

### Requirement: Validation Report Path Is Explicit and Overridable

The BV prewarmer SHALL accept an optional `bvReportPath` option to allow tests and alternate environments to point at a different validation report (e.g., a fixture report in `src/__tests__/fixtures/`). The default path MUST be `${process.cwd()}/validation-output/bv-validation-report.json`. The same overridability MUST apply to `cacheFilePath`.

#### Scenario: Test passes a fixture report path

- **GIVEN** a unit test with a fixture BV report at `src/services/encounter/__tests__/fixtures/bv-report-fixture.json`
- **WHEN** the test calls `prewarmCatalogBV(catalog, service, version, { bvReportPath: <fixture> })`
- **THEN** the prewarmer MUST load BV from the fixture, NOT from the production validation-output path
