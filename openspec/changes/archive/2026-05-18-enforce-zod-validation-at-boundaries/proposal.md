# Enforce Zod Validation at I/O Boundaries

## Why

The `validation-patterns` spec already requires boundary validation — "type
guards SHALL only be used at system boundaries" — but the codebase only
*partially* honours it. The equipment-catalog boundary is well-guarded: the
schema-bridge generates Zod schemas (`src/types/contracts/generated/*.zod.ts`)
from JSON Schema and a CI `Schema Bridge` job gates catalog conformance.

Other I/O boundaries are **not** validated. Data crossing them is trusted via
bare `as` casts and structural type assertions, so malformed input fails late,
deep inside domain code, with an unhelpful error — or worse, corrupts state
silently:

- **Persisted Zustand store hydration.** Per-type stores use the `persist`
  middleware backed by `localStorage`. A stale or hand-edited localStorage
  payload hydrates straight into store state with no schema check — an
  app-wide corruption vector.
- **Unit JSON loading.** `public/data/units/**/*.json` is read and cast to
  `IFullUnit`-shaped types. A `unit.zod.ts` schema already exists (generated by
  the schema-bridge) but the runtime loaders do not parse through it.
- **API route bodies.** Next.js API routes (e.g. the replay-library routes)
  accept request bodies and return response payloads with hand-rolled or absent
  validation.

The result is an inconsistent boundary posture: one boundary is rigorously
schema-validated, the rest are not.

## What Changes

- **Audit and enumerate I/O boundaries** — every point where untrusted data
  (filesystem, localStorage, network, user import) enters typed domain code —
  and record the inventory in `design.md`.
- **Parse persisted store state through a Zod schema on hydration.** Each
  per-type store's `persist` config gains a `merge` / migration step that
  validates the rehydrated payload against a Zod schema; an invalid payload
  falls back to the store's default state rather than corrupting it.
- **Parse unit JSON through the existing `unit.zod.ts` schema** at the loader
  boundary, so malformed unit data fails at load with a precise error.
- **Validate API route request bodies** with Zod `safeParse`, returning a
  `400` with the schema error on failure instead of throwing downstream.
- Per the project convention, **infer TypeScript types from the Zod schemas**
  at these boundaries rather than maintaining a parallel hand-written type, and
  use `.optional()` (never `.default()`) for new backward-compatible fields.

## Non-Goals

- **No re-derivation of the equipment-catalog boundary.** The schema-bridge
  already guards it; this change does not touch it.
- **No replacement of internal type guards** that are not at a boundary —
  `validation-patterns` already covers those.
- **No change to the BLK/MTF Python import converters.** Those are an offline
  ETL boundary guarded separately by `schema_gate.py`.
- **No new validation UI.** Boundary failures surface as load errors / `400`
  responses / default-state fallback, not new user-facing screens.

## Affected Specs

- `validation-patterns` — ADDED: a **Zod Schema Validation at I/O Boundaries**
  requirement covering persisted-store hydration, unit JSON loading, and API
  route bodies; types at these boundaries SHALL be inferred from the Zod
  schema.

## Test Strategy

- Infrastructure: exists — Jest + `@swc/jest`; Zod is already a dependency.
- Tests: tests-after — per-boundary tests feeding malformed payloads and
  asserting the failure mode (default-state fallback for stores, precise load
  error for unit JSON, `400` for API routes), plus happy-path round-trips.
- Agent QA: corrupt a localStorage store payload and confirm the app falls back
  to default state instead of crashing or corrupting.
