## ADDED Requirements

### Requirement: Personnel Substrate Architecture (Identity vs Employment Split)

The system SHALL store personnel using a two-substrate split: vault `IPilot` records carry **identity** (combat skills, career XP, abilities, traits, persistent biographical data) and persist across campaigns, while a single per-campaign roster entry type carries **employment** (campaign-scoped contract terms, current state, assignment, statistics, status) and references identity by `pilotId`.

This requirement supersedes the existing `IPerson` / `usePersonnelStore` substrate, which is a half-wired vestigial layer that never gets seeded by production paths. The migration from the legacy substrate to the new one is implemented in the `migrate-personnel-to-roster-employment` change; this requirement formally locks the destination shape so subsequent design discussions, code reviews, and follow-up changes target the agreed substrate.

#### Scenario: Identity is owned by the vault

- **GIVEN** a player character (PC) with vault pilot id `pilot-vault-1`
- **WHEN** the same PC is hired into Campaign A and Campaign B
- **THEN** there SHALL be exactly one vault `IPilot` record with id `pilot-vault-1`
- **AND** combat skills (gunnery, piloting), career XP, abilities, and traits SHALL be readable only from the vault record
- **AND** changes to the vault record (e.g., XP-spend in Campaign A) SHALL be visible to Campaign B without further synchronization
- **AND** no roster entry SHALL duplicate or shadow the vault skill/XP/ability fields

#### Scenario: Employment is owned by the per-campaign roster entry

- **GIVEN** a PC with vault id `pilot-vault-1` hired into Campaign A
- **WHEN** the campaign's salary deduction runs on day-advance
- **THEN** the salary value SHALL be read from the Campaign A roster entry's employment fields
- **AND** the same PC's hire date in Campaign B (a different roster entry) SHALL be independent of Campaign A's
- **AND** wounds, recovery time, kills-this-campaign, missions-this-campaign, and prisoner status SHALL be roster-entry-scoped â€” never written to vault

#### Scenario: PCs persist across campaigns; NPCs are campaign-scoped

- **GIVEN** the campaign personnel roster
- **WHEN** an NPC is generated by the random-pilot generator for Campaign A
- **THEN** the NPC's identity SHALL be carried inline on the roster entry as a frozen statblock
- **AND** there SHALL be no vault `IPilot` row for that NPC
- **AND** the NPC SHALL not appear in any other campaign's roster

- **GIVEN** the campaign personnel roster
- **WHEN** a PC is hired into a campaign
- **THEN** the roster entry SHALL reference the existing vault `IPilot` by id (not inline statblock)
- **AND** the same vault id MAY appear in roster entries across multiple campaigns

#### Scenario: Layer 4 (`ICampaignPilotInstance`) is retired

- **GIVEN** the personnel substrate after `migrate-personnel-to-roster-employment` ships
- **WHEN** any production code path needs a per-campaign pilot record
- **THEN** that code path SHALL use the campaign roster entry type
- **AND** `ICampaignPilotInstance` and `CampaignInstanceService` SHALL not exist in the codebase
- **AND** no API route SHALL reference `/api/campaign-instances/*`

#### Scenario: Architecture decision is documented in source

- **GIVEN** any new code that creates, reads, updates, or deletes personnel data
- **WHEN** an engineer searches for "which personnel record should I use"
- **THEN** there SHALL be a single canonical answer: vault `IPilot` for identity, campaign roster entry for employment
- **AND** the council decision document at `openspec/council-decisions/2026-05-01-personnel-architecture-path.md` SHALL be referenced from the migration change's archive
- **AND** there SHALL NOT be three or more parallel personnel layers in the codebase

### Requirement: Campaign Roster Employment Record Contract

The system SHALL define a single per-campaign roster entry type that holds the employment relationship between a vault `IPilot` (or inline NPC statblock) and a specific campaign. The type's field contract is locked here so the implementation in `migrate-personnel-to-roster-employment` has an unambiguous target.

#### Scenario: Identity reference (PC vs NPC discriminator)

- **GIVEN** a roster entry
- **THEN** the entry SHALL hold exactly one of: a `vaultPilotId: string` (PC case) referencing a vault `IPilot.id`, OR a `statblockData: IPilotStatblock` (NPC case) carrying an inline frozen statblock
- **AND** the two fields SHALL be mutually exclusive (XOR â€” never both set, never both null)
- **AND** the discriminator SHALL be expressible as a TypeScript discriminated union

#### Scenario: Employment fields (campaign-scoped contract terms)

- **GIVEN** a roster entry
- **THEN** the entry SHALL include the following employment fields:
  - `hireDate: Date` â€” when this character was hired into THIS campaign
  - `salary: Money | null` â€” campaign-specific salary override (null = use vault/role default)
  - `lifestyle: LifestyleTier | null` â€” for food/housing tax calculation (null = default)
  - `contractTerms: IContractTerms | null` â€” duration, role, restrictions (null = at-will)

#### Scenario: Current campaign state fields

- **GIVEN** a roster entry
- **THEN** the entry SHALL include:
  - `wounds: number` (0-6 for MechWarriors)
  - `recoveryTime: number` â€” mission cycles remaining until back to Active
  - `fatigue: number` â€” accumulated stress (0+)
  - `status: CampaignPilotStatus` â€” Active / Wounded / MIA / Captured / KIA / Departed (campaign-scoped)
  - `prisonerStatus: PrisonerStatus | null` â€” when status is Captured or holding prisoners

#### Scenario: Campaign-scoped statistics

- **GIVEN** a roster entry
- **THEN** the entry SHALL include:
  - `campaignXpEarned: number` â€” XP earned in THIS campaign (vault stores total XP across all campaigns)
  - `campaignKills: number`
  - `campaignMissions: number`
- **AND** these fields SHALL never be written to vault `IPilot.career` directly â€” the vault's career fields aggregate across all campaigns

#### Scenario: Assignment and training fields

- **GIVEN** a roster entry
- **THEN** the entry SHALL include:
  - `assignedUnitId: string | null` â€” current Force assignment (mirrors `IAssignment.unitId`, may be null)
  - `trainingInProgress: IVocationalTraining | null` â€” current training course with start-date and remaining days

#### Scenario: Identity fields are NOT duplicated on the roster entry

- **GIVEN** a roster entry
- **THEN** the entry SHALL NOT include skills (gunnery, piloting), abilities, traits, total career XP, total mission count, or biographical data â€” those fields live on vault `IPilot` for PC entries
- **AND** for NPC entries, those fields are accessible only via the inline `statblockData` field, never as direct properties on the roster entry
- **AND** any code that needs identity data SHALL perform a vault join (PC) or read `statblockData` (NPC); a roster entry alone is insufficient to answer "what is this character's gunnery skill"

#### Scenario: Roster entry persistence

- **GIVEN** a campaign with N roster entries
- **WHEN** the campaign is saved and reloaded
- **THEN** all roster-entry employment, state, statistics, assignment, and training fields SHALL round-trip without loss
- **AND** PC entries SHALL be re-joined to vault on load (a stale `vaultPilotId` referencing a deleted vault pilot SHALL surface a clear "pilot not in vault" UI state, not a crash)
- **AND** NPC entries SHALL load the inline `statblockData` directly without any external dependency
