# Design — Shipped Engine Spec Coverage

## Architecture Decisions

### D1 — Spec-only, no code changes

**Choice**: This change adds requirements + scenarios to source-of-truth specs ONLY. No file in `src/` is modified.

**Rationale**: The behavior already exists in main (PR #514 + archived swarm-harness work). The gap is documentation — source-of-truth specs do not reflect what the code does. Coupling code changes into a retrofit risks scope creep and confuses what's being retrofitted vs newly introduced.

**Trade-off**: A future PR that finds a behavior bug in PR #514 will need to author its own change. That's correct — this change is the retrofit, that change would be the fix.

---

### D2 — Two-tier BV prewarmer is canonical

**Choice**: The `quick-session` delta specs the two-tier resolution pattern (validation report → calculator fallback) as the canonical CLI-side BV source. We do NOT spec the validation report as authoritative across all consumers — only swarm CLI runner.

**Rationale**: Other consumers (the browser-side construction tool, the BV validator itself, the BV unit tests) MUST continue to compute BV via `calculateUnitBV()` directly. Speccing the validation report as canonical for ALL consumers would create a chicken-and-egg dependency (the report is generated by `validate-bv.ts` which itself uses `calculateUnitBV`).

The CLI swarm runner is a special case because it loads 4225 units up-front, the validation report covers >99% of them at >99.8% MUL parity, and per-unit recomputation is wasteful when a report exists. The fallback path keeps the runner correct for units the report misses.

---

### D3 — Acknowledge the determinism gap, don't fix it

**Choice**: Spec the engine's determinism contract as: same-seed runs produce identical event logs UP TO some turn N; beyond N the engine MAY diverge by ≤1 event per 300, and a follow-on change MUST close the residual.

**Rationale**: PR #514 raised `MAX_TURNS` to 100 because the lower ceiling masked real combat. Doing so exposed a pre-existing engine non-determinism. The honest move is to acknowledge it in the spec rather than silently tolerate or pretend it's fixed.

The follow-on `add-engine-determinism-audit` (named in `add-combat-fidelity-suite` tasks.md) is the dedicated forward fix. This retrofit just makes the gap formally visible.

---

### D4 — `Math.random` audit guard is a CI requirement, not a code style rule

**Choice**: Spec the audit as a CI gate (grep + fail) rather than relying on ESLint rules or code review.

**Rationale**: A code-review-only enforcement leaks regressions when reviewers miss the substring. A grep-based CI gate is mechanical and unambiguous. The guard is already implemented as part of `add-combat-fidelity-suite` P0; this retrofit just elevates it to a spec requirement so the gate cannot be silently removed.

**Trade-off**: A legitimate `Math.random` usage outside the audit scope (e.g., a UI sparkle effect) is unaffected. The grep is scoped to `src/utils/gameplay/` and `src/simulation/` — the only places combat determinism matters.

---

### D5 — Source-of-truth specs are the only target

**Choice**: This change writes deltas under `openspec/changes/add-shipped-engine-spec-coverage/specs/<capability>/` that target the same source-of-truth specs (`openspec/specs/<capability>/`) as the original PRs would have.

**Rationale**: OpenSpec's workflow is: deltas in changes → archive merges deltas into source-of-truth. By following the standard pattern (rather than directly editing source-of-truth), the retrofit is auditable in the same archive timeline as forward-going work.
