# Proposal: add-viewport-layout-sweep

## Why

**W5 of the Seam-First Ladder** (council decision `openspec/council-decisions/2026-07-09-ui-audit-testability-architecture.md`): the suite pays for viewport coverage in the most expensive currency available — whole-suite duplication — and gets almost no layout assurance back.

**The duplication is real and verified on this tree (`claude/viewport-layout-sweep@428f63aa9`, `npx playwright test --list`):** a bare local run schedules **2,984 test executions** across the five default projects — 712 under `chromium`, 697 each under `Mobile Chrome` (375×667), `Tablet Portrait` (768×1024), and `Tablet Landscape` (1024×768), plus 181 under `smoke` (a `grep /@smoke/` re-run of tests already scheduled elsewhere). 2,984 = 712 + 697×3 + 181 exactly: **2,272 of 2,984 executions (~76%) are duplicate re-runs of desktop-authored assertions**, all contending for one shared webServer and one shared SQLite file. The council's Librarian found no surveyed OSS repo that runs an identical suite across N viewport projects — the ecosystem norm is tag-then-scope (grep/grepInvert per project) with viewport as a *parameter dimension* (Chromatic modes), and Momus killed naive viewport×journey multiplication outright. Deleting the duplication cuts local full-suite cost ~75% (the council's second-order note), because CI never paid it in the first place: **no CI workflow or scripted runner references the responsive or smoke projects by name** — `pr-checks.yml:521` and `nightly-validation.yml:171` pin `--project=chromium`, every `qc:*`/`verify:qc:*` package script pins `--project=chromium`, and `scripts/qc/run-flow-audit.mjs` pins `--project=flow-audit`. The only surface a rationalization changes is the unflagged local `npx playwright test` / `npm run test:e2e` run.

**And the duplication buys almost nothing, because the responsive projects assert no layout geometry.** They re-run functional assertions (roles, testids, text) at narrower widths; the only geometry assertions in the whole suite are spec-local one-offs: `expectNoHorizontalOverflow`/`expectClickable` inside `e2e/coop-campaign-ui-audit.spec.ts:215-276` (co-op screens only), `expectNonBlankRender` inside `e2e/tactical-map-visual-smoke.spec.ts:187-225` (hardcoded to the hex-map container), and a 44×44px touch-target check in `mobile-navigation.spec.ts` (five routes). A screen that overflows horizontally at 375px, renders overlapping controls at 768px, or paints a blank canvas at 1024px passes today on the other ~60 screens. **Layout regressions below desktop are structurally uncaught** while 2,272 duplicate executions burn the local budget.

**Council-sketch drift, resolved here (verified facts override the sketch):** (1) the council's "707 × 4 ≈ 3008" figure is today 712 + 697×3 = 2,803 across the four viewport-duplicate projects (plus 181 smoke re-runs); (2) the "~44 standalone + pack-seeded screens" estimate refines to a verified inventory of **69 routable page files** = 28 standalone-reachable + 2 recovery edge-cases + 3 static-catalog detail routes + 18 pack-coverable route patterns + 13 excluded state-dependent patterns (12 with no covering pilot pack, plus the pilot detail route whose coverage no W4 contract promises — design D10) + 5 dev-only harness pages (`e2e/app-shell-route-manifest.json` arithmetic: 28+2+34+5 = 69, cross-checked against `find src/pages`); (3) the council's "BREAKPOINTS (375/768/1024/1280)" is **not** the `BREAKPOINTS` export in `src/constants/layout.ts` (SM:640/MD:768/LG:1024/XL:1280/XXL:1536 — no 375 key): the sweep binds 768/1024/1280 to `BREAKPOINTS.MD/LG/XL` by import and documents 375 as a deliberate below-SM device-width literal, exactly the precedent `e2e/flow-audits.spec.ts:82-94` already established; (4) no canonical screen-inventory artifact exists in the repo — this change produces one as a checked-in, guard-tested module; (5) no mechanical "W4-impl gate" runtime pattern exists in `e2e/` — and a runtime `test.skip`-on-missing-dependency would violate this capability's own "Conditional skip-on-missing-data is forbidden" scenario, so the W4 gate is an authoring-time STOP gate (the W2-task-4.0 ladder precedent), not a runtime skip.

**Dependency reality:** W4 (`add-scenario-packs`) exists on this tree as a declarative change folder only — `e2e/helpers/scenarioPackLoading.ts`, `e2e/scenario-packs/`, and `src/lib/scenarioPacks/` do not exist (implementation is in flight on a parallel branch). Every sweep entry that consumes `loadCampaignPack`/`loadEncounterPack` or a pilot pack sits behind an explicit STOP-on-absence gate check (design D7). W4 is also writing into the same `playwright.config.ts` project blocks this change rationalizes (its task 3.4 adds `'**/scenario-packs/**'` `testIgnore` globs to the three responsive projects) — a named same-file sequencing risk (design R1), reconciled at rebase time.

## What Changes

- **Shared layout helper module** (`e2e/helpers/layout.ts`, new): promote `expectNoHorizontalOverflow` and `expectClickable` (from `coop-campaign-ui-audit.spec.ts`, near-verbatim) and `expectNonBlankRender` (from `tactical-map-visual-smoke.spec.ts`, with its signature generalized to accept a target locator — today it hardcodes `getByTestId('hex-map-container')` — and its `measureRenderedContrast` pixel-analysis dependency moving with it); add the net-new `expectNoOverlap` (AABB bounding-box intersection among declared sibling check targets, ancestor-descendant containment excluded, 1px default tolerance, explicit reviewable opt-out for overlap-by-design elements — no prior art anywhere in the repo). The two source specs switch to importing the shared helpers; a helper self-test spec proves true-positive/false-positive behavior against synthetic `page.setContent` fixtures.
- **Checked-in screen inventory** (`e2e/layout-sweep/screenInventory.ts`, new): derived from `e2e/app-shell-route-manifest.json` (the existing route source of truth — never a forked route list), classifying every route as swept-standalone, swept-static-catalog, pack-seeded (W4-gated), or excluded-with-documented-reason; a guard test fails on any unclassified route so the inventory can never silently rot as routes are added.
- **One viewport sweep spec** (`e2e/layout-sweep/viewport-layout-sweep.spec.ts`, new): scheduled under the **chromium project only**, one test per swept screen, looping `page.setViewportSize` over the four sweep viewports (375 literal + `BREAKPOINTS.MD`/`LG`/`XL`) and asserting the layout invariants per screen×viewport against **per-viewport-applicable check targets** — responsive collapse the app performs by design (e.g. the top bar's three breakpoint-gated nav variants) is declared as inventory applicability data, never quarantined and never runtime-filtered; every swept screen declares an applicable primary affordance at every sweep width (design D9). Standalone + static-catalog screens sweep now; pack-seeded screens land in a gated follow-on group that loads state exclusively through W4's front-door loaders — one navigation-pack load amortized across the 17 campaign subroutes (16 by direct goto against the stamped campaign id read from the loader's post-navigation URL, the mission-launch briefing screen by front-door in-page discovery from the missions screen), the combat pack for the game-session screen; the pilot detail route is excluded-with-documented-reason until a contract covers it (design D10). Pre-existing violations found at first authoring are quarantined per screen×viewport with a documented reason and follow-up reference — never silently skipped, never blocking the change on unbounded layout-fix scope.
- **Project-matrix rationalization** (`playwright.config.ts`): delete the `Mobile Chrome`, `Tablet Portrait`, and `Tablet Landscape` projects (every touch-dependent block in the repo already self-configures via `test.use({ viewport, hasTouch, isMobile })` or its own `browser.newContext(...)` — verified across all 7 files referencing `hasTouch`/`isMobile`/`tap` — so those specs execute with identical semantics under `chromium`); delete the never-invoked `smoke` project (zero `--project=smoke` references repo-wide; its 181 scheduled tests are pure re-runs — the `@smoke` tag stays and remains greppable via `--grep @smoke`); add a thin tag-scoped **`mobile-touch`** project (`grep /@mobile-touch/`, 375×667 + `hasTouch` + `isMobile`) as the council-mandated home for specs that want project-level touch emulation, with the existing touch-interaction describes opting in by tag. The rationalization ships with **mechanical proofs**: before/after `npx playwright test --list` count tables and a CI/script grep proof that no workflow's or runner's effective selection changed (design D2 — a silent CI-selection change is the named failure mode).
- **Local lane**: a `verify:qc:viewport-sweep` package script (the `verify:qc:app-shell` shape — chromium project, `--workers=1`), not wired into `verify:qc`'s chain or any CI workflow. The sweep joins the nightly full run implicitly (nightly runs `--project=chromium`, which discovers the new spec) — no workflow edits.
- **Ledger**: `openspec/active-change-ledger.json` gains this change's entry (ships with this declarative PR, per the W1–W4 precedent).

## Capabilities

### Modified Capabilities

- `e2e-testing`: ADDED — Shared Layout Assertion Helpers; Viewport Layout Sweep; Playwright Project Topology — Tag-Then-Scope. MODIFIED — Deep-Play Playwright Project Scoping (its scenario text names the three responsive projects being deleted; the obligation is restated topology-agnostically with the same force). This capability already owns the suite-shape/infra requirements (project scoping, seeded-fixture conventions, coverage floors) — W5's content is journey-agnostic suite infrastructure, squarely inside its charter. The council ladder marks only W3/W4 "(NEW spec)"; W5 deliberately mints no new capability.
- `journey-qc`: MODIFIED — Seam Trust Anchor Journeys. Its requirement text and "Anchors are excluded from viewport multiplication" scenario name `Mobile Chrome`/`Tablet Portrait`/`Tablet Landscape` and the `testIgnore` mechanism; both are restated topology-agnostically (anchors scheduled under desktop chromium only, under any project topology) so the spec cannot go stale against the rationalized config. The anchors' authority, isolation, and stability obligations are untouched.

Not chosen as homes: `flow-audit-routines` is doctrinally disqualified (its own "Flow Audits Are Evidence, Never CI Gates" requirement — the sweep is blocking assertions by design); a new `viewport-sweep` capability would split suite-shape ownership that `e2e-testing` already holds (design D1).

## Impact

- **New code** (implementation phase, all additive test infrastructure): `e2e/helpers/layout.ts`, `e2e/layout-sweep/` (inventory module + guard test + sweep spec + helper self-test spec), one `verify:qc:viewport-sweep` package script, one ledger entry.
- **Modified test code**: `playwright.config.ts` (project matrix rewrite), `e2e/coop-campaign-ui-audit.spec.ts` + `e2e/tactical-map-visual-smoke.spec.ts` (import the promoted helpers, delete local copies — behavior-identical), `e2e/mobile-navigation.spec.ts` and any other touch-describe opting into `@mobile-touch` (title tag only), comment-level references to the deleted project names (e.g. `e2e/flow-audits.spec.ts`'s project-list comment).
- **Modified production code: none.** No `src/` file changes; `src/constants/layout.ts` is imported, never edited. Overlap opt-outs are inventory-side allowlist entries, not production attributes — this is the ladder's second zero-production-touch wave.
- **Ordering**: groups 1–4 (helpers, inventory, standalone sweep, rationalization) depend only on merged surface and can implement immediately after this authoring PR; group 5 (pack-seeded sweep entries) gates on the W4 implementation with a STOP-on-absence check. If W4's implementation merges first, its three `testIgnore` additions land in project blocks this change deletes — the rebase resolution is recorded in design R1.
- **CI wall-clock**: the sweep adds one chromium spec (~33 standalone screen tests now, ~51 after the pack group) to the nightly full run — bounded minutes against a 180-minute job timeout (design D8's budget table). PR lanes are untouched.
- **Risk**: the riskiest edit is the project-matrix rewrite; it is fenced by design D2's consumer-impact inventory and the tasks' mechanical before/after proofs. Everything else is additive.

## Non-goals

- **No screenshot-diff per screen×width.** The sweep asserts DOM-geometry invariants, never pixel fidelity — visual baselines are win32-only and nightly already runs `--ignore-snapshots`; per-screen×width screenshot baselines would be red-by-construction in CI and were explicitly excluded by the council row.
- **No PR-blocking gate for the sweep initially.** The sweep runs in the nightly full suite (implicitly, via chromium discovery) and the local `verify:qc:viewport-sweep` lane. No `pr-checks.yml` edits, no required checks, no branch-protection changes. Promoting the sweep into a PR gate is a future explicit delta to the Viewport Layout Sweep requirement, never a quiet CI edit.
- **No W6 scope.** No subsystem facets on the journey-qc registry, no subsystem CI lanes, no audit-spec tagging beyond `@mobile-touch`, no PR-gate wiring.
- **Pack-gated entries are explicit.** No sweep entry consumes a scenario-pack loader before the W4 implementation gate check passes; a pack-seeded screen the gate blocks simply stays un-swept (and classified as pack-seeded in the inventory) — never faked via store injection or hand-seeded state.
- **No production behavior changes.** No `src/` edits, no new testids, no layout fixes smuggled in: violations the sweep finds on first authoring are quarantined with documented follow-ups (design D5); only trivially-safe fixes inside already-modified test files are in scope.
- **No changes to the seven-journey catalog.** `requiredJourneyIds` in `scripts/qc/journey-qc-core.mjs` stays exactly seven; the sweep spec, helper module, and inventory are Playwright/test artifacts, never catalog journeys (the W2/W4 disclaimer applies verbatim).
- **No edits to the in-flight W4 change folder.** `add-scenario-packs`' own spec delta and its "responsive projects" scenario wording are W4's to reconcile at its ship/archive time (the deleted-projects wording becomes vacuously true; see Open Questions).
- **No new test runner, no fourth framework, no per-worker DATABASE_PATH keying.** The sweep is a plain Playwright spec on the existing harness; the shared-SQLite escalation stays the council's separate open risk.
