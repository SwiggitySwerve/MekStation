# Proposal: add-scenario-packs

## Why

**W4 of the Seam-First Ladder** (council decision `openspec/council-decisions/2026-07-09-ui-audit-testability-architecture.md`): time-to-state is the bottleneck for exercising campaign and encounter subsystems in the browser — every journey clicks through setup, and nothing today jumps a test to "mid-campaign, contract paid, repair bay busy" or "turn N, mid-battle, mirrored roster" as a named, reusable start state. Scenario packs close that gap **through the front door**: a campaign pack is PUT to the production campaigns route and then reached by `page.goto`, riding the campaign shell's auto-GET production load path (`useCampaignRouteLoader` fires `loadCampaign(routeCampaignId)` on mount, `src/pages-modules/gameplay/campaigns/campaignPageShell.tsx:150` → `loadCampaignAction` → migrate → deserialize → `writeLiveCampaign` → `restoreRosterProjection`, `src/stores/campaign/useCampaignPersistenceStore.ts:441-483`); an encounter pack is seeded into the browser IndexedDB match log and then cold-navigated, riding the exact recovery factory chain the recovery trust anchor exercises. Per the council's Phase 3 doctrine: **packs are caches; genesis journeys are the source of truth** — generator-minted (never hand-authored), front-door loaded (never store injection for anything with a mount seam), zod-validated at load (fail-loud), version-pinned, and CI re-captured-and-diffed so drift is a loud failure, not silent rot.

The council's recorded wounds are real and verified on this base, and this change's design defends against each:

- **The campaign PUT envelope guard is 4 shallow checks.** `isValidPutBody` (`src/pages/api/campaigns/[id].ts:51-69`) verifies only that the body is an object, `baseVersion` is a number ≥ 0, `envelope.campaignId` is a string, `envelope.schemaVersion` is a number, and `envelope.body` is a non-null object — its own doc comment says "Phase 1 trusts the caller for the deep envelope shape". On the read side, `CampaignPersistenceService` round-trips the payload through unchecked `JSON.parse(row.payload) as SerializedCampaign` casts (`src/services/campaignPersistence/CampaignPersistenceService.ts:79/130/201`) with no runtime re-validation. A stale or malformed pack envelope passing the 4 checks would persist and fail **silently** downstream — exactly Momus's wound. The pack loader therefore performs its own deep zod validation **before** the PUT (design D3); the production route is not touched (Non-goals).
- **Shared SQLite under `fullyParallel`.** One webServer process (`playwright.config.ts:190`), one default DB file (`DATABASE_PATH || './data/mekstation.db'`, `src/services/persistence/SQLiteService.ts:43`), auto workers locally. Packs write real rows through the production PUT, so parallel loads of the same pack would share a row. The loader stamps id-*templates* per load (`${packId}-w${workerIndex}-${uuid}`) — and the stamping must be **graph-aware**, not top-level-only: `SerializedCampaignBody` carries nested id cross-references (force ids, `rootForceId`, mission ids, unit-state record keys, and `rosterProjection`'s own duplicated `campaignId`, `src/types/campaign/SerializedCampaign.ts:148-281`) that a naive single-field stamp would desync (design D4).

**Council-sketch drift, resolved here (verified facts override the sketch):** (1) `buildSerializedCampaign` is pre-existing `campaign-persistence` production code (`src/lib/campaign/persistence/campaignEnvelope.ts:32-47`), not something W3 creates — W3 merely produces the fast-forwarded campaigns it dumps; this change cites the `campaign-persistence` envelope/schema-version/conflict contracts rather than restating them. (2) `schemaVersion` is already unconditionally stamped and already has a load-bearing, idempotent, forward-tolerant migration ladder (`migrateSerializedCampaign`, `src/lib/campaign/persistence/campaignMigration.ts:59-81`); packs pin and ride that machinery, never duplicate it (design D3). (3) The council's "zod-validated at load" binds the **pack loader**, not a production fix to `isValidPutBody` — the route fix is recorded as an open follow-up, not silent scope creep. (4) The maintenance subsystem has a structural surprise: `campaign.campaignInventory` (what the repair bay reads) is populated only by the CLEANUP-phase `inventoryProjectionProcessor` on `advanceDay` and is **not** part of `serializeCampaign`'s output — a loaded campaign pack renders an empty repair queue on first paint. Design D8 makes that call explicitly instead of assuming it away.

**Dependency reality:** the W2 (`add-seam-trust-anchor-journeys`) and W3 (`add-campaign-fast-forward-api`) implementations are 0% complete on this tree (both change folders are purely declarative; W2 is in flight on a parallel branch). The shared IndexedDB seeder `loadEncounterPack` reuses (`e2e/helpers/matchLogSeeding.ts`) exists only as W2 task 2.1; the pack-minting engine (`fastForwardCampaign()`) exists only as W3 tasks. Every W4 task consuming those artifacts carries an explicit W2-task-4.0-style **STOP-on-absence gate check** (design D10) — the stronger of the ladder's two gating precedents, chosen deliberately.

## What Changes

- **Pack library** (`src/lib/scenarioPacks/`, new — jest-tested test-support library, the W3 D3 placement precedent; jest ignores `e2e/` entirely via `testPathIgnorePatterns`, `jest.config.js:46`, so validation/stamping logic cannot live there and keep unit coverage): zod pack schemas (deep campaign-envelope shape, match-log pack shape with the `GameCreated`-first rule and a finite persisted `config.seed`, manifest entry shape), the graph-aware id-template stamper with its zero-residue guarantee, and the manifest types.
- **Pack registry** (`e2e/scenario-packs/`, new): a typed manifest module (the `e2e/flows/manifest.ts` precedent) with one entry per pack — `id`, `kind` (`campaign` | `encounter`), `subsystems` (the closed six-tag vocabulary), `viewports`, `targetRoute`, `parityAnchorJourney`, payload path, genesis provenance, and version pins — plus the generator-minted payloads: `campaign/*.campaign.json` and `encounter/*.matchlog.json`.
- **Loaders** (`e2e/helpers/scenarioPackLoading.ts`, new): `loadCampaignPack` (validate → stamp → PUT `{envelope, baseVersion}` to `/api/campaigns/:id` → `goto` the target route → wait for the production load to complete) and `loadEncounterPack` (validate → stamp → seed IndexedDB via W2's shared `matchLogSeeding` helper → cold `goto` → wait for recovery). Both enter exclusively through `page.goto` against production routes.
- **Minters**: mint-from-flow-checkpoint (drive a registered flow-audit flow to a server-persisted checkpoint and capture the campaign envelope), mint-from-fast-forward (dump a W3 `fastForwardCampaign()` run via `buildSerializedCampaign` — W3-gated), and capture-matchlog (capture a genesis journey's real IndexedDB event stream — W2-gated). Every mint ends with a canonicalization pass (design D11): the captured wall-clock production ids are rewritten to deterministic pack-scoped template ids, and cross-store references that would dangle in the loading browser (roster mission records' `encounterId`/`gameSessionId`) are stripped. Hand-authoring a pack payload is banned by spec.
- **Six pilot packs, one per subsystem** (design D6): navigation and personnel and experience mint now from merged flow-audit checkpoints (`contract-accepted`, `roster-updated`, `xp-surface-viewed` — all `holdSafe: true`); combat gates on W2 (captured from the fresh-construction anchor's session); economy and maintenance gate on W3 (fast-forward dumps). Each pack ships with a parity spec asserting its genesis journey's invariants on the pack-loaded state.
- **CI re-capture-and-diff**: a `scenario-pack-recapture` job in `.github/workflows/nightly-validation.yml` (the existing 5-job pattern: `needs: install-deps`, added to `notify-on-failure`'s `needs`) that re-validates committed payloads against the current schemas and pins, re-mints each pack from its genesis source, and compares fresh mint vs committed pack **at the invariant level** per genesis class — structural payload equality is unachievable by construction (production mints wall-clock ids, a random per-campaign RNG seed, and wall-clock engine seeds; design D7/D11). Drift detection only — never a PR gate.
- **Ledger**: `openspec/active-change-ledger.json` gains this change's entry (ships with this declarative PR, per the W1/W2/W3 precedent).

## Capabilities

### New Capabilities

- `scenario-packs`: the pack registry/manifest contract, the generator-minted genesis doctrine (packs are caches, genesis journeys are truth), the front-door loading contract, fail-loud validation at load, graph-aware parallel-safe id templating, version pinning plus CI drift recapture, and the parity binding + gate rule (packs gate nothing until parity-proven; anchors are never displaced). Scoped to the pack layer only: it does NOT re-specify the campaign envelope/migration/conflict contracts (`campaign-persistence` owns them), the fast-forward driver or its live-parity acceptance (`campaign-fast-forward-api` owns them), the seam trust anchors (`journey-qc` owns them, landing via W2), or the flow-audit registry (`flow-audit-routines` owns it).

### Modified Capabilities

- None — deliberate (design D1's rejected-alternatives matrix). The anchor-side "packs never displace an anchor" clause already exists in W2's `journey-qc` delta ("Seam Trust Anchor Journeys": packs "MAY complement the anchors but SHALL NEVER replace them"); W3's spec already names the pack registry/loaders/manifests as W4 scope; `campaign-persistence` and `e2e-testing` are consumed, not changed (each capability contracts its own specs' Playwright scoping, per W2 design D4). This change adds the **pack-side** obligations in its own new capability and cites the rest.

## Impact

- **New code** (implementation phase, all additive test-support): `src/lib/scenarioPacks/` (schemas, stamper, manifest types + colocated `__tests__/`), `e2e/scenario-packs/` (manifest + JSON payloads + per-pack parity specs), `e2e/helpers/scenarioPackLoading.ts`, minting scripts, one nightly workflow job, one `verify:qc:scenario-packs` package script, three `testIgnore` additions (`playwright.config.ts` responsive projects), one ledger entry.
- **Modified production code: one sanctioned additive exception** (orchestrator ruling 2026-07-11, design.md D3 amendment, the W3 D9 precedent): `src/lib/e2e/storeExposure.ts` gains a `campaignPersistence: typeof useCampaignPersistenceStore` entry on `window.__ZUSTAND_STORES__`, following that file's own existing pattern for its other exposed stores — additive, E2E-mode-only, production-behavior-neutral. It is the one signal `loadCampaignPack`'s post-`goto` load-outcome check (design D3) needs and that no other already-exposed surface or DOM testid provides. Every other existing `src/` file — `isValidPutBody`, the read-side casts, the migration ladder, the campaign shell, and the recovery factory — is consumed exactly as shipped, unedited.
- **Ordering**: implementation follows the ladder's strict sequencing — it starts after the W3 (`add-campaign-fast-forward-api`) implementation merges. Mechanically, task groups 1–3 (library, campaign loader, flow-checkpoint packs) depend only on already-merged surface; group 4 gates on W2 implementation artifacts and group 5 on W3 implementation artifacts, each with a STOP-on-absence gate task.
- **Ledger reconciliation at ship time**: this worktree's `active-change-ledger.json` lists `add-seam-trust-anchor-journeys` and `add-campaign-fast-forward-api` as in-progress; either may archive on `origin/main` before this PR lands (W2's implementation is in flight on a parallel worktree). The PR must land its ledger entry on top of a rebase against current `origin/main` or `qc:openspec-ci:validate` fails the build (the `scripts/qc/validate-openspec-ci-quality.mjs:142-167` folder↔ledger cross-check).
- **Risk**: low. Pure test-support surface. The risks needing care — half-stamped id graphs, W2/W3 artifact drift at gate time, shared-SQLite contention, Playwright discovery multiplication, recapture flake — are individually gated in `design.md` (R1–R10).

## Non-goals

- **Packs gate NOTHING until parity-proven.** No CI gating changes: no `pr-checks.yml` edits, no required checks, no branch-protection changes; the nightly recapture job is drift triage, not a gate. **Dissent recorded per the council:** Prometheus holds that parity-gated, front-door-loaded packs can eventually *be* the gate; Hephaestus/Oracle hold that seams (b)-client and (c) are structurally unpackable, so the genesis journeys keep authority forever. Near-term behavior is identical either way — packs gate nothing until proven — and any future promotion of packs into a gate requires an explicit delta to this capability's parity requirement plus a green W3 live-parity acceptance, never a quiet CI edit.
- **No store injection for anything with a mount seam.** Store-injection-seeded states (e.g. the flow-audit maintenance flow's `seedRepairInventory`, `holdSafe: false`) are legitimate for flow-audit's own doctrine but are NOT eligible pack genesis sources, and no loader writes application stores directly. "Seam-free" (council Phase 3 update 4) describes where W3's headless jest coverage is dominant — it is not an exemption from front-door loading for anything shipped under `e2e/scenario-packs/`.
- **No production behavior changes.** `isValidPutBody` stays as-is (the documented shallow-guard gap is defended loader-side; a zod-backed route guard is an Open Question for its own change with its own sanctioned-fix ceremony, per the W3 D9 precedent). No read-side re-validation, no migration-ladder changes, no `advanceDay`/day-pipeline changes, no per-worker `DATABASE_PATH` keying (council open risk 2's escalation stays a separate change).
- **No W5/W6 scope.** No viewport-layout sweep, no Playwright-project rationalization beyond this change's own `testIgnore` entries, no subsystem CI lanes, no PR-gate wiring. The manifest's `viewports` field is data for W5 to consume, nothing more.
- **No hand-authored packs, ever** — including "just this once" bootstrap payloads. A pack that cannot yet be generator-minted (its gate hasn't landed) simply does not exist yet; hand-faking a gated pack to reach six-subsystem coverage early is banned by spec.
- **No W2/W3 restatement.** The anchors' contracts, the fast-forward run report, and the live-parity acceptance are cited from their owning specs, never duplicated. No edits to any anchor spec, audit journey, or the seven-journey catalog (`requiredJourneyIds` stays exactly seven).
- **No golden traces.** Pack parity and recapture diffs assert invariants and normalized structural equality — never pinned seed→outcome values (the W1/W3 rule applies verbatim).
