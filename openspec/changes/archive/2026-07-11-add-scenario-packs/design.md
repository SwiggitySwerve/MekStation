# Design: add-scenario-packs

## Context

Everything a pack needs to enter through the front door already exists in production; nothing pack-shaped exists anywhere (verified greenfield: zero hits for `scenario-pack|loadCampaignPack|loadEncounterPack|parityAnchorJourney` outside planning documents).

**Campaign side (the R2 mechanism):**

- `PUT /api/campaigns/:id` (`src/pages/api/campaigns/[id].ts:98-125`) parses `req.body`, guards it with `isValidPutBody` (lines 51-69 — exactly 4 shallow checks: non-null object, `baseVersion` number ≥ 0, `envelope.campaignId` string + `envelope.schemaVersion` number + `envelope.body` non-null object; the doc comment says "Phase 1 trusts the caller for the deep envelope shape"), then `saveCampaign(body.envelope, body.baseVersion)`. Read-back paths trust `JSON.parse(row.payload) as SerializedCampaign` with no runtime re-validation (`CampaignPersistenceService.ts:79/130/201`) — corrupt JSON is caught, semantically malformed valid JSON is not.
- The campaign shell auto-loads on mount: `useCampaignRouteLoader` (`campaignPageShell.tsx:120-170`) fires `void loadCampaign(routeCampaignId)` when the route id doesn't match the loaded campaign; `loadCampaignAction` (`useCampaignPersistenceStore.ts:441-483`) runs GET → `migrateSerializedCampaign` → `deserializeCampaignRecord` → `writeLiveCampaign` → `restoreRosterProjection(id, migrated.body.rosterProjection)`. PUT + `goto` therefore rides the full production load path with zero store injection.
- `buildSerializedCampaign` (`campaignEnvelope.ts:32-47`) — **pre-existing `campaign-persistence` production code, not a W3 artifact** — stamps `schemaVersion: CURRENT_CAMPAIGN_SCHEMA_VERSION` (= 1, `campaignMigration.ts:21`), `campaignId`, `savedAt` (wall clock), `originDeviceId`, `version`, and `body`. `migrateSerializedCampaign` (`campaignMigration.ts:59-81`) is an ordered, idempotent ladder that no-ops gracefully on unknown future versions. Any W4 version-pinning integrates with this ladder; duplicating or shadowing it is forbidden.
- The envelope's id graph is deep (`SerializedCampaign.ts:111-281` — the roster-projection interfaces start at line 111, before the body interface): top-level `campaignId`; `body.id`; `body.forces` as `[forceId, IForce][]` where `IForce` carries `id`/`parentForceId`/`subForceIds[]`/`unitIds[]`/`commanderId` (`Force.ts:33-63`); `body.rootForceId`; `body.missions` as `[missionId, IMission][]`; `body.unitCombatStates`/`unitMaxStates` keyed by unit id; `body.rosterProjection` with its **own** nested `campaignId`, `units`, `pilots` (`ICampaignRosterEntry.pilotId`, `CampaignRosterEntry.ts:55`), and mission records (`SerializedCampaignRosterMissionRecord`, lines 129-140) carrying `campaignId` + `deployedUnitIds[]` — plus optional `encounterId`/`gameSessionId` (lines 134-135), **cross-store references** into the encounter store and the match-log IndexedDB that do not travel with a campaign pack (stripped at mint, D11).

**Encounter side (the R3 mechanism):**

- IndexedDB match-log schema: `mekstation-match-log` v2, stores `matchEvents` (keyPath `['matchId','sequence']`) and `matches` (keyPath `'matchId'`) (`src/lib/p2p/matchLogStorageSchema.ts:1-38`, creation in `matchLogStorage.helpers.ts:20-45`). Recovery requires `GameCreated`-first: `hydrateRecoverableSessionFromMatchLog` sorts by sequence and throws `InteractiveSessionRecoveryCorruptError` if `orderedEvents[0]?.type !== GameEventType.GameCreated` (`InteractiveSession.persistence.ts:57-81`). Post-W1, `IGameConfig.seed` (`GameSessionUnitTypes.ts:81`) rides on the `GameCreated` payload's `config` so recovery re-seeds deterministically.
- The seeding mechanism is proven: `e2e/active-session-recovery.spec.ts:85` (`seedMatchLog`) already writes both stores and cold-navigates. W2 task 2.1 extracts it to `e2e/helpers/matchLogSeeding.ts` "parameterized over match id + event stream + matches-row fields" — **not yet merged** (W2 is 0/23 tasks on this tree, in flight on a parallel branch).

**Test-infrastructure facts that shape placement:**

- jest ignores `e2e/` entirely (`testPathIgnorePatterns` includes `<rootDir>/e2e/`, `jest.config.js:46`) — pure logic needing jest coverage must live under `src/`. e2e files already import from `src/` (`e2e/flow-audits.spec.ts:37` imports `../src/constants/layout`); the reverse direction (src importing e2e) does not exist and is not introduced.
- Playwright `testDir: './e2e'` auto-discovers new spec files into `chromium` + the three responsive projects (Mobile Chrome / Tablet Portrait / Tablet Landscape, each with a `testIgnore` array); `smoke` selects by `grep /@smoke/`; the `flow-audit` project is env-gated with its own `testMatch`. One shared webServer (`playwright.config.ts:190`), `workers: CI ? 1 : undefined` (line 49), one shared SQLite file (`SQLiteService.ts:43`).
- `.github/workflows/nightly-validation.yml` has 5 jobs (`install-deps`, `full-statistical-proofs`, `e2e-full`, `coverage`, `notify-on-failure` with `needs: [full-statistical-proofs, e2e-full, coverage]`), cron + `workflow_dispatch`, no paths-filtering. `e2e-full` runs `--ignore-snapshots` because visual baselines are win32-only and the runner is ubuntu — Playwright screenshot diffing is NOT a viable recapture transport. The in-repo committed-artifact-diff precedent is jest `toMatchSnapshot` (SVG record sheets); the `validate-*-parity.mjs` scripts are claim-coverage validators, not recapture diffing (a naming red herring).
- The flow-audit registry (`e2e/flows/manifest.ts`) is the manifest precedent: a typed TS module, closed six-tag `FlowSubsystem` vocabulary (`navigation`/`combat`/`economy`/`maintenance`/`personnel`/`experience`), per-checkpoint `holdSafe` flags (true only when the state at the checkpoint is server-persisted). The maintenance flow is all `holdSafe: false` — its repair-bay state comes from store injection because `campaign.campaignInventory` is populated only by the CLEANUP-phase `inventoryProjectionProcessor` on `advanceDay` (`processorRegistration.ts:119`) and is absent from `serializeCampaign` output.

**Re-mint nondeterminism (verified) — what a generator can and cannot reproduce:**

- `createCampaign` mints the campaign id and root force id via `generateUniqueId` — `${prefix}-${Date.now()}-${random}` (`src/types/campaign/Campaign.ts:599-626`) — stamps a production-random `rngSeed` (`Math.floor(Math.random() * 0x100000000)`, `Campaign.ts:647`) that drives daily RNG and market rolls, and anchors the in-fiction calendar at wall clock (`currentDate: new Date()`), which propagates into contract deadlines, market offers, and transaction dates. Mission ids are `mission-${Date.now()}-${random}` (`useCampaignRosterStore.ts:173`, `campaignRosterStore.helpers.ts:36-38`).
- A fresh live session's engine seed defaults to wall clock — `this.seed = config.seed ?? Date.now()` (`src/engine/GameEngine.ts:97`) — so a re-captured matchlog carries a new seed and therefore a structurally different roll sequence, event stream, and mid-battle state on every capture.
- W3's Deterministic Reproducibility requirement promises **invariant-level** run-to-run equality only (unit survival sets, armor/structure deltas, balance delta, transaction types and amounts, XP counters, statuses, ticket counts — wall-clock-bearing fields excluded); it never promises structural payload equality, and mission ids sit outside its equality set entirely.

No genesis class can re-mint a byte-identical payload. This fact shapes D7 (the recapture comparison is invariant-level) and D11 (the minter canonicalizes what CAN be deterministic — the id graph — and strips what cannot travel).

Implements `specs/scenario-packs/spec.md` (new capability). No other spec deltas (D1).

## Goals / Non-Goals

**Goals:**

- Named, reusable, front-door-loaded start states: campaign packs (PUT + goto) and encounter packs (IndexedDB + cold goto), zod-validated fail-loud, parallel-safe via graph-aware id templating.
- A generator-minted registry with provenance, one pilot pack per subsystem (three now, three behind explicit W2/W3 gates), each bound to a parity anchor journey.
- The four-layer drift defense: generator minting, fail-loud validation, version pins, nightly CI re-capture-and-diff.

**Non-Goals:**

- Packs as gates (parity rule, Prometheus dissent recorded), store injection, production changes (including `isValidPutBody`), W5/W6 scope, hand-authored payloads, W2/W3 restatement, golden traces. See proposal Non-goals.

## Decisions

### D1 — New capability folder `scenario-packs`; zero deltas elsewhere

**Choice**: a NEW capability owns the pack contract, per the council's explicit table entry ("W4 | `scenario-packs` (NEW spec)"). The delta authors its own Purpose inline so the archive-time spec never ships a TBD placeholder (the W3 `campaign-fast-forward-api` pattern).

**Alternatives surveyed and rejected:**

| Option | Verdict | Why |
|---|---|---|
| Fold into `journey-qc` | **Rejected** | journey-qc owns journey-level QC coverage doctrine and the seam anchors (via W2's delta). Packs own real behavior of their own — registry schema, id templating, validation, minting, drift recapture — which would balloon a journey-coverage spec into a fixture-infrastructure spec. The anchor-side clause packs need ("Packs and shortcuts never displace an anchor") already exists in W2's ADDED requirement; this change adds the pack-side obligations and cites it. Council open risk 3 ("journey-qc MODIFIED deltas must accompany W2/W4/W6") is resolved by verified drift: no anchor requirement text changes in W4 — the anchors are referenced, untouched, so there is nothing to MODIFY; forcing a no-op delta would be churn, not compliance. |
| Fold into `e2e-testing` | **Rejected** | Plausible home for Playwright-adjacent registration mechanics, but the council names a new spec, and `e2e-testing`'s scoping requirements govern *its own* named specs — the W2 D4 precedent is that each capability contracts its own specs' project scoping. The pack parity specs' chromium-only obligation lives in the new capability. |
| Fold into `campaign-fast-forward-api` | **Rejected** | W3's spec explicitly names the registry/loaders/manifests as W4 scope (its Notes: "the pack registry, loaders, and manifests are W4 scope") and its own proposal non-goals exclude them. Fast-forward is one of three genesis sources, not the owner of the pack layer. |
| Delta `campaign-persistence` | **Rejected** | The envelope, schema-version ladder, server contract, and stale-write conflict requirements are consumed unchanged. The pack loader adds no persistence behavior; its zod validation is a client of the contract, not a modification of it. |

**Duplicate-owner guard**: the new spec must NOT restate the envelope shape, migration semantics, conflict detection (`campaign-persistence`), the fast-forward driver/run-report/live-parity contracts (`campaign-fast-forward-api`), the anchor invariants (`journey-qc` via W2), or the flow registry contract (`flow-audit-routines`). It references them and adds only pack-layer obligations.

### D2 — Registry shape: typed TS manifest + JSON payloads; `manifestVersion` distinct from campaign `schemaVersion`

**Choice**: `e2e/scenario-packs/manifest.ts` is a typed TS registry module (the `e2e/flows/manifest.ts` precedent — single source of truth, one entry per pack), importing its types from the pack library (D5). Payloads are generator-dumped JSON: `e2e/scenario-packs/campaign/<id>.campaign.json`, `e2e/scenario-packs/encounter/<id>.matchlog.json`.

**Manifest entry fields** (the council's list plus the drift-resolved additions): `id` (kebab-case, unique), `kind` (`'campaign' | 'encounter'`), `subsystems` (the closed six-tag vocabulary — same literal set as `FlowSubsystem`; a guard test asserts the two stay equal), `viewports` (data for W5; no W4 behavior), `targetRoute` (route template the loader substitutes the stamped id into), `parityAnchorJourney` (D9's binding reference: `flow:<flowId>@<checkpoint>` | `anchor:<anchor-spec-basename>` | `fast-forward:<fixtureId>`), `payloadPath`, `provenance` (`genesisSource`, `mintedAt` ISO date, `baseCommit` SHA), `pins` (campaign packs: the envelope `schemaVersion` the pack was minted at; encounter packs: the `MATCH_LOG_DB_VERSION` the capture ran against), and `postLoadActions` (D8 — closed vocabulary, initially `['advance-day']` only, empty for most packs).

**`manifestVersion` is its own field on the registry module, never conflated with the campaign envelope's `schemaVersion`.** The two evolve on different timelines: the envelope schema is `campaign-persistence`'s contract with its own migration ladder; the manifest format is this capability's contract. Conflating them would make a manifest-format bump look like (and get migrated as) a campaign-schema bump.

### D3 — Fail-loud zod validation lives in the loader; the production route is untouched; the existing migration ladder is ridden, never duplicated

**The wound (verified)**: `isValidPutBody`'s 4 shallow checks accept any object with the right top-level property types; the read side re-trusts the payload via unchecked casts. A drifted pack would fail silent.

**Choice**: the pack library ships zod schemas and the loaders parse the pack payload **before** any PUT or IndexedDB write:

- **Campaign pack schema — the floor is the deserializer, not the id graph**: the full envelope skeleton (`campaignId`/`schemaVersion`/`savedAt`/`originDeviceId`/`version`/`body`) plus **every field the production load path hard-depends on**. `deserializeCampaignBody` (`src/lib/campaign/persistence/serializeCampaign.ts:209-270`) hard-reads `body.id`, `name`, `currentDate`, `factionId`, `forces`, `rootForceId`, `missions`, `finances`, `factionStandings`, `options`, `campaignType`, `createdAt`, and `updatedAt`; `deserializeFinances` (`serializeCampaign.ts:82`) maps over `finances.transactions` and wraps `finances.balance` — absence throws. All of these are required by the schema. Date-time-valued strings (`currentDate`; `campaignStartDate` and transaction `date`s when present) are validated **parseable**: `new Date(<garbage or undefined>)` yields `Invalid Date` silently — the campaign loads and lies, which is exactly the silent-stale-envelope wound. On top of the deserializer floor sit the id-linkage fields the stamper depends on — the `forces` entry tuples and `IForce` id fields, `missions` entry tuples, the `unitCombatStates`/`unitMaxStates` record keys, and the complete `rosterProjection` linkage (`campaignId`, `units`, `pilots[].pilotId`, mission records' `campaignId` + `deployedUnitIds`). Unknown extra keys pass through (forward tolerance for additive envelope evolution — the ladder's own graceful-degradation stance); the zero-residue scan (D4) is the mechanical backstop for id-bearing fields the schema doesn't know. Validation failures throw naming the offending path — never a soft skip.
- **Encounter pack schema**: the `matches` row shape, the ordered `matchEvents` records, sequence-0 event type === `GameEventType.GameCreated` (`'game_created'`), and a **finite persisted `payload.config.seed`** on the `GameCreated` event — an unseeded matchlog is not replayable/deterministic and is rejected at load (the W1/W2 recovery re-seed contract is the reason packs can be trusted to recover identically).
- **schemaVersion pin check (campaign)**: the loader asserts the pack's `schemaVersion` equals the manifest pin and is ≤ `CURRENT_CAMPAIGN_SCHEMA_VERSION`. An older-but-migratable pack loads fine — the production GET path runs `migrateSerializedCampaign` exactly as it does for real saves (riding the ladder is the point); the staleness itself surfaces loudly through the recapture job (D7), not as a load failure. A pack claiming a FUTURE version fails loud (nothing can honestly have minted it).
- **`MATCH_LOG_DB_VERSION` pin check (encounter)**: `loadEncounterPack` asserts the manifest's match-log pin **strictly equals** the live `MATCH_LOG_DB_VERSION` (`src/lib/p2p/matchLogStorageSchema.ts:4`, currently 2) before any IndexedDB write. The semantics deliberately differ from the campaign pin: the match-log store has **no migration ladder**, so a stale capture seeded into a bumped DB is old-shaped records under new-shape consumers — an older pin is as invalid as a newer one, and the only remedy is a re-capture. A pin recorded but never consulted is a pin that cannot fail; this check is the consultation site.
- **`manifestVersion` pin check (registry)**: the registry tests assert the manifest module's declared `manifestVersion` equals the pack library's exported manifest-version constant, failing loud on mismatch. All three pins now name their comparison sites (see D7 layer 3).
- **Post-`goto` load-outcome check (the client-side soft-fail defense)**: `loadCampaignAction` folds every client-side failure into store state — the catch-all sets `saveState: 'error'` + `errorMessage` (`useCampaignPersistenceStore.ts:477-481`), and the 404 path sets `saveState: 'idle'` + `'campaign not found'` (`useCampaignPersistenceStore.ts:450-453`). A pack that passes loader validation but fails during the mount-time migrate → deserialize → restore chain would otherwise surface only as a generic mount-signal timeout. `loadCampaignPack` therefore determines the load outcome from the persistence store's own post-`goto` state: success is `saveState: 'saved'` with the loaded campaign id matching the stamped id; any reported `errorMessage` (whether via the `'error'` state or the not-found `'idle'` path) is thrown immediately, naming the store's message. A generic timeout is never the failure surface for a named error state.

**Rejected — fixing `isValidPutBody` in this change**: the architecturally attractive fix (zod-backed deep guard at the route, re-validation on read) is a production behavior change. Every prior wave carried at most ONE narrowly sanctioned production fix with its own decision ceremony (W3 D9); W4 needs none to meet its goals, because the loader validates strictly more deeply than the route does, on every payload it sends. The route gap remains real for non-pack callers and is recorded in Open Questions as its own follow-up change — never silently absorbed here.

**Amendment (orchestrator ruling, 2026-07-11) — the one sanctioned production touch, W3 D9 precedent**: implementation of the post-`goto` load-outcome check above surfaced a signal gap group-1 flagged but could not resolve read-only: `useCampaignPersistenceStore` (the store that actually owns `saveState`/`errorMessage`/`campaignId`) is never on `window.__ZUSTAND_STORES__` (`src/lib/e2e/storeExposure.ts`), and the DOM offers only a generic, testid-less not-found state. Mirroring the store's decision logic against the raw GET response status (the loader's first-draft approach) would have inferred load success from an unrelated store's side effect (`writeLiveCampaign` touching the already-exposed `campaign` store) rather than reading the actual named-error contract D3 promises. Per the same class of exception W3 D9 used for its own narrowly sanctioned production fix, this change adds exactly ONE additive touch: `src/lib/e2e/storeExposure.ts` gains a `campaignPersistence: typeof useCampaignPersistenceStore` entry on `window.__ZUSTAND_STORES__`, following the file's own existing pattern for every other already-exposed store (a normal `create<T>()` Zustand hook with `.getState` attached as a static — the same shape as `campaignRoster`/`force`/`pilot`/etc., not the lazy-wrapper special case `campaign` needs). The touch is: (a) E2E-only (the whole exposure module no-ops unless `NEXT_PUBLIC_E2E_MODE=true`); (b) production-behavior-neutral (adds a read surface, changes no existing behavior); (c) additive (no existing exposure entry, type, or call site is edited). `loadCampaignPack`'s post-`goto` check reads `campaignPersistence.getState()` directly — `saveState: 'saved'` with a matching `campaignId` is success; any `errorMessage` (the `'error'` state or the not-found `'idle'` state, `useCampaignPersistenceStore.ts:450-453/477-481`) throws immediately naming the store's own message. This is the ladder's second sanctioned production touch (after W3 D9) and, like that one, is recorded here rather than silently folded into the "zero-production-touch" framing above — proposal.md's Impact section names the same single exception.

### D4 — Id templating is graph-aware: one remap table per load, zero-residue guarantee

**The naive reading of the council line ("the loader stamps `${packId}-${workerIndex}-${uuid}` before PUT/seed") under-specifies the work**: stamping only `envelope.campaignId` would leave `body.id` disagreeing with the route id (the shell's route loader compares them — a mismatch re-fires `loadCampaign` forever), and would leave `rosterProjection.campaignId`, mission records, and every internal cross-reference pointing at the template ids.

**Choice**: per load, the stamper builds a **remap table** (original id → stamped id) and applies it consistently across the whole document:

1. **Campaign packs**: `campaignId` remaps to `${packId}-w${workerIndex}-${uuid}`; the remap is applied to `envelope.campaignId`, `body.id`, `body.rosterProjection.campaignId`, and every rosterProjection mission record's `campaignId`. Campaign-internal ids (force ids, mission ids, unit ids, pilot ids, unit-state record keys) are remapped through the same table **when and only when they collide across rows** — group-1 verification (task 1.2) settles whether any of them leak into shared server tables beyond the single campaign row (the roster projection restores client-side from the envelope; if campaign surfaces additionally require `/api/pilots` rows, pilot ids join the remap and the loader creates those rows front-door — R10). Internal ids that never leave the row MAY keep their minted values (self-contained graph), but any id the remap table touches is rewritten at **every** occurrence.
2. **Encounter packs**: `matchId` remaps to the template; applied to the `matches` row key, every `matchEvents` record's `matchId` key component, and every event-payload field carrying the match/session id. Deployed session unit ids (`${side}-${slot}-${unitRef}` composites) are session-scoped and never remapped — they cross no row boundary and rewriting them would corrupt the event stream's internal references.
3. **Zero-residue guarantee (the mechanical backstop)**: after stamping, the serialized pack SHALL contain zero occurrences of any id the remap table maps from. This catches un-enumerated id-bearing locations (including future additive envelope fields that pass zod's passthrough) without pretending the field enumeration is complete forever. The stamper throws naming the residue path.

**Template deviation, made explicit**: today's e2e uniquing convention is `${workerIndex}-${Date.now()}` (`e2e/replay-player.spec.ts:363/446/736`). The council's template uses a uuid; this design adopts `${packId}-w${workerIndex}-${uuid}` deliberately — `Date.now()` collides across two loads inside one millisecond (cheap to hit when a spec loads several packs in a loop), `crypto.randomUUID()` does not, and the `workerIndex` component is kept for row-ownership debuggability rather than uniqueness. The deviation is confined to the pack loader; no existing spec's convention changes.

**Mint-side symmetry (D11)**: the ids the load-time remap maps FROM are not raw production ids — the minter canonicalizes the captured id graph to deterministic pack-scoped template ids before the payload is written (D11). The two passes share the remap machinery and the same id-family scope: whatever task 1.2's inventory puts in the remap set gets canonicalized at mint and re-stamped at load.

### D5 — Placement: pure logic in `src/lib/scenarioPacks/`; page-touching loaders in `e2e/helpers/`; data in `e2e/scenario-packs/`

**Choice**: zod schemas, the stamper, and the manifest types live in `src/lib/scenarioPacks/` with colocated `__tests__/` — jest ignores everything under `e2e/` (`jest.config.js:46`), so this is the only placement that gets the pure logic real unit coverage, and it follows W3's D3 precedent (test-support library code in `src/lib` because multiple consumers — Playwright loaders, minters, the CI recapture script — import it). The Playwright-dependent loaders (`loadCampaignPack`/`loadEncounterPack`) live in `e2e/helpers/scenarioPackLoading.ts` and import the library (the established e2e→src import direction, `flow-audits.spec.ts:37`); no `src/` file imports from `e2e/`. Payloads and the manifest live in `e2e/scenario-packs/` beside the parity specs that consume them.

### D6 — The six pilot packs: genesis map, gates, and the store-injection boundary

Genesis sources are chosen per the packs-are-caches doctrine — every pack's payload is captured from a real journey or a sanctioned generator run, at a **server-persisted** state (for campaign packs: a `holdSafe: true` checkpoint or a fast-forward dump; browser-only states cannot be enveloped by construction):

|---|---|---|---|---|---|

| 2 | combat | encounter | Mid-battle, post-Initiative, mirrored canonical `unitRef` pair (the #1019-class roster) | captured matchlog from the W2 fresh-construction anchor's launched-and-advanced session | **W2 impl** (task 4.0 gate) |

**Store-injection boundary (resolved explicitly)**: the flow-audit maintenance flow seeds `campaign.campaignInventory` by store injection (`holdSafe: false` on every checkpoint) precisely because that projection is not serialized — legitimate for flow-audit's capture-tolerant doctrine, **ineligible as a pack genesis source**. "Seam-free" in the council's Phase 3 update 4 grants headless-jest dominance to W3's invariants; it grants no exemption from front-door loading to anything under `e2e/scenario-packs/`.

### D7 — Four-layer drift defense; the recapture comparison is invariant-level, because structural re-mint equality is unachievable by construction

Each layer is a separate obligation (a pack passing three of four is still in violation):

1. **Generator minting with provenance**: every payload is written by a registered minter (mint-from-flow-checkpoint, mint-from-fast-forward, capture-matchlog); the manifest records `genesisSource`, `mintedAt`, `baseCommit`. Hand-authoring or hand-editing a payload is banned; a hand-edit shows up as a schema/pin/invariant violation in the recapture job or as unexplained drift in the canonical id graph (D11).
2. **Fail-loud validation at load** (D3).
3. **Version pins** (D2), each with a named consultation site (D3): envelope `schemaVersion` pin — asserted by `loadCampaignPack` (equality with the manifest pin, ≤ current); `MATCH_LOG_DB_VERSION` pin — asserted by `loadEncounterPack` (strict equality with the live constant; no ladder exists for the match-log store); `manifestVersion` — asserted by the registry tests against the library's exported constant. A pin mismatch at load fails loud; a `CURRENT_CAMPAIGN_SCHEMA_VERSION` advance shows up in the recapture job (the committed payload's pin re-validation) prompting a re-mint, while old campaign packs stay loadable through the production ladder.
4. **CI re-capture with invariant-level comparison**: a `scenario-pack-recapture` job in `nightly-validation.yml` (pattern-identical to the existing jobs: `needs: install-deps`, checkout + shared setup action; appended to `notify-on-failure.needs` so drift lands on the existing tracking-issue convention with zero new notification machinery). Per manifest entry the job does three things: **(a)** re-validates the **committed** payload against the current pack schemas and all three pins — catching schema/pin evolution that staled a committed pack; **(b)** re-mints from the recorded genesis source and validates the fresh mint identically — catching genesis drift that broke mintability; **(c)** compares the fresh mint against the committed pack **at the invariant level**, per genesis class: flow-checkpoint packs assert the checkpoint's declared invariant predicates on both payloads (run-generic predicates — a re-mint legitimately accepts a *different* contract from a differently-rolled market, because `rngSeed` is production-random per creation, so amount-level equality would be dishonest); fast-forward packs assert W3's enumerated invariant-level field set between the two (deterministic per fixture — exactly what W3's Deterministic Reproducibility contract promises, and nothing more); matchlog packs assert the anchor's seed-agnostic invariant set plus schema/grammar validity (the engine seed is wall-clock-fresh per capture). Any violation fails the job loud, naming the pack and the violated invariant or path.

**Structural JSON diff with a normalization list — the council sketch's literal transport — is rejected as red-by-construction** (Context: re-mint nondeterminism). Every flow re-mint differs in campaign/force/mission ids (D11's canonicalization fixes those) *and* in the whole in-fiction calendar and the `rngSeed`-derived market content; every matchlog re-capture differs in every roll; W3's determinism contract is invariant-level only — its exclusion list normalizes wall-clock bookkeeping fields, not payload structure, and reusing it as a structural-diff normalization set misreads it. A structural diff would fail every night and pressure exactly the forbidden normalization widening. The structural checking that remains is honest and bounded: schema + pin validation of both payloads, and D11's canonical id graph, which IS identical across re-mints of the same deterministic state.

**Transport decision**: Playwright `toHaveScreenshot` is rejected (nightly runs `--ignore-snapshots`; baselines are win32-only, the runner is ubuntu); the mechanism is re-mint + invariant comparison in a node script. The `validate-*-parity.mjs` scripts are claim-coverage validators — a naming red herring, not this pattern.

### D8 — Maintenance pack: front-door post-load action, not projection injection

**The structural problem (verified)**: `campaign.campaignInventory` — what the repair bay renders via `selectRepairBay` — is populated only by the CLEANUP-phase `inventoryProjectionProcessor` on `advanceDay` and is absent from `serializeCampaign` output. A loaded maintenance pack renders an empty repair queue on first paint no matter how it was minted.

| Option | Verdict |
|---|---|
| (a) Scope the pack to unrepaired-damage state only; leave repair-queue rendering to flow-audit's (pack-ineligible) store-injection seeder | Rejected as primary: it silently gives up the pack's whole point (a busy repair bay as a named start state) and the subsystem's pack would assert almost nothing. |
| (b) `loadCampaignPack` supports a manifest-declared **post-load action**: after `goto`, drive the production Advance Day control once, so the CLEANUP processor populates the projection for real | **Chosen** — stays front-door-consistent (a real UI interaction on the mounted page, no store writes). Trade-offs recorded: the observed state is minted-state-plus-one-production-day (repair hours partially applied, a day's costs posted); the parity spec asserts at the post-advance point, and the pack's target state is documented as "day N+1 after load". |

`postLoadActions` is a closed manifest vocabulary (initially exactly `['advance-day']`); each action must be a front-door UI interaction. Adding a new action verb requires a delta to the front-door-loading requirement — the field is deliberately hard to grow so it never becomes a scripting escape hatch around mount seams.

### D9 — Parity binding and the gate rule

**Choice**: every pack declares `parityAnchorJourney` — the genesis journey whose live-reached state is the truth the pack caches:

- **Flow-checkpoint packs** bind `flow:<flowId>@<checkpoint>`; the pack's parity spec loads the pack and hard-asserts the same invariants the flow asserts at that checkpoint, at the same route.
- **Anchor-captured packs** bind `anchor:<spec-basename>` (the combat pack binds the fresh-construction anchor); the parity spec asserts the anchor's route-mount invariants (distinct per-instance unit identities, no unjustified terminal outcome, active status) on the pack-recovered session.
- **Fast-forward-minted packs** bind `fast-forward:<fixtureId>`; the parity spec asserts the loaded route surfaces agree with the invariant summary recorded at mint (from W3's run report), and the packs inherit W3's recorded consequence verbatim: while the `campaign-fast-forward-api` live-parity acceptance is failing or unproven, fast-forward-minted packs are triage-only.

**The gate rule (Prometheus's dissent contained)**: packs gate NOTHING until parity-proven — and even then, promotion of any pack into a PR gate requires an explicit delta to this capability's parity requirement plus the cited W3 acceptance green; near-term behavior is identical to the dissent's absence. The W2 anchors are never displaced regardless of parity results (journey-qc's "Packs and shortcuts never displace an anchor" scenario, landing via W2 — by ladder ordering that requirement is canonical before any W4 pack exists).

### D11 — Mint-time canonicalization: payloads are true id-templates; cross-store references are stripped; the calendar is not rewritten

Every minter mode ends with a canonicalization pass over the captured payload, before JSON is written:

1. **Id-graph canonicalization**: the wall-clock production ids the capture carries (`campaign-<epoch>-<rand>` + the root force id from `generateUniqueId`, `mission-<epoch>-<rand>` mission ids, the captured `matchId`) are rewritten to **deterministic, role-ordinal, pack-scoped template ids** — shape `pack:<packId>:<role>[:<NN>]` (e.g. `pack:navigation-briefing:campaign`, `pack:…:mission:01` in `missionNumber` order, `pack:…:match`). The exact scheme is the implementer's under four constraints: deterministic across re-mints of the same state, pack-scoped, visually unmistakable for either a production id or a load-time stamped id, and never URL-reachable (the loader always remaps before PUT/`goto`, so canonical ids never hit a route). Canonicalization uses the same graph-aware remap machinery as the load-time stamper (D4) — same id families, same every-occurrence application, same zero-residue scan: zero occurrences of any mapped-from id after the pass, and for campaign packs additionally no `campaign-|force-|mission-<epoch>-`-shaped wall-clock id surviving anywhere in the remapped families. Payloads thereby become **true id-templates**: two mints of the same deterministic state carry identical id graphs, and an intentional re-mint committed to git shows real drift in the PR diff, not id churn.
2. **Cross-store reference stripping**: roster-projection mission records carry `encounterId` and `gameSessionId` (`SerializedCampaign.ts:134-135`) — references into the encounter store and the match-log IndexedDB, rows that do NOT travel with a campaign pack. Remapping cannot fix them (the referenced rows don't exist in the loading browser); left alone they dangle silently — the load succeeds, parity passes (parity asserts genesis invariants at the target route), and mission-history affordances (view encounter, replay, session links) break later. **The canonicalizer strips both fields.** Both are optional on the interface, so absence is the legal pre-link shape — no schema violence. Recorded consequence: pack-loaded campaigns' mission history carries no encounter/replay links, and pack parity specs never assert those affordances. Task 1.2's inventory sweeps the remaining body fields (`pendingBattleOutcomes`, `processedBattleIds`, `dailyBattleAudit`, salvage records, …) for further references into either store: verified cross-store references join the strip list; campaign-internal idempotency markers stay.
3. **The in-fiction calendar is NOT canonicalized**: flow-created campaigns anchor their calendar at wall-clock creation time (`currentDate: new Date()`) and that anchor propagates into contract deadlines, market offers, and transaction dates; a post-hoc uniform date shift is fragile (sub-day components, cross-field consistency, day-boundary effects) and D7's invariant-level comparison does not need it. Fast-forward fixtures pin their start date at fixture level (fixture identity per W3 includes the start date). Recorded consequence: a committed re-mint of a flow pack churns date-valued fields in the PR diff — reviewers read the canonical id graph and the invariant summary, not raw dates.

## Risks

- **[R1] W2/W3 artifact drift at gate time** — helper names/paths/signatures may land differently than their declarative contracts state. → Gates 4.0/5.0 re-verify by grep against the then-current tree and STOP on mismatch; tasks cite the upstream task numbers so the report names exactly what drifted.
- **[R2] Shared SQLite under local `fullyParallel`** — id templating removes row collisions; it does not prove absence of write contention on the single shared DB file. Mitigations mirror W2 D6: unique ids, teardown, single-worker verify lane. The council's escalation (key `DATABASE_PATH` by workerIndex) remains a separate change (open risk 2), explicitly out of scope.
- **[R3] Half-stamped id graph** — the worst silent failure: a pack that loads but whose internal references point at template ids. → The remap table applies to every occurrence; the zero-residue scan is the mechanical guarantee; the stamper's unit tests include adversarial fixtures (id-bearing fields the schema doesn't enumerate) proving the residue scan throws.
- **[R4] Formatter/validator quote race** — per repo memory, the double-quote hook vs `oxfmt` single-quote breaks token-matching validators. → After any spec-delta edit: formatter, then `npx openspec validate add-scenario-packs --strict`, then touched validators' jest wrappers.
- **[R5] Ledger race at ship time** — W2's and W3's entries may archive off `origin/main` mid-flight. → The PR rebases onto current `origin/main` and lands the ledger with only the then-active entries; `qc:openspec-ci:validate` is the hard gate.
- **[R6] Payload bloat in git** — committed JSON grows the repo per re-mint. → Pilot packs are minimal named states (bounded day counts, small rosters); the recapture job reports payload sizes so growth is visible; binary/screenshot payloads are banned (packs are JSON only).
- **[R7] Scope creep into W5/W6** — viewport sweeps, subsystem lanes, PR gates, and the route-guard fix are adjacent and tempting. → All are explicit non-goals; any such edit is a change-scope violation.
- **[R8] Playwright discovery multiplication** — new specs under `e2e/scenario-packs/` auto-join chromium + all three responsive projects, silently 4×-ing runs. → One `'**/scenario-packs/**'` `testIgnore` glob per responsive project; `npx playwright test --list` proof in the tasks; no `@smoke` tags (the smoke project greps titles).
- **[R9] Recapture flake for journey-captured packs** — re-minting the combat pack replays a live Playwright journey in CI. → chromium-only at `--workers=1` inside the recapture job; the comparison is invariant-level (D7), never pixels and never byte-structural; a flaky genesis journey is a W2 anchor stability bug to fix there (its own 3×-green obligation), never a reason to hand-freeze the pack.
- **[R10] Campaign surfaces may need `/api/pilots` rows beyond the envelope's inline roster projection** — the projection restores client-side, but some surfaces may fetch pilots by id. → Group-1 verification answers it; if rows are needed, the loader creates them front-door (POST the production pilots route) with remapped pilot ids — the remap table already accommodates it; store injection is not the fallback.
