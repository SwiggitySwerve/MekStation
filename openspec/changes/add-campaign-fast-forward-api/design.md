# Design: add-campaign-fast-forward-api

## Context

The production campaign loop already contains every piece this change needs — none of them are connected end-to-end headlessly:

- `useCampaignStore` `advanceDayAction` (`src/stores/campaign/useCampaignStore.dayActions.ts:181-252`) calls `registerBuiltinProcessors()` then `getDayPipeline().processDay(campaign)`, running 20+ phase-sorted processors (`src/lib/campaign/processors/processorRegistration.ts:62-122`) including `scenarioGenerationProcessor` (Monday-gated + `useAtBScenarios`-gated, `scenarioGenerationProcessor.ts:224-232`) and `scenarioEncounterBridgeProcessor` (writes `campaign.bridgedEncounters`/`bridgedScenarioIds`, idempotent on `scenarioId`).
- `materializeCampaignMissionEncounter` (`src/lib/campaign/encounter/materializeCampaignMissionEncounter.ts:287`) already accepts an injectable `fetchImpl = fetch` and drives POST `/api/forces`, PUT `/api/forces/assignments/:id`, POST+PATCH `/api/encounters`, PUT `/api/encounters/:id/player-force|opponent-force` — all Pages Router handlers (`src/pages/api/...`, `NextApiRequest`/`NextApiResponse`).
- In-process invocation of those exact handlers from jest against real `:memory:` SQLite is proven working: `src/__tests__/api/forces/forceMultiAssignment.real.test.ts` imports the handler modules, drives them with `node-mocks-http` `createMocks`, and initializes via `getSQLiteService({ path: ':memory:' }).initialize()` — synchronous (`better-sqlite3`), no live Next server (`src/pages-modules/api/routeHelpers.ts:24-40`).
- The **only** production chain that runs `GameEngine.runToCompletion` end-to-end is the SP auto-resolve hook `usePreBattleLaunch.ts:186-262`: `buildPreparedBattleData` (real `adaptUnit`/CompendiumAdapter catalog pipeline, `src/components/gameplay/pages/preBattleSessionBuilder.ts`) → `new GameEngine({seed})` → `runToCompletion(player, opponent, gameUnits, linkage)` → `deriveCombatOutcome(session, {contractId, scenarioId})` → `publishCombatOutcome(event)`.
- The campaign store subscribes to the outcome bus on first construction (`useCampaignStore.ts:135-148`) and enqueues campaign-linked outcomes into `pendingBattleOutcomes`; `postBattleProcessor` drains the queue on the **next** `advanceDay()` (`postBattleProcessor.ts:295-332`), applying XP (`applyPilotDelta` → `awardScenarioXP`/`awardKillXP`), contract closure (`applyContractClosure`, `TransactionType.Income`, exactly-once via `processedFulfilledContractIds` + `flippedToTerminal`), salvage, and repair-queue building.
- **The engine→campaign pilot chain is severed (verified during authoring).** `deriveCombatOutcome` keys `unitDeltas[].unitId` by session unit id — `buildSessionUnitId` composes `${side}-${slotIndex + 1}-${unitRef}` (`preBattleSessionBuilder.ts:83-89`) and `unitToDelta` copies `unit.id` (`combatOutcome.ts:154,192-194`) — while `applyOutcomeDeltas` resolves roster entries with `entriesByPilotId.get(delta.unitId)` keyed on vault `pilotId` (`postBattleProcessor.ts:67,79-80`), and `IUnitCombatDelta` carries no pilot linkage (`CombatOutcome.ts:81-101`). `IGameUnit.pilotRef` (`GameSessionUnitTypes.ts:152`) already carries the vault pilot id through the launch path but is dropped at derivation. Result: a real engine-derived outcome yields zero `pilotsUpdated` (only the `applyPilotDelta` "Unknown pilot id" warning, `postBattleProcessor.helpers.ts:102-107`); every green XP test passes by hand-rigging `unitId === pilotId` (`'unit-A'`: `phase4CampaignRoundTrip.test.ts:236`, `campaignCombatLoop.test.ts:136`). This change carries the one sanctioned production fix for it (D9).
- **Scenario generation carries a third gate beyond Monday + `useAtBScenarios`**: each combat team rolls d100 against its role's base battle chance (`checkForBattle`, `battleChance.ts:120-138`); force-derived teams all get FRONTLINE = 20% (`scenarioGenerationProcessor.ts:93-97`, table `battleChance.ts:50-58`). Explicit `campaign.combatTeams` wins over derivation when present (`scenarioGenerationProcessor.ts:115-124`) — the fixture's lever for a seed-independent ≥1-bridged guarantee (D8). `getActiveContracts` filters on end date only, never status (`campaignCommandSelectors.ts:241-260`) — a terminal contract keeps generating scenarios, which D8's fixture bounds and R9 records as a production gap.
- `src/lib/campaign/fastForward/` does not exist; `assertSessionInflictedDamage` exists nowhere in `src/` (both verified). `src/__tests__/integration/phase4CampaignRoundTrip.test.ts` is the boot-the-world precedent (Zustand store seeding, five singleton resets in `beforeEach`/`afterEach`, synthetic outcomes via `publishCombatOutcome`).

Implements `specs/campaign-fast-forward-api/spec.md` (new capability) and the `specs/campaign-combat-loop/spec.md` delta.

## Goals / Non-Goals

**Goals:**
- Jest-speed, browser-free, live-server-free multi-day campaign advancement through the production day pipeline, with AtB Monday scenario generation reachable and asserted (≥1 scenario bridged, fail-loud expectations).
- Real-handler, real-engine battle execution per bridged scenario, closing the engine→campaign continuous-chain gap and shipping `assertSessionInflictedDamage` against the twice-recurred 0-damage class (#998, #1019).
- Economy/XP/maintenance invariants over fast-forwarded runs; deterministic reproducibility from the campaign's persisted seed; live-parity acceptance per council open risk 1.
- The substrate W4 pack minting dumps from.

**Non-Goals:**
- W4 packs, W2 anchors (untouched, never replaced), CI gating (W6), UI, production behavior changes beyond the D9 pilot-attribution fix, the R9 contract-closure production fix, XP-spend driving, scenario-composition-faithful OpFor, golden-trace determinism. See proposal Non-goals.

## Decisions

### D1 — New capability folder `campaign-fast-forward-api`, scoped to the orchestration layer only

**Choice**: A NEW capability owns the driver contract (loop semantics, expectations, invariants, damage guard, determinism, parity acceptance). One MODIFIED delta to `campaign-combat-loop` (transport-agnostic materialization). Nothing else.

**Alternatives surveyed and rejected:**
- *Own it under `campaign-management`* — rejected: that spec owns campaign creation/persistence/options/store composition, not the scenario→bridge→materialize→outcome pipeline; a fast-forward driver there would be a foreign body.
- *Own it under `campaign-combat-loop`* — rejected as owner: that spec owns the bridge/launch/enqueue **mechanics**; the fast-forward is an orchestrator *around* those mechanics plus invariant suites over three other subsystems (finances, progression, repair). Folding it in would balloon a mechanics spec into a test-methodology spec. `campaign-combat-loop` gets exactly one MODIFIED requirement instead.
- *Delta `simulation-system` for `runToCompletion`* — rejected: the engine is already specified as headless-invocable, seed-deterministic, and batch-runnable ("Batch Monte Carlo Runner", "Engine Determinism Contract", "SeededRandom for All Combat Randomness"). W3 consumes that contract; the new behavior is orchestration, not engine.
- *Delta `journey-qc`* — rejected: verified no browser/Playwright surface anywhere in W3 scope; journey-qc's domain is untouched.
- *Delta `personnel-progression` / finance specs for the XP/ledger invariants* — rejected: the math contracts are already speced and unit-tested there; what is NEW is "these invariants hold across a fast-forwarded multi-day engine-driven run", which this capability owns, cross-referencing rather than modifying.

**Duplicate-owner guard**: the new spec must NOT restate the bridge idempotency, force-sizing, launch-blocking, or outcome-enqueue requirements — `spec-purpose-lint`'s owner machinery treats `src/lib/campaign/encounter/` and `scenarioEncounterBridgeProcessor.ts` as `campaign-combat-loop` territory. The new requirements reference those contracts and add only fast-forward-specific obligations.

### D2 — The combat runner drives the mission materializer's REST surface, not `launchCampaignEncounter` (council-sketch drift resolved)

**The drift**: the council row's literal chain — `bridgedEncounters → materializeCampaignMissionEncounter(fetchImpl → in-process router)` — does not exist in production. Verified: `materializeCampaignMissionEncounter`'s one production call site is the roster-driven mission launch (`missionLaunchPage.launch.ts:232`), which never reads `bridgedEncounters`; `bridgedEncounters`' consumer is `launchCampaignEncounter` (direct-call `EncounterService`, no fetch layer), and its `EncounterService.launchEncounter` creates only a shell session via `buildGameUnitsFromEncounter` (DEFAULT-skill, non-catalog-adapted, `src/services/encounter/encounterToGameSession.ts`) — it never constructs a `GameEngine`. No production path runs a bridged scenario through `runToCompletion` today.

**Choice (Option A)**: per bridged scenario, the runner synthesizes a `CampaignMissionSource` view (the materializer's input is `Pick<ICampaign, 'id'|'name'|'missions'>`; the runner builds a one-mission map keyed by the scenario id) and calls `materializeCampaignMissionEncounter` **unchanged** with the in-process router as `fetchImpl`, then reads the created forces back through the same router (GET `/api/forces/:id`), builds units via `buildPreparedBattleData`, and runs the engine — mirroring `usePreBattleLaunch`'s auto-resolve chain step for step.

**Rejected (Option B)**: `launchCampaignEncounter` + new session→engine glue. It preserves the bridge's BV-matched `opForConfig` and scenario-derived map/victory config, and needs no fetch layer — but (a) it exercises zero API handlers, killing the council's drift-breaks-loudly goal and the parity between fast-forwarded state and live-created SQLite rows; (b) it requires NEW glue production code (no session→`IAdaptedUnit[]` bridge exists), where Option A requires none; (c) its shell session would be orphaned (created, never played) — a row the runner would have to explain or delete. Option A reuses two production surfaces unchanged; Option B modifies none but builds new plumbing beside one.

**Trade-off recorded**: the materializer stamps its own deterministic opponent selection (`selectOpponentUnits({count, seed: campaignId:missionId})`), radius-8 clear-terrain map, and DestroyAll victory — NOT the bridge's BV-matched OpFor or scenario-derived conditions (`buildEncounterFromScenario`). Fast-forwarded battles are therefore **campaign-linkage-faithful** (true `contractId`/`scenarioId` flow through the outcome so contract/scenario progress applies correctly) but **not scenario-composition-faithful**. Acceptable for W3's purpose — economy/XP/maintenance invariants and the damage guard depend on real combat with real linkage, not on OpFor fidelity. Scenario-faithful materialization over the REST surface is a candidate future change, noted in Open Questions.

### D3 — The in-process router imports the real handlers; drift breaks loudly

**Choice**: `fastForward/inProcessApiRouter.ts` exposes `createInProcessApiFetch(): typeof fetch`. It maintains a static route table mapping method+path patterns to the **imported real handler modules** (`src/pages/api/forces/index.ts`, `src/pages/api/forces/[id].ts`, `src/pages/api/forces/assignments/[id].ts`, `src/pages/api/encounters/index.ts`, `src/pages/api/encounters/[id]/index.ts`, `src/pages/api/encounters/[id]/player-force.ts`, `src/pages/api/encounters/[id]/opponent-force.ts` — implementer verifies the exact file set by grep before wiring). Each call builds `createMocks({method, body, query})` (`node-mocks-http`, the `forceMultiAssignment.real.test.ts` pattern), awaits the default-export handler, and adapts the mock response into a minimal `Response` (status, `ok`, `json()`) — exactly the surface `materializeCampaignMissionEncounter`'s `readApiJson` consumes. An unmatched route throws with the attempted method+path (fail loud, never 404-silently). SQLite is the real service at `:memory:` via the `routeHelpers` initialization path.

**Rationale**: the council accepted the router as a maintenance surface precisely because it imports real handlers — a renamed route, changed payload contract, or new required field breaks the fast-forward suite in jest instead of shipping drift. Mocked handlers or a parallel service-call re-implementation would rot silently, which is the failure mode packs-as-caches are designed against. The facts confirm in-process invocation is viable (real precedent test, synchronous `better-sqlite3` init, no server process) — so no honest-alternative fallback is needed.

**Placement**: the router ships in `src/lib/campaign/fastForward/` as library code (not under `__tests__/`) because W4's pack-minting runs consume it too; it has no Next.js runtime dependency beyond the handler imports themselves.

### D4 — Engine seeds derive from the campaign's persisted seed; determinism is invariant-level, never golden

**Choice**: per battle, `seed = deriveBattleSeed(resolveCampaignSeed(campaign), scenarioId)` — reusing the `campaignRng.ts` keyed-stream pattern (audit D-10) with a small FNV-style mix; each battle gets its **own** `new GameEngine({ seed })` (the `QuickResolveService` determinism-contract precedent: never share an engine across runs). The seed keys on the scenario's **stable identity only**: `buildScenarioId` already embeds the generation date, contract, and force (`scn-<contractId>-<dateKey>-<forceId>`, `scenarioGenerationProcessor.ts:149-156`), so the seed is a property of the scenario fixed at generation — NEVER of the day the battle is elected to run. This is load-bearing for D7's ordering variation: a fight deferred one day resolves with the identical seed, so ordering differences test orchestration, not dice. `Date.now()` never appears in the fast-forward module tree as a seed source — the exact footgun every current UI call site has (`seed: seedOverride ?? Date.now()`), which fast-forward must not inherit.

**What determinism is promised**: campaign day-processor randomness is already deterministic per audit D-10 (`createDailyRandom` keyed by campaign seed + day + processor id; verified no raw `Math.random` remains on that surface). Scenario ids, bridge output, and the materializer's opponent selection are deterministic. Engine combat is seed-deterministic post-W1 (`config.seed` stamped by `runToCompletion`). Therefore: **two runs of the identical fixture (same `rngSeed`, start date, options, roster) for the same day count produce equal invariant-level end states** — unit survival sets, armor/structure deltas, balance delta, transaction types+amounts, XP counters, scenario/contract statuses, repair-ticket counts/kinds. NOT promised: byte-identical state dumps — transaction ids, ticket `createdAt`, and `updatedAt` fields embed real wall-clock (`nowIso`), and are excluded from every equality assertion. No test may pin seed→outcome golden values (W1 D6's version-pin rationale applies verbatim).

**Council-brief correction folded in**: the council's "campaign scenario generation … own RNG source" concern is already resolved in the codebase (D-10 sweep); the only open determinism gap W3 closes is the engine-construction seed point. One verification obligation remains: read `src/simulation/core/SeededRandom.ts` and confirm no internal `Math.random` fallback before the design's determinism claims are asserted in tests (tasks 3.1).

### D5 — Outcomes travel the production bus and drain on the next day; the runner never applies outcomes directly

**Choice**: after `runToCompletion` returns, the runner calls `deriveCombatOutcome(session, {contractId, scenarioId})` + `publishCombatOutcome(event)` directly (`src/lib/combat/outcome/combatOutcome.ts`, `src/engine/combatOutcomeBus.ts`) — the same glue `publishAutoResolvedCampaignOutcome` composes, imported from the lib/engine layer rather than from the React-hook module (`usePreBattleLaunch.ts`), keeping the headless tree free of component-layer imports for the publish step. The campaign store's bus subscription enqueues into `pendingBattleOutcomes`; the **next** `advanceDay()` drains it through `postBattleProcessor`. Two hard rules fall out:

1. **Subscription liveness**: `useCampaignStore()` (the getter that lazily constructs the singleton and registers `subscribeToCombatOutcome`, `useCampaignStore.ts:135-148`) MUST be invoked before the first battle publishes, or outcomes vanish silently. The driver does this in its setup step and the fixture contract states it.
2. **Drain-before-return**: a battle fought after day K applies during day K+1. The driver advances additional drain day(s) until `pendingBattleOutcomes` is empty before returning, and records them in the report — so callers never receive a "green" run with un-applied outcomes.

**Rejected**: injecting outcomes into `campaign.pendingBattleOutcomes` directly (the `campaignCombatLoop.test.ts` fixture shortcut) — that is exactly the bypass this change exists to close; it skips the bus, the `isCampaignLinkedOutcome` filter, and the enqueue-dedup contract.

### D6 — `assertSessionInflictedDamage` contract: three failure modes, armor/structure-delta primary

**Choice**: `assertSessionInflictedDamage(session, outcome)` throws (with per-unit diagnostics) when any of:
1. **Zero-armor start** (#998 shape): any deployed unit whose starting armor+structure totals are zero — the record-sheet-empty-maps class.
2. **Zero cumulative damage on a completed battle**: every unit's remaining armor+structure (from the derived outcome's `unitDeltas`) equals its starting totals — **this condition alone trips the guard**, regardless of the event log (the armor/structure comparison matches the historical bug shape and needs no turn-count context, per the `sumLocations` pattern in `campaignMissionEncounterLaunchIntegrity.test.ts`). The event log is a *diagnostic classifier*, never an absolver: if damage-bearing events ARE present while deltas are unchanged, the failure names the **phantom-damage signature** — events emitted against ids that never mutated unit state, precisely the dual-id class (#1019 family) — distinguishing it from an event-free stalemate. An earlier draft AND-ed "no damage-bearing event" into the trip condition; that conjunction would have let phantom damage pass (events present + state untouched satisfies neither clause pair), which is the exact bug class the guard exists for. Rejected.
3. **Unjustified terminal outcome** (#1019 shape): a terminal outcome with no preceding combat/withdrawal/concession events — the same predicate `campaign-combat-loop`'s "Terminal outcomes require justifying events" scenario states, applied mechanically.

The runner applies the guard to **every** completed fast-forward battle unconditionally — it is a standing tripwire, not an opt-in assertion. A theoretically legitimate zero-damage turn-limit stalemate would fail it; that is intentional (fail loud, then decide), and softening requires a delta to the requirement.

### D7 — Live-parity acceptance compares against a discrete-step jest live-equivalent; the open-risk-1 consequence is recorded

**Honesty constraint (verified)**: no existing e2e journey drives real combat through the browser — `campaign-flow.spec.ts` / `campaign-economy-browser.spec.ts` seed pre-computed post-battle state via store injection for display/persistence assertions and stop before any launch click. There is no browser-side "live-played" oracle to compare against today, and building one is W2/W4 territory.

**Choice**: the parity acceptance runs the SAME fixture and seeds twice at jest level: (1) `fastForwardCampaign(N days)`; (2) a hand-cranked **live-equivalent loop** — new test scaffolding, one player-paced step at a time: advance one day; when a scenario bridges, materialize and fight it through the production primitives; advance the next day to apply; repeat — no shortcuts, no batching, mirroring the ordering a live player session produces. Equality is asserted on the **seam invariant set**: unit survival set, per-unit armor/structure deltas, finances balance delta, transaction types+amounts (ids/timestamps excluded), roster `xp`/`campaignXpEarned`/`campaignKills`/`campaignMissions`, scenario and contract statuses, repair-ticket counts/kinds.

**Anti-tautology rules** (without them the acceptance collapses into the 5.2 run-twice determinism test with different loop bookkeeping):

1. **Independence of the code under test**: the live-equivalent MUST NOT import `fastForwardCampaign` or `fastForwardCombatRunner` (or any fast-forward orchestration helper) — its day stepping, battle election, and fight-then-apply sequencing are hand-inlined directly against the production primitives (store `advanceDay`, `materializeCampaignMissionEncounter`, `buildPreparedBattleData`, `GameEngine`, `deriveCombatOutcome` + `publishCombatOutcome`). Shared surface is limited to three modules whose sharing is either a precondition of the comparison or independently unit-tested: the fixture builder (identical fixture is the premise), `deriveBattleSeed` (identical seeds are the premise — an independently re-derived formula would be the same code twice, buying nothing), and the in-process API router (transport, unit-tested per-route in group 1).
2. **Honest coverage claim**: with those three shared, a bug INSIDE a shared primitive (wrong scenario fought by the materializer, broken handler, mis-derived seed) passes both sides identically — the acceptance certifies the fast-forward **orchestration** (day batching, subscription liveness, drain timing, battle ordering), not the primitives. The spec requirement states this bound explicitly so a green parity run is never read as validating more than it does; the primitives' own suites (router unit tests, materializer contract scenarios, W2 anchors) own primitive correctness.
3. **Ordering variation**: identical-ordering parity alone under-tests player realism, so the suite includes at least one live-equivalent variant that defers fighting a bridged scenario by ≥1 day within the same total day window (players don't always fight the day the scenario lands). D4's generation-day-stable seed makes this meaningful: the deferred fight uses the identical seed, so any divergence is orchestration-order-sensitivity, not dice. If a variation legitimately diverges (a production semantics reason, e.g. a window-edge interaction), the parity claim is narrowed to identical-ordering-only via an explicit delta to the requirement — never silently.
4. **Day-count pin**: both sides advance the SAME total in-fiction days (drain days included) and end on the same in-fiction date. Daily cost transactions post per day, so any day-count mismatch produces a guaranteed balance-delta divergence that is pure red herring; the suite asserts the day counts match before comparing invariants.

Because processor registration order is phase-sorted and deterministic, and both sides derive the same battle seeds, the seam fields are genuine must-match fields — a divergence means the fast-forward's orchestration does not reproduce player-paced progression, which is precisely council open risk 1.

**Consequence stated**: per the council decision, if fast-forwarded states diverge from live-played states on seam invariants, **packs stay triage-only permanently** (Prometheus's crux). The spec requirement records that consequence so W4 cannot quietly promote packs to gating on a failed or skipped parity acceptance. What this acceptance does NOT claim: browser-mount parity — the W2 anchors keep that authority; the parity fields are deliberately the same aggregates the anchors assert at the route mount, so W4 can later close the loop end-to-end.

### D8 — Test hosting: existing `unit` jest project, `phase4CampaignRoundTrip` boot pattern, no new lane

**Choice**: suites land as `src/lib/campaign/fastForward/__tests__/*.test.ts` (colocated units) plus `src/__tests__/integration/fastForwardCampaign*.test.ts` (capstone, determinism, parity) — all matched by the existing `unit` project's `__tests__` glob (verified: no separate integration project exists; directory naming is convention only). Boot/reset follows `phase4CampaignRoundTrip.test.ts`: `resetCampaignStore`, `_resetCombatOutcomeBus`, `_resetContractFulfilledBus`, `_resetDayPipeline`, `_resetBuiltinRegistration` in `beforeEach`/`afterEach`, plus SQLite `:memory:` re-init per test and `useCampaignRosterStore.setState` fixture seeding (the `campaignCombatLoop.test.ts` `seedRoster` precedent). The jest environment docblock matches whatever `forceMultiAssignment.real.test.ts` declares (implementer verifies — `better-sqlite3` must load).

**Fixture contract**: the shared fixture builder explicitly sets `useAtBScenarios: true` (default false — mandatory or generation never fires), makes an explicit `useRoleBasedSalaries` choice per test (default false exercises `dailyCostsProcessor`; true exercises `financialProcessor` — never assume defaults exercise the intended path), pins `rngSeed`, pins a start date whose window covers ≥1 Monday, and seeds an active contract + combat teams whose forces resolve. Two constraints added after review:

- **Seed-independent battle guarantee**: the third generation gate is the per-team d100 battle-chance roll (Context). The fixture declares **explicit `campaign.combatTeams`** (which beat FRONTLINE-only force derivation, `scenarioGenerationProcessor.ts:115-124`) using the highest-base-chance role (`PATROL` = 60; `checkForBattle` reads `BASE_BATTLE_CHANCE[team.role]`, ignoring the `battleChance` field) with a team count × covered-Monday count sized so P(zero battles) is negligible for any seed — e.g. 8 PATROL rolls give 0.4^8 ≈ 0.07%, and the builder documents the arithmetic. The pinned seed exists for determinism (D4), never for battle occurrence; after an upstream draw shifts the daily 'scenario-generation' stream, re-pin freely — the roll budget keeps the ≥1-bridged expectation practically certain, so no cherry-picking is ever needed and an expectation miss indicts the battle-chance gate by name.
- **Bounded contract windows** (economy exactly-once, R9): each contract's `endDate` (or the fixture's Monday coverage) is set so no scenario can be generated for the contract after it first reaches terminal status — one live Monday per contract is the simple shape. Without this, production's status-blind `getActiveContracts` lets a terminal contract fight again, and a terminal-to-terminal flip re-pays it (R9) — a seed-lottery flake in any multi-Monday window.

**Layering note**: the runner imports `buildPreparedBattleData` from `src/components/gameplay/pages/preBattleSessionBuilder.ts` — a component-layer import into lib code. Accepted deliberately: sameness with the live construction path is the point (the #998/#1019 classes live in construction), and relocating the builder is a production refactor out of W3 scope. Recorded in Open Questions as a candidate hoist.

### D9 — Sanctioned production fix: engine-derived outcome pilot attribution

**The problem (Context, verified)**: engine-derived deltas key by session unit id; post-battle application resolves roster entries by vault pilot id; no linkage field exists. The real chain grants zero pilot XP — the XP capstone ("roster XP fields move" without hand-built outcomes) is unimplementable against unmodified production, and the only workaround is rigging fixture pilot ids to session-id-shaped strings, i.e. institutionalizing the dual-id masking this change exists to catch.

**Choice**: one narrow, additive, fallback-preserving production fix, specified as the `campaign-combat-loop` delta's ADDED requirement "Engine-Derived Outcome Pilot Attribution":

1. `IUnitCombatDelta` gains optional `readonly pilotRef?: string | null` (`src/types/combat/CombatOutcome.ts`) — additive; no version bump needed for an optional field, implementer confirms against `COMBAT_OUTCOME_VERSION` consumers.
2. `unitToDelta` populates it from `IGameUnit.pilotRef` (`src/lib/combat/outcome/combatOutcome.ts` — the unit is already in hand at derivation).
3. `applyOutcomeDeltas` resolves the roster entry AND the vault pilot by `delta.pilotRef` when present, falling back to `delta.unitId` when absent (`src/lib/campaign/processors/postBattleProcessor.ts:67,79-80`). Kill attribution lookup (`reportUnitsById.get(delta.unitId)`) stays keyed by session unit id — report rows are session-keyed and already consistent.

**Backward compatibility**: linkage-free outcomes (persisted saves, every existing hand-built fixture) hit the fallback and behave byte-for-byte as today — the existing suites MUST stay green unmodified, which is itself the compat gate. Opponent/NPC `pilotRef`s that match no roster entry take the existing skip-with-warning path (`applyPilotDelta` entry===null).

**Alternatives rejected**:
- *Rig fixture pilot ids to session-id shapes* — assertion theater; masks the #1019 family; banned outright by the fast-forward spec's XP requirement.
- *Parse `${side}-${slot}-${unitRef}` in the processor* — couples the campaign layer to a session-id format that is explicitly session-scoped; breaks the moment the composition changes; still doesn't reach a pilot id.
- *Re-scope the XP requirement to what unmodified production grants (zero)* — guts the change's highest-value coverage (the council's stated priority) and leaves a known-severed production chain unfixed while a spec claims to cover it.
- *Full unit-id unification refactor* — correct long-term, but a cross-cutting production change (session builder, outcome schema, repair keying) far beyond W3; noted in Open Questions.

- **[R1] Handler/Response shim mismatch** — `node-mocks-http` responses are not `fetch` Responses; a subtle mismatch (missing `ok`, unparsed JSON) would fail materialization confusingly. → Mitigation: the shim implements exactly the surface `readApiJson` consumes, unit-tested against every route in the table; the `forceMultiAssignment.real.test.ts` precedent proves handler-side viability; unmatched routes throw with method+path.
- **[R2] Store-singleton lifecycle in jest** — a driver run that forgets subscription liveness (D5) or a suite that skips a singleton reset leaks state across tests and drops outcomes silently. → Mitigation: the driver owns the `useCampaignStore()` touch internally; the capstone suite asserts `battles fought > 0 AND outcomes applied == battles fought`; reset pattern copied from `phase4CampaignRoundTrip`.
- **[R3] Wall-clock fields poison equality assertions** — transaction ids embed `nowIso`; naive deep-equals between runs flakes. → Mitigation: the invariant comparator strips/excludes the enumerated wall-clock-bearing fields (D4); a non-wall-clock divergence is a real determinism bug to fix, never to mask in the comparator (W1 R3's rule verbatim).
- **[R4] Ledger sign-convention error** — `Transaction.amount` stores unsigned magnitude; per-`TransactionType` sign application is required for reconciliation, and a naive sum silently "reconciles" wrong. → Mitigation: the reconciliation helper maps each `TransactionType` to its sign explicitly and fails on unknown types.
- **[R5] Zero-damage false positive** — a legitimate turn-limited stalemate would trip `assertSessionInflictedDamage`. → Accepted: the guard's job is to fail loud on the twice-recurred class; a legitimate trip requires a spec delta to soften, never a silent skip.
- **[R6] Formatter/validator quote race** — per repo memory, the double-quote hook vs `oxfmt` single-quote breaks token-matching validators. → After spec-delta edits, run the formatter + `npx openspec validate add-campaign-fast-forward-api --strict` together.
- **[R7] Stale ledger at ship time** — this worktree's `active-change-ledger.json` predates PR #1041 (W1 archived on `origin/main`). → The PR rebases onto current `origin/main` and lands the ledger with only the then-active entries; `qc:openspec-ci:validate` is the hard gate.
- **[R8] Scope creep into W4/W6** — pack dumping, CI wiring, and browser parity are adjacent and tempting. → All three are explicit non-goals; any such edit is a change-scope violation.
- **[R9] Terminal-to-terminal contract re-payout (production gap, discovered during authoring)** — `getActiveContracts` never filters status (`campaignCommandSelectors.ts:241-260`), so a terminal contract keeps generating Monday scenarios; a second battle whose result differs flips terminal→terminal with `flippedToTerminal === true` (`postBattleProcessor.helpers.ts:334-338` — the predicate is `previousStatus !== nextStatus`, not previous-was-non-terminal), `applyContractClosure` has no once-only guard (`contractClosure.ts:33-56`), and `processedFulfilledContractIds` appends without a membership check (`postBattleProcessor.ts:210-212`): double payout + duplicate ledger entry, seed-lottery-triggered on multi-Monday windows. → Mitigation here: fixtures bound contract windows (D8) so the shape is unreachable in the suites, and the economy invariant claims only production's real guarantee (per-flip + same-matchId dedup). Fixing the gap is a separate production change, recorded in Open Questions — this change's spec must not paper over it with an invariant production cannot honor.
- **[R10] Pilot-attribution fallback regression** — the D9 fix touches the post-battle hot path; a botched fallback would break every persisted linkage-free outcome. → Mitigation: fallback branch unit-tested both ways (linkage present / absent / unresolvable), and the pre-existing hand-built-fixture suites (`campaignCombatLoop`, `phase4CampaignRoundTrip`, `postBattleProcessor.retryIdempotency`) MUST pass unmodified — any needed edit to them is a red flag that compat broke.

## Migration Plan

Additive except for one sanctioned production fix: a new module tree under `src/lib/campaign/fastForward/`, new test suites, two spec deltas, one ledger entry, plus the D9 pilot-attribution fix (optional field + populate-at-derivation + prefer-with-fallback resolution across three files — linkage-free outcomes behave exactly as today, so persisted saves need no migration). Rollback = delete the new files and the ledger entry and revert the three D9 touches (safe: the field is optional and unread once the preference branch is reverted). Implementation is deferred (declarative-first PR) and starts after the W2 implementation merges, per the ladder's strict sequencing.

## Open Questions

- **Scenario-composition-faithful materialization** (D2 trade-off): should a future change materialize the bridge's BV-matched `opForConfig` and scenario-derived map/victory config over the REST surface so fast-forwarded battles reproduce AtB composition? Deferred — not needed for W3's invariants; would also serve the live mission-launch path.
- **`buildPreparedBattleData` hoist** (D8 layering note): candidate future refactor moving the construction path from the component layer into `src/lib/`, benefiting both this runner and `usePreBattleLaunch`. Out of W3 scope.
- **Bridged-encounter player-force resolution**: `buildEncounterFromScenario` records a `playerForce` reference only when `campaign.forces.get(teamId)` resolves; the fast-forward fixture must guarantee a resolvable combat-team force. Whether the runner should also fight `Draft`-status bridged encounters (no resolvable force) by substituting the roster projection is left to implementation — the spec requires only that expectation-declared battles happen and fail loud otherwise.
- **Contract-closure exactly-once production fix** (R9): should `applyContractDelta`/`applyContractClosure` gate on `previousStatus` being non-terminal (or `processedFulfilledContractIds` membership), and should scenario generation use the status-aware contract selector? Both look like the correct fix; out of W3 scope, needs its own change with save-compat analysis.
- **`unitCombatStates` session-id keying** (D9 sibling): `applyUnitDelta` writes `campaign.unitCombatStates[delta.unitId]` — also session-unit-id keyed. Internally consistent (repair reads the same keys), but it means unit combat state is keyed by per-session composites rather than roster unit ids, so cross-battle state for the same roster unit does not accumulate under one key. D9 deliberately does NOT touch unit-state keying (pilot resolution only); implementers hitting this during the maintenance invariants should surface it rather than widen D9's scope.
