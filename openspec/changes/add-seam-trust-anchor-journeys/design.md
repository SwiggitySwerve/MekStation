# Design: add-seam-trust-anchor-journeys

## Technical Approach

Build the three council-designated seam trust anchors as route-mounted Playwright journeys that seed *state*, never *seams*: state may enter through the front door (direct IndexedDB seeding, live wizard/API creation), but every anchor crosses its guarded seam through the exact production path — a cold `page.goto` for the recovery factory, a real launch click for the materialization handoff, a real `?seed=N` pre-battle mount for construction. Nothing new is invented on the product side: the anchors assert already-shipped behavioral contracts (`campaign-combat-loop` "Campaign-Linked Encounter Launch" from PR #1015, "Launched Campaign Sessions Start Battle-Ready" from PR #1019) plus the W1 determinism contracts, at the layer where repo history proves jest cannot see (#1019: "jest passing while live fails means the test skipped the route-mount recovery path"). The change is declarative-first; implementation is strictly sequential (one worker per group) and starts only after the W1 implementation merges.

## Architecture Decisions

### Decision: D1 — Recovery anchor hardens `e2e/active-session-recovery.spec.ts` in place; the mirrored-roster fixture is the load-bearing choice

**Entry mechanism**: direct IndexedDB seeding via `page.evaluate` + raw `indexedDB.open` against `mekstation-match-log` v2 — the exact mechanism the existing spec's inline `seedMatchLog` already uses (`e2e/active-session-recovery.spec.ts:1-244`), writing `matchEvents` records `{matchId, sequence, event, savedAt}` (a `GameCreated` event MUST be sequence 0 — `hydrateRecoverableSessionFromMatchLog` throws `InteractiveSessionRecoveryCorruptError` otherwise, `src/engine/InteractiveSession.persistence.ts:57-81`) and a `matches` record `{matchId, hostPeerId, guestPeerId, status: 'active', lastActivity}` (schema: `src/lib/p2p/matchLogStorageSchema.ts:3-38`). Then a **cold** `goto('/gameplay/games/:id')` — no prior store state — which forces the full production chain `useGameSessionLifecycle` → `loadRouteSession` → `loadSession` → `recoverInteractiveSession` → `hydrateRecoverableSessionFromMatchLog` → `fromSessionAsync` → `deriveAdaptedUnitsFromSession`, which is precisely what makes seed-then-goto exercise recovery *harder* than click-through (council Phase 3, point 1a).

**Choice — extend, not fork**: the anchor is the existing spec file, extended with a mirrored-roster test; no new `e2e/seam-recovery-*.spec.ts` is created. The existing file already implements seed → cold goto → assert → reload → re-assert and is already wired into `verify:qc:replay-recovery` (`package.json:181`). Forking would duplicate the seeding machinery and split the recovery trust surface across two files. The inline `seedMatchLog` is extracted to `e2e/helpers/matchLogSeeding.ts` (pure move) so both tests and future W4 `loadEncounterPack` work share one seeder.

**The load-bearing fixture decision**: the new test's seeded roster MUST mirror at least one canonical `unitRef` across sides — e.g. both player and opponent fielding `atlas-as7-d` under distinct deployed ids (`player-1-atlas-as7-d` / `opponent-1-atlas-as7-d`). The existing fixture uses two units with DIFFERENT `unitRef`s and would not have caught PR #1019's recovery-side collision (fix: `adapted.push({ ...adaptedUnit, id: gameUnit.id })` in `deriveAdaptedUnitsFromSession`, `git show 91104d43a`). Without mirroring, the anchor is generic recovery coverage, not a #1019-class trust anchor.

**Hard invariants asserted** (via the exposed gameplay store, `src/lib/e2e/storeExposure.ts:29-51`): `isLoading === false`, `error === null`, `session.id === matchId`, `currentState.status === 'active'`, `interactiveSession` non-null; for EVERY deployed id in the mirrored roster `getMovementCapability(deployedId)` non-null; `getMovementCapability(bareCanonicalRef)` null (mirrors the #1019 regression jest at `src/engine/__tests__/InteractiveSession.recovery.test.ts:243-294`, now at route level); unit count === full seeded roster count; no terminal-outcome event beyond the seeded log; all re-held after `page.reload()`.

**Determinism source**: the fixed seeded event-log fixture. Pre-W1, the anchor asserts ONLY structural post-recovery state and drives no dice-consuming continuation — today `fromSessionAsync` hardcodes `SeededRandom(0xc0ffee)` and passes no `d6Roller` (`src/engine/InteractiveSession.ts:322-346`), so any continuation rolls `Math.random` dice and would flake. The route-level recover-twice continuation scenario activates only with W1's "Recovered Session Dice Re-Seeding" contract (persisted `config.seed`, re-seed from position zero) and is gated in tasks group 4.

### Decision: D2 — Handoff anchor is a new spec entering through the live click-through; hard assertions replace the C4 soft-finding class only in the anchor

**Entry mechanism**: live click-through, deliberately NOT seeded past the seam — the seam IS the client half of `launchSinglePlayerMissionFromPage` → `materializeCampaignMissionEncounter` → `router.push(campaignEncounterHref(...))` (`src/pages-modules/gameplay/campaigns/missionLaunchPage.launch.ts:209-252`). The journey creates a fresh campaign through the wizard (unique `Date.now()`-suffixed name), accepts a contract, opens the mission launch briefing, selects ALL available roster units via the `mission-readiness-panel` checkboxes capturing the selected count, clicks `launch-mission-direct`, and lands on `/gameplay/encounters/:id` → pre-battle — the same verified click-path as `e2e/ux-deep-play-audit.spec.ts:423-513` (`acceptContractAndOpenLaunch` / `launchMissionToPreBattle`), whose testids (`mission-launch-briefing`, `mission-readiness-panel`, `launch-mission-direct`, `pre-battle-page`, `player-force-card`, `player-unit-list`, `player-force-bv`) are all confirmed live in `src`.

**Hard invariants asserted**: `player-unit-list` rendered unit count `=== selectedUnitCount` (the exact collapse signature the audit only soft-records at `ux-deep-play-audit.spec.ts:515-536` — `collapsed := selectedMany && (/1\s+unit/ || /0\s+Battle Value/ || playerUnitListCount === 1)` feeding `driver.finding`, no `expect`); `player-force-bv` matches a nonzero-BV pattern; each rendered unit row exposes the pilot-assignment indicator — the pre-battle rows render a literal `(pilot assigned)` marker plus unit name and position, NOT the pilot's name (`src/components/gameplay/pages/PreBattlePage.sections.tsx:119-143`; `IForceSummary` carries only aggregate `avgGunnery`/`avgPiloting`, no per-unit pilot fields, `src/utils/gameplay/forceSummary.ts`), and pilot names first render post-launch in battle via the real SP producer (#1015) — so the DOM clause is scoped to the indicator while pilot IDENTITY is carried by the data-layer cross-check (group-1 audit determines whether the indicator has a stable testid; if not, adding one on the EXISTING marker is the change's only sanctioned product touch — adding a rendered pilot-name element would be a product display change and is out of scope); and a data-layer cross-check — capture the encounter id from the URL, resolve the player force id via the live API, `GET /api/forces/:id` (returns `{ force: IForce }`, `src/pages/api/forces/[id].ts`), assert exactly N assignments with N distinct unit ids and pilots preserved. The DOM count and the API count catch different collapse shapes; asserting both closes the "unit count in === unit count out" invariant end to end.

**Boundary with the audit journeys**: the deep-play audit spec is NOT modified, refactored, or converted to hard assertions — its capture-tolerant contract (journey-qc "Single-Player Campaign Deep-Play Journey": "MUST NOT hard-assert outcomes that are currently broken") is a different instrument (triage breadth) from the anchor (blocking depth). The anchor implements its own helpers in a new shared module (`e2e/helpers/seamCampaign.ts`) mirroring the audit's verified click-paths rather than importing from a spec file; converging the audit spec onto the shared helpers is deliberately deferred (touching the umbrella is a W2 non-goal).

**Cleanup (new surface)**: `materializeCampaignMissionEncounter` creates durable `/api/forces` ×2 and `/api/encounters` rows it never deletes; the existing campaign-flow cleanup pattern (`e2e/campaign-flow.spec.ts:637-658` — module-level id set + `test.afterAll` with a standalone `APIRequestContext`) only deletes campaigns. The anchor extends the pattern to DELETE the created forces and encounter (handlers verified: `src/pages/api/forces/[id].ts:64`, `src/pages/api/encounters/[id]/index.ts:65`), keeping the afterAll-not-afterEach discipline that spec already documents (page/context fixtures are rejected in afterAll on Playwright 1.57; standalone request context required).

### Decision: D3 — Construction anchor materializes once and launches twice with the same `?seed=N`; player-side mirrored refs make the per-instance id path unconditional

**Entry mechanism**: reuse D2's helpers to materialize ONE fresh NvN encounter, then two independent page loads of `/gameplay/encounters/:id/pre-battle?seed=1337`. A fixed literal seed in the spec is legitimate — the assertions are structural invariants and run-to-run equality, never a pinned seed→outcome mapping, consistent with W1's "Determinism Tests Assert Invariants, Not Golden Traces" (dice-system delta). Materializing once matters: opponent selection is seeded by `${campaign.id}:${missionId}` (`materializeCampaignMissionEncounter.ts:339-342`), so two separately-created campaigns would field different OpFors and the two runs would not be comparable; one encounter + two seeded launches holds units constant and varies nothing.

**The mirrored-ref guarantee lives player-side**: whether the deterministic opponent selector mirrors the player's canonical refs depends on data, so the journey cannot rely on cross-side mirroring. Instead the campaign roster is seeded so the player force fields ≥2 units sharing one canonical `unitRef` — under the journey's own control — which forces the `{side}-{slot}-{canonicalRef}` per-instance id machinery (`src/components/gameplay/pages/preBattleSessionBuilder.ts`, `buildSessionUnitId`, PR #1019) to disambiguate regardless of what the OpFor fields. The 2N-distinct-ids assertion is unconditional either way (it catches any collapse), but the mirrored player pair makes the #1019 path provably exercised, not data-dependent.

**Hard invariants asserted**: immediately post-launch, 2N distinct per-instance unit identities counted by distinct ids from the unit testids (`unit-token-${id}` / `rail-unit-${id}`, per the #1019 root-cause note), NOT DOM node totals — N-not-2N is the collapse signature; advance out of Initiative and through ≥2 further rounds, asserting after each advance no terminal outcome (no victory/defeat/draw surface) absent justifying events and both sides retaining ≥1 alive unit — the live-route counterpart of `campaign-combat-loop` "Launched Campaign Sessions Start Battle-Ready", and strictly stronger than the #1019 jest's single `advancePhase()` (`campaignMissionEncounterLaunchIntegrity.test.ts:223` — which also mocks `adaptUnit` and never mounts the route); then the second same-seed launch, asserting per-round outcome markers (phase, status, alive counts per side) equal between the two runs.

**Relationship to W1's own e2e (task 5.1 there)**: W1 ships an auto-resolve same-seed outcome-equality spec. This anchor does not duplicate it — it covers the campaign-materialized construction path with mirrored player refs and mid-battle structural assertions W1's outcome-level check cannot express. Both specs may coexist; if W1's 5.1 lands first, this anchor cites it rather than absorbing it.

**W1 dependency (hard gate)**: `?seed=N` does not exist on this branch — `usePreBattleLaunch.ts` hardcodes `seed: Date.now()` (line 224) and `pre-battle.tsx` parses no seed param; W1 tasks 4.1/2.1/1.2 create the contract. Tasks group 4 opens with an explicit gate check and STOPs if the contract is absent.

### Decision: D4 — Spec deltas live in `journey-qc` only: 1 MODIFIED + 4 ADDED; no new capability folder

Decision matrix (adding a capability folder is a deliberate architectural decision and is surfaced here explicitly):

| Option | Verdict | Why |
|---|---|---|
| New capability folder (e.g. `seam-anchor-qc`) | **Rejected** | The anchors own no product behavior — they are journey-level QC coverage architecture, which is `journey-qc`'s literal Purpose ("repeatable journey-level QC for major MekStation user flows"). The council Phase 3 framing treats all three seams as test-process architecture, and `journey-qc` already holds the precedent of registering named live E2E journeys outside the seven-journey catalog ("Playable command screen E2E journeys", spec line 249). A new folder would also need a ≥50-char real Purpose under `openspec validate --strict` for no owned behavior. |
| Home in `e2e-testing` | **Rejected as primary home** | Plausible (it registers specific Playwright specs, e.g. "Two-Browser Co-op Campaign Continuity Journey") — but the council directive and open-risk 3 name `journey-qc`, and splitting the three anchors from the journey-coverage doctrine they extend would scatter the seam story across two capabilities. `e2e-testing`'s "Deep-Play Playwright Project Scoping" requirement is intentionally left untouched: it governs the two audit specs; the anchors' identical single-desktop-project obligation is stated by the new umbrella requirement for its own three specs (no contradiction, each capability contracts its own specs). |
| `journey-qc` ADDED-only | **Rejected** | Leaves the canonical "Readiness and customizer journey validates deployment handoff" scenario overclaiming ("verify encounter materialization receives the selected campaign roster") — verified false today: `campaign-customizer-handoff.spec.ts` ends at the enabled launch button (file ends line 345, no click). The MODIFIED truth-repair re-scopes that journey to the enabled-launch boundary and delegates the crossing to the new anchor. Council open-risk 3 also expects journey-qc MODIFIED deltas to accompany W2. |
| `journey-qc` 1 MODIFIED + 4 ADDED | **Chosen** | Truth-repair + umbrella doctrine + one requirement per anchor, each single-concept. Behavioral invariants are cited from `campaign-combat-loop` / `game-engine-orchestration` (single source of truth), never duplicated as new product SHALLs. |

**Collision analysis vs the other active change**: `add-flow-audit-routines`' journey-qc delta contains ONLY two ADDED requirements ("Flow-Scoped Walkthrough Invocation", "Interactive Evidence and CI Authority Split") — neither header exists canonically, neither is touched here, and this change's MODIFIED target ("Playable command screen E2E journeys") appears in neither delta. The two changes' journey-qc deltas are header-disjoint in every merge/archive order.

**Full-replacement fidelity**: MODIFIED deltas replace the whole requirement at merge; the delta reproduces all four canonical scenarios, three byte-identical, only the readiness scenario and the requirement paragraph amended.

### Decision: D5 — Assertion doctrine: hard invariants, no golden traces, no pixel diffs

Anchors use blocking `expect()`s on structural invariants (counts, ids, statuses, capability lookups, API row shapes) and run-to-run equality for determinism — never pinned seed→outcome values (W1's version-pin rule: any engine change may legally reshuffle what seed 1337 produces; same-seed self-consistency within one build is the durable invariant), and never screenshot/pixel comparisons (the anchors guard seams, not layout; visual baselines are win32-pinned and are W5's problem space). Capture-tolerant `driver.finding` recording is FORBIDDEN inside anchors — that mode exists for audit breadth and directly contradicts trust-anchor semantics. Selector discipline: `data-testid` anchors and distinct-id counting, no text-regex outcome sniffing except where a testid genuinely does not exist and cannot be added (none currently known).

### Decision: D6 — Isolation and parallel-safety: per-run unique ids, extended teardown, single-worker lanes; browser-context isolation for the recovery anchor

The suite runs one shared Next.js dev server + one shared SQLite DB (`playwright.config.ts:148-166`), `fullyParallel: true` with auto workers locally (`:40`, `:49`). The anchors therefore: (1) mint unique per-run identifiers for every server-persisted row (campaign names/ids `Date.now()`-suffixed, per the `campaignSeeders.ts:163` pattern) so cross-spec collisions cannot occur even under local parallelism; (2) clean up in `test.afterAll` via a standalone `APIRequestContext`, extended beyond campaigns to forces and encounters (D2); (3) run at `--workers=1` inside their dedicated lane. The recovery anchor is naturally the most isolated: IndexedDB is per-browser-context and `/gameplay/games/:id` recovery touches no SQLite rows — zero server-side state, zero cleanup surface. PR #1032's isolation proof ran at workers=1; true-parallel contention remains council open risk 2 with the documented escalation (key `DATABASE_PATH` by workerIndex) explicitly out of W2 scope.

### Decision: D7 — Registration: one `verify:qc:seam-anchors` lane, `testIgnore` scoping, no catalog/graph/CI entries

One package script following the `verify:qc:replay-recovery` precedent (`package.json:181`): targeted regression jests first (`src/engine/__tests__/InteractiveSession.recovery.test.ts` — the #1019 recovery-side aliasing suite; `src/__tests__/integration/campaignMissionEncounterLaunchIntegrity.test.ts` — the producer-side integrity suite), then the three anchor specs via `node scripts/playwright/run-playwright.mjs test --project=chromium <spec> --workers=1`. One lane rather than three keeps W6 a single wiring hook; individual specs stay directly runnable. The three spec files are added to the `testIgnore` arrays of Mobile Chrome / Tablet Portrait / Tablet Landscape (`playwright.config.ts:102/117/130`, mirroring the two audit-spec exclusions) — seam anchors are not viewport-layout journeys, and without this every bare `npm run test:e2e` runs them 4× for zero coverage benefit. Note the intended side effect: `active-session-recovery.spec.ts` currently DOES run under all four projects; scoping it to chromium is a deliberate cost cut sanctioned by the umbrella requirement. NOT registered: the seven-journey catalog (`requiredJourneyIds`, exactly-seven contract in `scripts/qc/journey-qc-core.mjs:24-31` — adding entries breaks the validator), the QC validation graph (not universally required for e2e specs; existing precedent is prose-only), and any CI workflow (nightly `e2e-full` sweeps unfiltered specs in automatically; per-PR gating is W6).

## Risks

- **R1 — W1 ordering.** The construction anchor and the recovery continuation-determinism scenario consume contracts that exist only declaratively at authoring time. Gate: implementation starts after the W1 implementation merges; tasks group 4 opens with a mechanical contract check (grep `IGameConfig.seed`, `?seed` parsing in `pre-battle.tsx`, `config.seed` re-seed in `fromSessionAsync`) and STOPs on absence. The recovery (structural) and handoff groups are W1-independent by construction.
- **R2 — Shared SQLite under local `fullyParallel`.** Mitigated by unique-id minting, afterAll teardown, and single-worker lanes; NOT solved in general (council open risk 2). If the anchors flake under bare parallel local runs, the escalation is per-worker `DATABASE_PATH` keying — a separate change, not a W2 patch.
- **R3 — Flake budget.** Live journeys are the flake-prone tier. Discipline: bounded `expect.poll`/web-first assertions only, no fixed sleeps, no retry loops masking nondeterminism; every anchor must pass 3 consecutive local runs (umbrella requirement scenario) before registration and after any modification. A flake traced to product nondeterminism (e.g. a resolver reaching `Math.random` post-W1) is a product bug to fix, never a `test.retry` to add.
- **R4 — Formatter/validator quote race.** Per repo memory, the double-quote formatter hook vs `oxfmt` single-quote breaks token-matching QC validators. After any spec-delta or validator-adjacent edit: run the formatter, then `npx openspec validate add-seam-trust-anchor-journeys --strict`, then the touched validators' jest wrappers.
- **R5 — Testid drift / no per-unit pilot-name element on pre-battle.** All click-path testids were verified live in `src` this session. The pre-battle unit rows render NO pilot-name element at all — only unit name, a literal `(pilot assigned)` marker, and position (`src/components/gameplay/pages/PreBattlePage.sections.tsx:119-143`), with the force summary exposing only aggregate skills (`src/utils/gameplay/forceSummary.ts`) — so a DOM pilot-NAME assertion would require adding a rendered element, a product display change outside this change's sanctioned scope (additive testids only). The anchor's DOM clause therefore asserts the pilot-assignment indicator, and pilot identity is asserted by the API-layer cross-check. The indicator span has no testid today; group-1 audits it and, if absent, the sanctioned additive testid goes on the EXISTING marker rather than falling back to text sniffing. All file:line references are re-verified at execution time (W1 will have landed in between).
- **R6 — MODIFIED-delta fidelity.** The MODIFIED requirement replaces the canonical block wholesale at archive-merge. The delta must keep the three untouched scenarios byte-identical; any accidental rewording is a silent spec change. Verification battery diffs the delta's untouched scenarios against `openspec/specs/journey-qc/spec.md:249-266`.
- **R7 — Teardown gaps leaving rows.** A crashed run can skip afterAll. Unique per-run ids make leaked rows non-colliding (correctness preserved; hygiene degraded); teardown asserts 2xx on each DELETE so silent auth/route regressions surface. No shared fixture rows are ever reused across specs.
- **R8 — Scope creep into W3–W6.** The tempting adjacent work — pack loaders, fast-forward, CI wiring, tightening the tolerant combat specs, refactoring the audit umbrella onto shared helpers — is all explicitly out (proposal non-goals). Any such edit is a change-scope violation, not a bonus.
