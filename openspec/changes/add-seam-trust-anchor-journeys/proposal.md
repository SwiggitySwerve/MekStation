# Proposal: add-seam-trust-anchor-journeys

## Why

**W2 of the Seam-First Ladder** (council decision `openspec/council-decisions/2026-07-09-ui-audit-testability-architecture.md`): the council's seam doctrine is that this codebase's worst bugs do not live inside subsystems — they live at the mount/handoff/construction **seams**, and every fast-path mechanism the ladder builds later (W3 headless fast-forward, W4 scenario packs) deliberately skips those seams. The Phase 3 seam-coverage map names exactly three seams that must keep permanent, un-sliced, route-mounted live journeys as CI trust anchors that packs may NEVER replace. All five 2026-07-07 playtest criticals were cross-subsystem mount/timing/state-propagation bugs found by live play; none were subsystem-isolated.

Each seam has a verified coverage hole today:

- **Seam (a) — recovery rehydration.** The cold route mount `/gameplay/games/:id` drives `useGameSessionLifecycle` → `loadSession` → `recoverInteractiveSession` → `hydrateRecoverableSessionFromMatchLog` → `fromSessionAsync` → `deriveAdaptedUnitsFromSession` (`src/engine/InteractiveSession.ts:660-665`, `src/engine/InteractiveSession.recovery.ts:30-51`) — harder than click-through does, because a live-created session already sits in the store. PR #1019 (`91104d43a`) found the instant-defeat bug's **second** collision point in exactly this factory: mirrored player/OpFor forces sharing a canonical `unitRef` (e.g. both sides fielding `atlas-as7-d`) collided when recovery re-derived ids from `unitRef`; the postmortem lesson is verbatim "jest passing while live fails means the test skipped the route-mount recovery path". The existing e2e (`e2e/active-session-recovery.spec.ts`) already seeds IndexedDB and cold-navigates — but its fixture uses two units with **different** `unitRef`s (`atlas-as7-d` vs `marauder-mad-3r`) and would NOT have caught the #1019 class.
- **Seam (b) — roster→materialization handoff.** Zero e2e coverage crosses into actual encounter materialization: `e2e/campaign-customizer-handoff.spec.ts` stops at the `launch-mission-direct` button becoming enabled and never clicks it; `materializeCampaignMissionEncounter` (`src/lib/campaign/encounter/materializeCampaignMissionEncounter.ts` — POST `/api/forces` ×2, POST `/api/encounters`, PUT force attachments) appears in zero `e2e/*.spec.ts` files. The deep-play audit records the roster-collapse class (playtest critical C4, fixed in PR #1015 / `a6decc272`) only as a soft `driver.finding` (`e2e/ux-deep-play-audit.spec.ts:515-536`) — no `expect()` fails when the roster collapses. The canonical journey-qc scenario "Readiness and customizer journey validates deployment handoff" currently claims materialization verification that no blocking spec delivers.
- **Seam (c) — fresh pre-battle construction.** Unpackable by definition: a scenario pack is minted **past** construction. The #1019 producer-side regression test (`src/__tests__/integration/campaignMissionEncounterLaunchIntegrity.test.ts`) is a route-bypassing jest — it mocks `adaptUnit` wholesale, calls `new GameEngine({ seed: 42 })` directly (never mounting the pre-battle page or parsing a URL), and drives a single `advancePhase()`. It cannot express `?seed=N` threading, and the #1019 commit itself documents the jest-green/live-red divergence class this blind spot creates.

Both bug classes' behavioral contracts already exist canonically — `campaign-combat-loop` "Campaign-Linked Encounter Launch" (roster parity, from #1015) and "Launched Campaign Sessions Start Battle-Ready" (no instant defeat, from #1019). What is missing is their **route-mounted, hard-asserting live proof** — the thing the council designated as the permanent trust anchors.

## What Changes

- **Recovery rehydration anchor (hardens the existing spec).** Extend `e2e/active-session-recovery.spec.ts` with a mirrored-canonical-ref fixture (both sides fielding the same `unitRef` under distinct deployed ids) and route-mount-level id-collision assertions: every deployed id resolves a movement capability, the bare canonical ref does not, the full roster count survives, no terminal outcome is manufactured, and all invariants re-hold after reload. The inline `seedMatchLog` helper is extracted to a shared e2e helper for reuse.
- **Roster→materialization handoff anchor (new spec).** `e2e/seam-roster-materialization-handoff.spec.ts` click-drives contract acceptance → mission launch briefing → full roster selection → the launch click → pre-battle, and converts the C4 soft-finding class into hard assertions: rendered player-unit count === selected count, nonzero player-force BV, the pilot-assignment indicator present on every rendered unit row (the pre-battle roster renders an assignment marker, not pilot names — pilot identity is verified at the data layer), plus an API-layer cross-check that the materialized force carries exactly N distinct `{unitId, pilotId}` assignments with pilots preserved. Teardown deletes the created campaign, forces, and encounter (DELETE routes verified: `src/pages/api/forces/[id].ts:64`, `src/pages/api/encounters/[id]/index.ts:65`).
- **Fresh-construction no-instant-defeat anchor (new spec, W1-gated).** `e2e/seam-fresh-construction-no-instant-defeat.spec.ts` materializes one NvN encounter (player force deliberately fielding ≥2 units sharing a canonical `unitRef`), launches it from `/gameplay/encounters/:id/pre-battle?seed=N`, asserts 2N distinct per-instance unit identities, advances through Initiative plus at least two further rounds asserting no unjustified terminal outcome, then relaunches the same encounter with the same seed and asserts run-to-run-equal outcome markers. Depends on the W1 (`add-sp-combat-determinism`) implementation landing.
- **Registration and scoping.** One `verify:qc:seam-anchors` package script (targeted regression jests + the three specs at `--project=chromium --workers=1`, following the `verify:qc:replay-recovery` precedent at `package.json:181`); the three anchor specs added to the responsive projects' `testIgnore` lists (`playwright.config.ts:102/117/130`) so they schedule under exactly one desktop project. The nightly `e2e-full` lane (`.github/workflows/nightly-validation.yml`, unfiltered `npx playwright test --project=chromium`) sweeps the new specs in automatically with zero workflow edits.
- **Spec deltas.** `journey-qc` only: 1 MODIFIED (truth-repair of the readiness/customizer scenario's materialization clause, delegating the launch crossing to the new anchor) + 4 ADDED (the anchor umbrella doctrine and one requirement per anchor). No new capability folder (design D4).
- **Ledger.** `openspec/active-change-ledger.json` gains this change's entry (ships with this declarative PR, per the W1 precedent).

## Scope

### In

- `journey-qc` — MODIFIED "Playable command screen E2E journeys"; ADDED "Seam Trust Anchor Journeys", "Recovery Rehydration Trust Anchor Journey", "Roster Materialization Handoff Trust Anchor Journey", "Fresh Construction No-Instant-Defeat Trust Anchor Journey".
- Implementation (deferred until W1's implementation merges): the three anchor specs + shared e2e helpers, the `verify:qc:seam-anchors` lane, the `testIgnore` scoping additions, and — as the ONLY sanctioned product-file touch — adding missing `data-testid` hooks on existing pre-battle elements (e.g. a testid on the existing per-unit pilot-assignment marker) if the group-1 audit finds them absent.

### Out (Non-goals)

- **W6 CI gating.** No `pr-checks.yml` edits, no branch-protection changes, no per-PR required checks. W2 builds the journeys, their local runnability, and their determinism; making them PR-blocking is W6 scope. The `verify:qc:seam-anchors` lane is W6's wiring hook, nothing more.
- **W3–W5 scopes.** No `fastForwardCampaign` (W3), no scenario-pack registry or loaders (W4), no viewport-layout sweep or Playwright-project rationalization beyond the three anchors' own `testIgnore` entries (W5).
- **No walkthrough-umbrella or audit-journey changes.** `qc:ux-audit`, the deep-play journeys, and their capture-tolerant contract ("MUST NOT hard-assert outcomes that are currently broken") are untouched. The C4 soft-finding recorder in `ux-deep-play-audit.spec.ts` stays as-is — hard assertion lives in the new anchor, triage lives in the audit.
- **No tightening of existing tolerant specs** beyond the three anchors: `quick-play.spec.ts`, `playtest-sp-smoke.spec.ts`, `encounter-combat-continuity.spec.ts` keep their tolerant assertions.
- **No product behavior changes.** The anchors assert already-shipped behavior (#1015, #1019 fixes; W1 contracts). An anchor failure is a product regression to fix via its own change, never by softening the anchor. Exception carve-out: additive `data-testid` hooks only.
- **No seven-journey catalog changes.** The anchors are Playwright specs, not catalog journeys — `requiredJourneyIds` in `scripts/qc/journey-qc-core.mjs` (exactly-seven contract) is untouched, and no validation-graph node is required (existing anchors precedent: `active-session-recovery` appears only as evidence prose in `docs/qc/mekstation-qc-validation-graph.json:1464`).
- **No per-worker `DATABASE_PATH` keying.** Shared-SQLite contention under local `fullyParallel` remains a council revisit trigger (open risk 2), mitigated here by unique-id minting + teardown + single-worker lanes.

## Impact

- **Affected specs**: `journey-qc` (1 MODIFIED, 4 ADDED). The behavioral invariants cited — `campaign-combat-loop` "Campaign-Linked Encounter Launch" and "Launched Campaign Sessions Start Battle-Ready", `game-engine-orchestration` "Single-Player Launch Seed Threading" and "Recovered Session Dice Re-Seeding" (the latter two landing via W1) — are referenced, not duplicated.
- **Affected code** (implementation phase; workers re-verify line numbers against the then-current tree): `e2e/active-session-recovery.spec.ts`, new `e2e/seam-roster-materialization-handoff.spec.ts`, new `e2e/seam-fresh-construction-no-instant-defeat.spec.ts`, new shared helpers under `e2e/helpers/`, `package.json` (one script), `playwright.config.ts` (three `testIgnore` additions), plus at most additive `data-testid` attributes on pre-battle surfaces.
- **Ordering dependency (hard)**: implementation of this change starts only after the `add-sp-combat-determinism` (W1) **implementation** merges to main. At this authoring's base (`main@4bc1597bd`) only W1's declarative folder is merged: `fromSessionAsync` still hardcodes `new SeededRandom(0xc0ffee)` with no `d6Roller`, `IGameConfig` has no `seed` field, and no SP surface parses `?seed=N` — all six W1 task groups are unchecked. The recovery and handoff anchor groups have no runtime dependency on W1 mechanics, but the construction anchor and the recovery continuation-determinism scenario consume W1's contracts directly (design D1/D3, tasks group 4 gate).
- **Risk**: low. Test-only surface (plus additive testids); no engine, store, or API changes. The two risks needing care — shared-SQLite parallel contention and flake budget on live journeys — are gated in `design.md` (R2, R3).
- **Terminology guard**: "journey" is overloaded — (1) the seven catalog journeys run by `qc:journeys` (untouched), (2) the capture-tolerant UX walkthrough/deep-play journeys (untouched), (3) these three hard-asserting Playwright seam anchors (this change). "Seed" is likewise overloaded (see W1's guard); here it always means the engine RNG seed threaded via `?seed=N`.
