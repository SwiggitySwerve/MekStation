## Goals / Non-Goals

**Goals:**

- One shared, self-tested layout assertion vocabulary (`e2e/helpers/layout.ts`) replacing spec-local copies.
- One chromium-scheduled sweep asserting layout invariants per screen×viewport across the verified inventory — standalone now, pack-seeded behind the W4 gate — with a guard-tested inventory that cannot silently omit routes.
- The project matrix rationalized to tag-then-scope with mechanical proof that no CI or scripted selection changed, cutting the default local run from 2,984 scheduled tests to ~750.

**Non-Goals:** screenshot-diffing, PR gating, W6 lanes/facets, production edits, layout-bug fixing beyond quarantine bookkeeping, W4-folder edits, catalog changes. See proposal Non-goals.

## Decisions

### D1 — Capability fit: MODIFIED deltas to `e2e-testing` + `journey-qc`; no new capability

**Choice**: `e2e-testing` hosts the three ADDED requirements (helpers, sweep, topology) and the MODIFIED Deep-Play scoping requirement; `journey-qc` gets one MODIFIED requirement (Seam Trust Anchor Journeys) to de-hardcode the project names its text carries.

| Option | Verdict | Why |
|---|---|---|
| New `viewport-sweep` capability | **Rejected** | The council ladder marks only W3/W4 "(NEW spec)"; W5 has no such marker. The content — shared assertion helpers, suite topology, a cross-cutting sweep — is journey-agnostic suite infrastructure, and `e2e-testing` already owns the structurally-identical "Deep-Play Playwright Project Scoping" requirement plus the seeded-fixture and coverage-floor conventions. A new capability would split suite-shape ownership across two specs forever. |
| Primary home in `journey-qc` | **Rejected** | journey-qc owns the seven-journey catalog, walkthrough/flow-audit evidence doctrine, and the seam anchors. The sweep is not a journey and must never join the catalog. journey-qc still needs a coordinated MODIFIED delta — W2 baked the literal project names into the anchor requirement — but that is name hygiene, not ownership. |
| Home in `flow-audit-routines` | **Rejected — doctrinal conflict** | Its own requirement says "Flow Audits Are Evidence, Never CI Gates"; the sweep is blocking assertions by design. Putting assertion infrastructure under an evidence-only capability would contradict the capability's charter in its own file. |

**Duplicate-owner guard**: the deltas do not restate the anchors' invariants (journey-qc owns them), the pack loader contract (`scenario-packs` owns it — the sweep *cites* front-door loading), or the flow registry. ### D3 — The thin `mobile-touch` project: tag-scoped, opt-in, small by construction

**Choice**: `{ name: 'mobile-touch', grep: /@mobile-touch/, use: { ...devices['Desktop Chrome'], viewport: {375,667}, hasTouch: true, isMobile: true } }`. Selection is the tag — the project schedules **only** tests whose titles carry `@mobile-touch` (the exact mechanism the smoke project proved for years: literal title substrings + config `grep`). The existing touch-interaction describes (`mobile-navigation.spec.ts` "Touch Interactions"; `tactical-map-visual-smoke.spec.ts` isometric touch block) opt in by tag during implementation; their `test.use({...})` blocks stay, so their semantics are identical under either project and the tag is pure selection, not behavior.

**Why keep it at all, given the evidence that nothing needs it**: every current touch consumer self-configures (Context), so deletion-without-replacement would also be safe today. The council row explicitly reserves a thin mobile-touch project for hasTouch emulation, and it costs a handful of scheduled tests (grep-scoped, not suite-wide) while giving future touch specs a project-level home that doesn't require `test.use` boilerplate. The verified drift (self-sufficiency) is honored by making it **opt-in and near-empty** rather than a 697-test clone. If it accrues no real tenants by W6, deleting it is a one-line follow-up (Open Questions).

**Guard rails**: anchors, deep-play/walkthrough specs, sweep, and pack parity specs SHALL NOT carry `@mobile-touch` (spec'd in the deltas) — a tag-scoped project must never re-introduce double-scheduling of chromium-contract specs. Tagged tests run under both chromium and mobile-touch (two scheduled executions) — acceptable, explicit, and bounded to the tagged set.

### D5 — Sweep architecture: inventory-driven, one test per screen, resize loop, seed-once-sweep-many, quarantine for pre-existing violations

**Inventory as code, derived — never forked**: `e2e/layout-sweep/screenInventory.ts` imports `e2e/app-shell-route-manifest.json` (the existing route SoT that `app-shell-route-proof.spec.ts` already consumes) and layers sweep metadata per route: `class` (`standalone` | `recovery` | `static-catalog` | `pack-seeded` | `excluded`), concrete goto target (literal route, catalog slug, or pack-route template + pack id — with the launch screen flagged for in-page discovery, D10), per-screen check targets **with per-viewport applicability (D9)** — primary-affordance locator(s) for `expectClickable`, sibling group(s) for `expectNoOverlap`, optional canvas locator for `expectNonBlankRender` — optional `remountPerViewport` flag, optional `quarantine` entries. **A guard test fails when any manifest route (primary, recovery, or delegated pattern) has no inventory classification, when any `excluded` entry lacks a non-empty reason, and when any swept screen lacks at least one applicable primary affordance at any sweep viewport (D9)** — the manifest's own arithmetic (28+2+34 UI surfaces; harness routes auto-excluded) keeps the inventory exhaustive as routes are added.

Initial classification (from the Context inventory): swept-standalone = 28 + 2 recovery; swept-static-catalog = 3 (known slugs, the `compendium.spec.ts` precedent); pack-seeded = 18 patterns (17 campaign subroutes — 16 direct goto + the launch screen via in-page discovery, D10 — + games/[id]) — W4-gated; excluded = 13 patterns (the 12 no-covering-pack patterns + `/gameplay/pilots/[id]` per D10's contract-gap reason), each with its reason and (where an existing seeder makes later inclusion cheap — e.g. forces/[id] via the force.spec.ts precedent) a follow-up note.

**One spec, one test per screen**: `e2e/layout-sweep/viewport-layout-sweep.spec.ts` generates one `test()` per swept inventory entry (parallelizable, per-screen triage granularity, quarantine addressable by screen×viewport). Each test: `goto` the target once, then loop the four sweep viewports — `setViewportSize` → `expectNoHorizontalOverflow` → `expectNoOverlap(checkTargets)` → `expectClickable(primaryAffordance)` → `expectNonBlankRender(canvasLocator)` where declared — resolving the viewport-applicable target set per D9 at each width (an applicable target resolving to no visible element fails naming target and viewport; targets the app hides by design at a width are inapplicable by inventory declaration, never runtime-filtered). **Resize-loop, not goto-per-viewport**: the council sketch says loop `setViewportSize`, responsive behavior in this codebase is runtime-reactive (`src/constants/layout.ts`'s doc names `useDeviceType`/`responsive.ts` as derivations; CSS breakpoints reflow on resize), and goto-per-viewport quadruples navigation cost. Recorded trade-off: mount-time viewport branching is exercised only at the initially-loaded width; task 1.3 audits `useDeviceType` consumers for mount-bound (non-reactive) branching, and any screen found mount-bound sets `remountPerViewport: true` (goto per width for that screen only).

**Pack-seeded entries (post-gate) — seed once, sweep many, front-door observables only**: one `loadCampaignPack` call (navigation pack — ungated genesis, unlocks the campaign shell mount) per worker, then per-subroute goto+resize-loop for the 16 single-segment campaign subroutes against the stamped campaign id **read from the loader's post-navigation URL** — the one consumer-visible id surface W4's spec actually promises (its Registry requirement contracts `targetRoute` as "a template the loader substitutes the stamped id into" and its Front-Door Loading Contract lands the browser on that route via `page.goto`, so the URL carries the stamped campaign id by contract; W4 task 3.1's return-value mention is task-level implementation convenience this sweep does not rely on, and pack payload internals are never read). The launch screen (`…/missions/[missionId]/launch`) is reached by front-door in-page discovery per D10 — never by constructing a mission id. `loadEncounterPack` (combat pack) covers `/gameplay/games/[id]`. `/gameplay/pilots/[id]` is excluded per D10. Subsystem-*populated* content richness (a full ledger on finances, tickets in the repair bay) is NOT this sweep's concern — layout invariants on the mounted screen are; the matching-pack-per-subsystem refinement is W6-adjacent follow-up territory. All loading rides W4's front-door loaders exclusively — no store injection, no hand-seeded rows (the `scenario-packs` spec owns that contract; the sweep cites it).

**Quarantine (pre-existing violations)**: the sweep must land green without smuggling layout fixes (Non-goals). Each screen×viewport×check violation found at authoring gets a `quarantine` entry `{ screen, viewport, check, reason, followUp }` that converts that one assertion to a documented, visible skip-with-annotation; the guard test rejects quarantine entries missing reason or follow-up reference. Quarantine is shrink-only bookkeeping: removing an entry when the underlying bug is fixed is routine; adding one post-landing requires the PR to name the regression it grandfathers (reviewable, never helper-side widening).

### D6 — `expectNoOverlap`: AABB on declared sibling targets, containment excluded, reviewable opt-out, false-positive budget

**Mechanics**: for a declared set of check targets (locators resolved to visible elements), compute `boundingBox()` for each and assert pairwise AABB disjointness with tolerance ε=1px (the `LayoutValidator.boxesOverlap` math, reimplemented in the helper — importing armor-pip domain code into e2e would be a nonsense coupling). Failure names both elements (tag/testid/label) and their boxes. **A passed target that resolves to no visible element fails the assertion naming the missing target** — the helper never silently filters invisible targets (a silent filter would let a vanished-affordance regression pass vacuously); excluding a target at a given viewport is the consuming inventory's per-viewport applicability data (D9), visible in review, never a helper-side visibility filter.

**False-positive control is the design center** — overlap is the noisiest layout invariant (CSS overlays elements *by design* constantly):

1. **Declared targets, not `body *`**: the helper never scans the whole DOM pairwise (O(n²) and false-positive-saturated). The inventory declares per-screen sibling groups where overlap is a genuine defect — nav/tab bars, toolbar button rows, card grids, form action rows. Default when unspecified: the screen's top-level nav affordances (the same affordance set `app-shell-route-proof.spec.ts` already names per route).
2. **Containment is not overlap**: ancestor-descendant pairs are excluded structurally (an element containing its own child is CSS working as intended); only sibling-ish pairs (neither contains the other) are compared.
3. **Opt-out convention**: overlap-by-design pairs (badge-on-icon, avatar stacks, dropdown chevrons) are excused via an inventory-side allowlist (`allowedOverlaps: [pairs by testid/label]`) — an *inventory data* change visible in review, never a helper-code tolerance bump and never a production `data-*` attribute (zero-production-touch).
4. **Self-test with a budget**: the helper self-test spec (`page.setContent` synthetic fixtures) proves: true positive (two absolutely-positioned overlapping siblings → fails naming both), containment pass, ε-adjacent pass (1px-abutting elements do not flap), allowlist pass, a declared-but-hidden target failure (a `display:none` target fails naming the missing target — the no-silent-filter rule made mechanical), and a realistic flex-wrap fixture pass. Acceptance for the sweep itself: **zero un-quarantined, un-allowlisted overlap failures across 3 consecutive full sweep runs** — flapping overlap checks are a defect of the check, not of the screen (R6).

**Promotion notes for the other three**: `expectNoHorizontalOverflow` and `expectClickable` move near-verbatim (same thresholds: +1px scroll tolerance; 32×32px floor — the 44×44px guideline stays `mobile-navigation.spec.ts`'s own stricter assertion, not the shared floor); `expectNonBlankRender(locator, label)` is generalized per Context and its `sharp` dependency moves with it; both source specs delete their local copies and import (behavior-identical — task 2 proves their suites stay green).

### D7 — The W4 gate: authoring-time STOP, never runtime skip

**Choice**: pack-seeded sweep entries live in their own task group behind a W2-task-4.0-style **STOP-on-absence gate** (the ladder's stronger, established pattern, adopted by W4 itself as its own design's gate decision): before authoring any pack-consuming sweep entry, grep-verify on the then-current tree that `e2e/helpers/scenarioPackLoading.ts` exports `loadCampaignPack`/`loadEncounterPack` (W4 tasks 3.1/4.1 contracts) and that the navigation and combat pilot packs exist in `e2e/scenario-packs/` (the experience pack left this change's scope with D10's pilots/[id] exclusion). Absent → STOP the group and report naming the missing artifact; never stub a loader, never hand-seed a substitute state. The same gate step verifies D10's discovery precondition — the navigation pack's missions screen surfaces the launch-briefing affordance — whose failure mode is the documented reclassify-to-excluded fallback (D10), not a STOP.

**Runtime `test.skip`-on-missing-loader is rejected explicitly**: `e2e-testing`'s own "Conditional skip-on-missing-data is forbidden" scenario bans skip-guards that let a spec pass without exercising what it names — a sweep that silently skipped 18 screens when packs are absent would vacuously green exactly the way that requirement forbids. Until the gate passes, the pack-seeded entries **do not exist in the sweep spec** (they exist in the inventory as `pack-seeded`, visibly un-swept), and the guard test asserts the classification, so the coverage gap is loud in the inventory rather than silent in a skip.

### D9 — Per-viewport check-target applicability: responsive collapse is data, never quarantine

**The problem this kills**: the route-proof affordance sets are desktop-authored, and the app shell hides them by design below its breakpoints (Context: TopBar's three mutually exclusive nav variants — labeled nav ≥1024, icon nav 768–1023, hamburger-only <768). A sweep that asserted one fixed target set at all four widths would fail-by-construction at 375 on every dashboard-class screen, and quarantine is defined for *violations with follow-ups*, not for CSS working as intended. Left unspecified, implementation would improvise one of two bad answers: silently filter invisible targets (vacuous pass — a vanished affordance at a width where it should exist sails through) or fail on by-design collapse (permanent false positives).

**Choice**: every check target in the inventory (primary affordance, overlap-group member, canvas locator) carries **viewport applicability** — applicable at all four sweep viewports by default, narrowable to a declared subset — and a screen MAY declare distinct targets per viewport. Rules, each load-bearing:

1. **Full clickability coverage**: every swept screen declares, for each of the four sweep viewports, at least one applicable primary affordance. The elements MAY differ per width — the dashboard's is the Browse/Tools/Gameplay menus at 1024/1280 (labeled) and 768 (icon variants) and the `aria-label="Open menu"` hamburger at 375 (Context's verified TopBar profile). The inventory guard enforces the coverage: no swept screen×viewport without an applicable primary affordance.
2. **By-design collapse is applicability data**: an affordance the app intentionally hides at a width is declared inapplicable at that width in the inventory — an inventory-data change visible in review — never a quarantine entry (quarantine requires a follow-up reference; CSS-by-design has no follow-up) and never a helper-side weakening.
3. **Applicable-but-missing fails loud**: at each width the sweep asserts exactly the applicable target set; an applicable target that resolves to no visible element fails naming the target and the viewport (D6's helper-side no-silent-filter rule). The sweep never filters targets by runtime visibility.
4. **State-conditional affordances never default in**: route-proof `stateAffordances` are any-of alternatives (empty state vs populated cards) and are NOT check targets by default. The default check-target set is the route's *unconditional* route-proof affordance set (filtered through applicability). A state-conditional target must be declared explicitly, naming the state the swept screen is deterministically in — the fresh-app empty state for standalone screens, the pack's documented target state for pack-seeded screens.
5. **Shared chrome group, defined once**: the app-shell chrome (the TopBar variants) is one shared target-group definition with per-viewport variants, referenced by screens rather than re-declared per screen — per-screen copies of the same chrome profile invite drift.

**Rejected alternatives**: (a) *targets must be visible at all swept widths* — excludes the entire app-shell nav from the sweep, gutting the clickability check exactly where responsive collapse makes it interesting; (b) *runtime visibility filtering* (assert whatever happens to be visible) — vacuous-pass machinery, the same class as the capability's banned conditional skip; (c) *quarantining by-design collapse* — misuses the violation ledger, drowns real findings in permanent non-findings, and forces fake follow-up references.

### D10 — The two W4-contract-risky routes: launch via in-page discovery, pilots/[id] excluded

Two of the council sketch's 19 pack-coverable patterns assume semantics W4's `scenario-packs` spec never promises (verified against `add-scenario-packs/specs/scenario-packs/spec.md` + its design on this tree).

**(a) `/gameplay/campaigns/[id]/missions/[missionId]/launch` — swept, but only via front-door in-page discovery.** The route carries a second dynamic segment W4 never sources: the Front-Door Loading Contract promises no return value at all, mission ids are remapped-or-kept internals ("when and only when they collide across rows"), and no contract makes a mission id consumer-visible. **Choice**: the launch entry's navigation is the pack-seeded missions screen (`…/[id]/missions`, campaign id from the post-load URL per D5) followed by clicking the mission row's launch/briefing affordance — arriving at the launch route with a real mission id minted by the app itself, exactly as a user would. Constraints: the discovery click navigates to the briefing screen only — the sweep never actuates the launch control itself (that crossing is W2 seam (b) territory, and the navigation pack's genesis stops strictly before it); and the launch screen's check targets exclude encounter-link/replay affordances on mission history (W4's minter strips those cross-store references and its spec bans asserting them on pack-loaded state — the sweep honors that ban, which costs nothing since layout invariants don't need them). Task 5.0's gate verifies at authoring time that the navigation pack's documented target state (contract accepted, at the mission-launch briefing boundary) actually surfaces the discovery affordance on the missions screen; if it does not, the entry reclassifies to `excluded` with that documented reason — never reached via a constructed mission id, an uncontracted loader return value, or pack payload internals.

**(b) `/gameplay/pilots/[id]` — excluded-with-documented-reason.** Three verified contract gaps: pilot ids are not in W4's enumerated remap families (they join only conditionally, per W4's own R10, if its group-1 verification finds campaign surfaces fetching `/api/pilots` rows); no contract makes the experience pack's `targetRoute` a pilot detail route (its documented target state is "pilot with XP surface rendered" — a campaign surface); and the global pilot roster on a fresh app is empty, so in-page discovery from `/gameplay/pilots` has no promised row to click either. **Choice**: classified `excluded`, reason recorded in the inventory, with a follow-up note naming the two unlock paths — W4's R10 verdict landing on "loader creates pilot rows front-door" (which would make roster in-page discovery contract-backed), or the existing non-pack seeder precedent the manifest itself names for this pattern (`force.spec.ts`/`replay-player.spec.ts`-style generated roster objects). Consequence: the sweep consumes exactly two packs — navigation and combat; the experience pack leaves this change's scope entirely (D7's gate list shrinks to match).

**Net counts**: pack-seeded = 18 (16 direct-goto campaign subroutes + 1 discovery launch screen + games/[id]); excluded = 13; post-gate sweep total = 51 screens.
