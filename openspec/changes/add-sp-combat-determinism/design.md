# Design: add-sp-combat-determinism

## Technical Approach

Wire the existing seeded-RNG infrastructure into the four places SP combat dice escape it today тАФ session creation, session recovery, the launch surfaces, and the MaxTech heat-scale critical location roller (the one leak that also reaches auto-resolve) тАФ reusing `SeededD6Roller` (`src/simulation/core/SeededD6Roller.ts`, already production code in `src/utils/gameplay/damage/critical.ts`) and copying the MP wiring precedent (`ServerMatchHostBootstrap.ts` builds a roller from a seed and passes it as the 9th positional `InteractiveSession` constructor arg). Nothing new is invented: no new PRNG, no new roller class, no new query-param convention тАФ the only new code artifact is a small uniform location-index derivation helper (D7). The change is sequenced as five implementation groups plus a verification group, run strictly sequentially by one worker each (parallel agent fan-out has failed on this repo).

## Architecture Decisions

### Decision: D1 тАФ Inject a dedicated seeded dice stream in `createInteractiveSession`

**Choice**: `createInteractiveSession` constructs `new SeededD6Roller(this.seed).asD6Roller()` and passes it as the 9th positional argument (today the literal `undefined` at `GameEngine.ts:253`). The dice stream is a **dedicated** `SeededRandom(this.seed)` (the `SeededD6Roller(number)` constructor overload builds its own), NOT the engine's shared `this.random`.

**Rationale**: A dedicated stream decouples combat dice from bot-AI draw counts тАФ `this.random` also feeds `BotPlayer` decisions (`InteractiveSession.ts:183`), so sharing it would reshuffle every dice outcome whenever bot logic changes its number of PRNG draws. The MP path already models this separation (`diceSeed`'s roller is independent of any AI random). Determinism holds either way; the dedicated stream just shrinks the blast radius of future AI changes on the version-pinned seedтЖТoutcome mapping.

**Constraint**: `runToCompletion`'s rule-off stream composition (which shares `this.random` for dice, `GameEngine.ts:156`) is NOT changed тАФ it is already deterministic there, and re-composing its stream would churn the existing auto-resolve / `QuickResolveService` seedтЖТoutcome mappings for no benefit. The single sanctioned touch is D7's MaxTech heat-scale location roller at `GameEngine.ts:211`, which draws only when that optional rule is enabled тАФ a path that was `Math.random`-nondeterministic before, so no existing mapping churns.

### Decision: D2 тАФ Uniform resolver stream; the vibro-claw fallback becomes engine-unreachable, not removed

**Choice**: All resolvers keep reading `context.d6Roller` / `this.d6Roller`. Because engine-created sessions now always inject a roller, every resolver тАФ including the vibro-claw special case at `InteractiveSession.ts:570` (`this.d6Roller ?? (() => this.random.nextInt(6) + 1)`) тАФ consumes the single injected stream. The `?? seeded-random` fallback line is left in place: it remains the behavior for roller-less direct constructions (MP-recovered legacy sessions, direct test constructions). The constructor's `d6Roller?` parameter stays optional, and the stale doc comment at `InteractiveSession.ts:146-155` ("When omitted, resolvers fall back to тАж `Math.random` тАФ preserves single-player + hot-seat behavior") is rewritten тАФ SP engine callers now always inject.

**Rationale**: Making the constructor default the roller itself (e.g. `d6Roller ?? seeded-from-this.random`) would silently flip **MP-recovered** sessions from `Math.random` to a fixed `0xc0ffee`-correlated stream тАФ an MP behavior change this change's non-goals forbid. Explicit injection at the SP call sites gets full uniformity on every SP path with zero MP drift.

### Decision: D3 тАФ Persist the resolved seed on `IGameConfig`

**Choice**: `IGameConfig` (`src/types/gameplay/GameSessionUnitTypes.ts`) gains `readonly seed?: number | null` (doc-commented, mirroring the `encounterId` optional style). `runToCompletion` stamps `seed: this.seed` into its inline `gameConfig` (`GameEngine.ts:128-137`). For the interactive path, `InteractiveSessionConstructorArgs` gains a trailing optional `seed?: number` (12th positional, consistent with the tuple's append-only history), threaded by `createInteractiveSession` and stamped by `buildInteractiveSessionGameConfig` (`InteractiveSession.setup`). Direct constructions that omit it (MP, tests) persist no seed тАФ unchanged behavior.

**Rationale**: The seed must live in persisted state for recovery to reconstruct anything: today `IGameConfig` has no seed field and no roll/advance counter exists anywhere, so recovery cannot honestly be reproducible. Config is the natural home тАФ it already round-trips through `createGameSession`, the `GameCreated` event, and `hydrateGameSessionFromEvents`.

**Constraint**: The hydration round-trip must be verified, not assumed тАФ if the `GameCreated` payload or the hydrator prunes unknown config fields, the seed dies before recovery (R2). A jest asserts `hydrate(persist(session)).config.seed === session.config.seed`.

### Decision: D4 тАФ Recovery re-seeds from position zero; continuation is explicitly out of scope

**Choice**: `fromSessionAsync` derives `session.config.seed`; when present it constructs `new SeededRandom(config.seed)` for the `random` param (replacing the hardcoded `0xc0ffee` at `InteractiveSession.ts:332`) and `new SeededD6Roller(config.seed).asD6Roller()` for the `d6Roller` param (replacing `undefined` at `:338`). When absent тАФ all pre-change sessions and every MP-recovered match тАФ it passes exactly what it passes today (`SeededRandom(0xc0ffee)`, `undefined`), byte-identical behavior.

**Rationale тАФ the honest contract**: nothing persisted records how many rolls the interrupted session consumed, so exact stream **continuation is unsupported**: roll N+1 after recovery will generally differ from what roll N+1 would have been in an uninterrupted run. Re-seeding from position zero is the simplest contract the persisted state supports, and it satisfies the adopted determinism contract ("assert invariants across seeds, never golden traces") тАФ reproducibility of a *recovered-and-replayed* run is an invariant across identical recovery inputs, not a claim of splice-perfect resumption. The alternative (persist a monotonic roll counter + fast-forward the PRNG) needs new event/counter machinery on every roll and is rejected as W1 scope.

**Trade-off recorded**: a battle interrupted and recovered will not produce the same remaining-roll sequence as the same battle left uninterrupted. Accepted; documented in the spec delta so no future test asserts otherwise.

**Proof obligation тАФ the recovery seam gets its own determinism jest, not just seed survival**: the invariant this contract creates ("two identical rehydrations of one persisted seeded session, driven identically, produce identical normalized continuation streams") is asserted directly by a recover-twice jest that goes through the SAME factory chain the live route mounts (`hydrateRecoverableSessionFromMatchLog` тЖТ `fromSessionAsync` тАФ the path `recoverInteractiveSession` composes, `InteractiveSession.ts:660-665`). W2's seed-then-goto journeys ride exactly this factory, and repo history (instant-defeat recovery-collision bug, PR #1019: "jest passing while live fails means the test skipped the route-mount recovery path") is specifically about tests that dodge this seam. Seed *survival* (the persistтЖТhydrate round-trip, R2) and fresh-session determinism (D6's proof) do not cover it тАФ neither ever drives a recovered session.

### Decision: D5 тАФ `?seed=N` threading mirrors the MP convention and the Pages-Router query pattern

**Choice**: `src/pages/gameplay/encounters/[id]/pre-battle.tsx` (the sole page mounting both hooks тАФ `usePreBattleSkirmish` at `:215`, `usePreBattleLaunch` at `:228`) destructures `seed` from `router.query` alongside its existing params (`:158-159` pattern), parses it with `Number.parseInt(first(seed), 10)` guarded by `Number.isFinite` тАФ the exact guard `server.js:590-595` uses for the MP WS param тАФ and passes the resulting `seedOverride?: number` into both hooks. The hooks change `seed: Date.now()` to `seed: seedOverride ?? Date.now()`. The quick-game page (`src/pages/gameplay/quick/index.tsx`) does the same parse and threads `seedOverride` into the three store actions (`startBattle` / `startSpectatorMode` / `startInteractiveSkirmish`) so `createEngineForQuickGame` (`useQuickGameStore.actions.ts:26-38`) resolves `seed: seedOverride ?? Date.now()`.

**Rationale**: One convention across MP and SP (`?seed=N`, parseInt + isFinite, silent fallback on garbage) and one repo idiom for reading it (Pages Router `router.query` destructure тАФ `router.query` values are `string | string[] | undefined`, hence the defensive first-value + parse). The quick-game surface is folded in because it is the identical one-line bug class on a live page; fixing pre-battle while leaving quick-game unseeded would be an inconsistent half-fix (verified finding beyond the council's W1 row). The scenario-generator seed in the same store file stays untouched (non-goal тАФ map cosmetics, W4).

### Decision: D6 тАФ Determinism proofs assert invariants, never golden traces

**Choice**: The jest proof creates two engines with the same seed and identical unit inputs, drives both spectator-mode interactive sessions through identical `advancePhase` loops to completion (bounded by the turn limit), and asserts the two **normalized** event streams are deep-equal тАФ normalization strips wall-clock fields (`timestamp`, `startedAt`-derived values) and non-deterministic ids, keeping ordering, event types, phases, roll-bearing payload semantics, and the final outcome. A secondary check runs a third session on a different seed and merely confirms the run is valid (divergence is expected but NOT asserted as required тАФ two seeds may legally collide on outcome). The e2e proof drives the pre-battle route with the same `?seed=N` twice through auto-resolve and asserts identical winner + turn count + event count.

**Rationale тАФ version-pin**: seedтЖТoutcome mappings are pinned to the engine version; any rules change, resolver reorder, or AI tweak may legally reshuffle what seed 42 produces. Tests that hardcode roll sequences or outcome tables for specific seeds turn every legitimate engine change into a fixture-regen chore. Same-seed *self*-consistency within one build is the durable invariant тАФ it survives every version bump. (Existing dice-system scenarios that feed scripted stub rollers are unaffected: a stub that *produces* [3,5,2] is an injected input, not a golden seed assertion.)

**Coverage extensions (both invariants, no golden traces)**: (1) the same-seed jest runs a second variant with `optionalRules: ['maxtech-heat-scale']` so the optional-rule heat-crit dice path (D7) is inside the detector's blast radius тАФ with default rules the heat-scale code never executes and a leak there is invisible to the proof; (2) a recovery-determinism jest asserts the recover-twice invariant on the live factory chain (see D4's proof obligation) тАФ fresh-session determinism and seed survival do not exercise the recovery seam this change itself modifies.

### Decision: D7 тАФ Seed the MaxTech heat-scale critical location roller (the one live resolver-level `Math.random` escape in BOTH modes)

**Choice**: `resolveHeatPhase`'s `maxTechCriticalLocationRoller` option тАФ which defaults to `() => Math.floor(Math.random() * 8)` (`src/utils/gameplay/gameSessionHeat.ts:48-50`) and is consumed at `gameSessionHeat.effects.ts:306` whenever `hasMaxTechHeatScaleRule(session.config.optionalRules)` matches (`'maxtech-heat-scale'` / `'maxtech_heat_scale'` / `'maxtech-heat'` / `'tacops-heat-scale'`, `gameSessionHeat.helpers.ts:3-14`) тАФ is supplied at BOTH production call sites, derived from that mode's existing seeded stream via one shared helper `createLocationIndexRollerFromD6(d6: D6Roller): () => number` added to `gameSessionHeat.helpers.ts` (next to the existing `createD6RollerFromDiceRoller`):

- `GameEngine.ts:211` (auto-resolve): derive from the `runToCompletion` `d6Roller` closure тАФ everything stays composed from `this.random`, consistent with that mode's existing stream design.
- `InteractiveSession.phases.ts:228` (interactive): derive from `context.d6RollerForResolvers()`; pass `undefined` when no roller was injected, so roller-less direct constructions keep the legacy `Math.random` default тАФ the same containment shape as D2. Recovered sessions inherit the fix automatically because they drive phases through this same driver.

The helper's reference scheme: three d6 draws mapped as `((a-1)*36 + (b-1)*6 + (c-1)) % 8` тАФ 6┬│ = 216 is divisible by 8, so the index is EXACTLY uniform over 0тАУ7 (matching the `Math.random` default's distribution) while consuming only the injected stream. Implementation may substitute any scheme that is exactly uniform over 0тАУ7 and consumes only the injected stream; the invariant tests pin no sequences.

**Rationale**: with the rule enabled, a failed heat-scale crit check rolled its location from `Math.random` in every execution mode тАФ falsifying both the "auto-resolve is already deterministic" premise and the spec's "no resolver SHALL fall back to Math.random" contract, and SP launches thread encounter `optionalRules` straight through (`usePreBattleLaunch.ts:228` тЖТ engine; `encounterToGameSession.ts:98`), so the rule is live-reachable. This is precisely the W1 bug class (a wiring gap, not a missing mechanism тАФ the option already exists). Critically, R4's same-seed detector could NOT have caught it with default rules, which is why the proof suite runs a rule-on variant (D6). Rule-off behavior in both modes is stream-identical to pre-change: the location roller is constructed but never draws unless the rule is on and a crit check fails, so no existing rule-off seedтЖТoutcome mapping churns; rule-on auto-resolve was nondeterministic before, so there is no prior mapping to preserve.

**Survey of the remaining resolver-level `Math.random` defaults** (verified this session): BA swarm-fire's roller default is engine-unreachable today тАФ engine-created sessions never supply `swarmFireOptions` (`InteractiveSession.lifecycle.ts:97-111` omits it), so the End-phase swarm block no-ops; whoever wires `swarmFireOptions` into engine contexts later MUST thread the injected roller then. The aerospace crit-table default (`aerospace/criticalHits.ts:66`) is overridden by its resolver and aerospace is out of W1 scope (PR-M). Vibro-claw is covered by D2. Roller-parameter defaults deeper in the chain (`applyPilotDamage`, `rollInitiative`, etc.) only fire when a caller omits the roller тАФ the same-seed proofs are the detector for any such dropped-thread (R4).

## Risks

- **R1 тАФ MP recovery drift through shared `fromSessionAsync`.** `MatchRecovery.rebuildSessionFromEvents` (`MatchRecovery.ts:85`) calls the same factory. Gate: the seeded path activates ONLY on a present `config.seed`, which MP sessions never carry (`buildHostSession` / `buildInteractiveSessionGameConfig` receive no seed on the MP path). Verify with the MP recovery + transport jest suites and a grep confirming no MP construction site passes the new seed arg.
- **R2 тАФ Seed dies in the persistence round-trip.** If the `GameCreated` event payload or `hydrateGameSessionFromEvents` prunes unknown `IGameConfig` fields, recovery finds no seed and silently stays legacy. A dedicated jest asserts the persistтЖТhydrate round-trip preserves `config.seed`; the hydrator/creator is adjusted in this change if it prunes.
- **R3 тАФ Event-stream comparison flakiness.** Raw event objects carry wall-clock timestamps and generated ids that differ across the two same-seed runs. The comparison MUST normalize (strip/replace non-semantic fields) rather than sleep or retry; if a payload field turns out to be non-deterministic for a non-RNG reason (e.g. iteration order), that is a real determinism bug to fix, not to mask in the normalizer.
- **R4 тАФ Hidden `Math.random` in a resolver bypassing the injected roller.** The injection only covers resolvers that honor `context.d6Roller`. The same-seed jest proof is the detector: any resolver still reaching `defaultD6Roller`/`Math.random` makes the streams diverge. Triage such a divergence to the offending resolver and route it through the context roller (that is precisely the W1 fix surface). **Known detector blind spot, closed in this change**: code paths gated behind optional rules or unit types the proof fixtures don't exercise are invisible to a default-rules run тАФ the MaxTech heat-scale location roller (D7) was exactly such a leak, so the proof runs a `maxtech-heat-scale` rule-on variant, and the survey in D7 records the remaining engine-unreachable defaults (BA swarm-fire, aerospace) that MUST be roller-threaded if a later change wires them into engine contexts.
- **R5 тАФ `router.query` shape.** Values are `string | string[] | undefined` and unpopulated on first render before hydration. Parse defensively (first value, `Number.parseInt`, `Number.isFinite` guard, fallback to `Date.now()`); never throw on garbage тАФ mirror the MP parser's silent-fallback semantics.
- **R6 тАФ Formatter / validator quote race.** Per repo memory, the double-quote formatter hook vs `oxfmt` single-quote can break token-matching QC validators. After spec-delta edits, run the formatter + `npx openspec validate add-sp-combat-determinism --strict`.
- **R7 тАФ Scope creep into MP or stream-continuation.** The tempting "while we're here" fixes тАФ defaulting the constructor roller, seeding MP recovery's `0xc0ffee`, persisting roll counters тАФ are all explicitly rejected above (D2, D4). Any such edit is a change-scope violation, not a bonus.
