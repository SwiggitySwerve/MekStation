# Design: add-subsystem-lanes-and-ci

## Context

All facts below verified on `claude/subsystem-lanes-and-ci@6e0ed7868` (main + W1/W2/W3 implementations and archives; W4/W5 declarative folders only) plus live GitHub evidence (`gh api` / `gh run view`, 2026-07-10/11).

**The PR gate (the surface this change wires into):**

- `pr-checks.yml` (740 lines) defines 22 job blocks expanding to exactly **29 check runs** (unit-test-shards ×6, build-test ×3). Branch protection on `main` requires only **4 contexts** — `Lint and Test`, `Build Test / win`, `Build Test / mac`, `Build Test / linux` (live `gh api repos/SwiggitySwerve/MekStation/branches/main/protection` snapshot; matches `.github/BRANCH_PROTECTION.md:46-50`; `strict: true`, `enforce_admins: true`). The other 25 checks are transitively required through `Lint and Test`, a 1-minute aggregator whose `needs` array carries 17 entries (`pr-checks.yml:573-599`) — the config comment says it exists precisely to "preserve the 'Lint and Test' required-check name from before the fan-out." The `e2e-testing` spec's "PR-Gated Coverage Floor" requirement already contracts aggregator-`needs` membership as the blocking mechanism ("the coverage job SHALL be a member of the aggregator's required `needs` set").
- **The aggregator's failure path is unsound as it stands (verified live).** `lint-and-test` carries no `if:` condition — no job in `pr-checks.yml` does (per-step guards are the repo convention) — so under GitHub's default `needs` semantics a failed fan-out job **skips** the aggregator rather than failing it. Verified on run 29059132882 (`gh run view`): `Unit Tests (4/6)` concluded `failure` and `Lint and Test` concluded `skipped` — completed-skipped, not pending. GitHub's documented skipped-but-required semantics: a job skipped by a condition reports its status as success, and a pull request requiring that check may not be blocked. So aggregator-`needs` membership alone does NOT guarantee a red fan-out job blocks the merge — a latent hole predating this change, which this change must close (D2.3) and prove closed red-path (D2.4), because the seam gate's entire blocking authority rides this mechanism.
- **CI Playwright runs retry by default.** `playwright.config.ts:46` sets `retries: process.env.CI ? 2 : 0`; GitHub-hosted runners export `CI=true`, so every CI Playwright invocation inherits **2 retries** unless the invocation passes `--retries=0` explicitly. Local runs (`CI` unset) genuinely run at 0 retries. A gate that inherits the CI default can retry a flaky anchor to green — and burn-in would then record clean evidence for a flaky gate (R4).
- **The three seam anchors never run per-PR.** The only Playwright invocation in `pr-checks.yml` is `npx playwright test e2e/tactical-map-visual-smoke.spec.ts --project=chromium` (line 521, the `E2E Smoke` job, L476-531). That job's shape is the template for any sibling: `needs: [detect-changes, install-deps]`, every step guarded by `if: needs.detect-changes.outputs.code == 'true' || …e2e == 'true'` so docs/openspec-only PRs no-op in ~4s while the check name stays green (verified on a real docs-only run), chromium browser cache keyed on the Playwright version, failure-only report upload.
- **The headless-invariant half of the council's gate already runs per-PR.** The W2/W3 jest suites (`src/engine/__tests__/InteractiveSession.recovery.test.ts`, `src/__tests__/integration/campaignMissionEncounterLaunchIntegrity.test.ts`) are ordinary members of the `unit` jest project (testMatch `**/__tests__/**`, `jest.config.js`), are not caught by the perf-sensitive exclusion (which names only 4 `src/simulation/__tests__` files), and therefore run on every PR inside the 6-shard matrix. Measured: 2 suites / 7 tests / 6.8s locally. Nothing pins this — a future exclusion-pattern edit could silently drop them.
- **Timings (measured on real runs):** code-touching PR wall-clock ≈ 6m09s–7m41s over 3 sampled runs; run 29140472238's critical path is Prepare Build Artifacts (237s) → Build Test/win (197s) ≈ 434s of a 461s total. `E2E Smoke` costs ~187–205s when it actually runs (2m21s for the test step alone on run 29134719618) and ~4s when skipped. `Validate Combat Suite` (333s) is the longest lane but runs parallel, off the critical path.
- **The local seam lane cannot be reused in CI.** `verify:qc:seam-anchors` (`package.json:184`) shells through the literal binary `npm.cmd` (Windows-only — the repo-wide `verify:qc:*` convention, 13+ scripts) and runs the three anchor specs as **three sequential Playwright invocations**. Each invocation on a fresh runner pays its own webServer boot (`reuseExistingServer: true` only helps when a server is already listening; each invocation's self-started server dies with its process) — roughly 3× the single-invocation cost. Every existing CI job instead calls `npx jest`/`npx playwright test` directly.
- **Nightly reality:** `nightly-validation.yml` (cron 09:00 UTC + dispatch; jobs `install-deps`, `full-statistical-proofs`, `e2e-full` = whole chromium project `--ignore-snapshots`, `coverage`, `notify-on-failure` with `needs: [full-statistical-proofs, e2e-full, coverage]` + `if: failure()` opening/commenting a tracking issue) has failed **10/10 of its last 10 scheduled runs** on the same 3 lanes, driven by 8 pre-existing failing specs (campaign, encounter-combat-continuity, gm-campaign-ledger-control-plane, integration, playable-command-feature-screens, playtest-coop-route-smoke, quick-play + 1 flaky). The three anchors pass inside that red run (667 passed / 707) — unverifiable-by-anyone signal until now. Red nightly is the steady state this design must treat as the starting condition.

**The subsystem/registry surfaces:**

- The journey catalog (`docs/qc/mekstation-journey-scenarios.json`, 7 required journeys) carries no subsystem-like field — the only domain taxonomy is `module` (`roster`/`construction`/`combat`/`campaign`), validated as a bare non-empty string (`journey-qc-core.mjs:357-405`). Closed-set validation precedent: `allowedModes`/`allowedTiers` Sets at `journey-qc-core.mjs:51-52`. The graph query command and validators live in the same module; jest wrappers for the QC validators live in `scripts/__tests__/` (e.g. `journey-qc.test.ts` running `scripts/qc/validate-journey-qc.mjs`) and are matched by the unit project's `**/__tests__/**` glob — i.e. **guards placed there run in every PR's unit shards automatically**.
- The closed six-tag vocabulary already exists and is already double-bound: `e2e/flows/manifest.ts:25-31` (`FlowSubsystem`, doc-commented "do not add tags here without amending the spec text first"), and W4's unmerged pack manifest contract reuses the same literal set with a guard test. `mekstation-qc-registry.json` is a **different** taxonomy (27 architectural surfaces; its `maintenance-code-health` surface is codebase-health, unrelated to the gameplay `maintenance` tag) — bolting the facet onto it would conflate axes; `run-qc-lanes.mjs`/`select-qc-lanes.mjs` (near-duplicate filter engines over that registry, 9 filter keys, spawn-replay of static `commands` arrays) cannot synthesize a `--grep` invocation.
- **Three tag mechanisms coexist; only two work.** Playwright `--grep` matches titles and native tags. (1) JSDoc docblock "tags" are invisible to grep — of 28 files whose text contains `@smoke`, only 12 embed it in a `test.describe(...)` call; all three `playtest-subsystem-*` specs claim docblock tags that select nothing. (2) Title-embedded tags work (the legacy `@smoke`/`@campaign`/`@combat` vocabulary documented at `playwright.config.ts:14-32`). (3) Native `test.describe(title, { tag: [...] })` works and keeps titles clean — precedent `e2e/playtest-sp-smoke.spec.ts:287`. A bare `@combat` subsystem tag would collide with the legacy title-tag vocabulary; namespacing avoids it.
- **Store reads from e2e are a solved problem:** `src/lib/e2e/storeExposure.ts` exposes `campaign`, `campaignRoster`, `pilot`, `force`, `encounter`, `gameplay`, … on `window.__ZUSTAND_STORES__`; `e2e/helpers/store.ts` provides the read/wait helpers; `e2e/helpers/campaignSeeders.ts` (e.g. `seedHiringHall`) is the blessed seeding tier the e2e-testing spec already names for subsystem specs.

## Goals / Non-Goals

**Goals:**

- Cash in the ladder: the three seam anchors PR-blocking through the existing aggregator — hardened so its failure path actually fails, retry-free so flake can never green, red-path-proven so "blocks the merge" is demonstrated rather than assumed — with zero branch-protection edits, zero renames, quantified wall-clock, and burn-in-before-blocking; the unit-lane invariants pinned.
- One subsystem vocabulary, three consuming surfaces: the journey-catalog facet (+ query mode), native `@subsystem:` tags on audit specs, nightly sliced lanes — all guard-locked to the same closed six-tag set.
- The four write-flow mini-journeys closing the verified assertion holes, and an honest, machine-validated replacement for the false 19-spec SHALL.

**Non-Goals:** branch-protection changes, check renames, PR-gating anything beyond the anchors, new projects/runners/harnesses, vocabulary growth, W4/W5 folder edits, anchor edits, production edits, nightly-red remediation, repo-wide docblock-tag cleanup. See proposal Non-goals.

## Decisions

### D1 — Capability fit: MODIFIED+ADDED deltas to `journey-qc` and `e2e-testing`; no new capability; W5-collision avoidance is a hard constraint

**Choice**: `journey-qc` hosts the catalog facet (MODIFIED Required Journey Scenario Catalog), the query mode (MODIFIED QC Graph Query Command), and the anchors' CI authority (ADDED Seam Anchor Pull-Request Gate — it cites the anchors' requirement, never modifies it). `e2e-testing` hosts the honest coverage contract (MODIFIED Subsystem Validation Coverage) and the two suite-infrastructure additions (ADDED Subsystem Tag Taxonomy; ADDED Nightly Subsystem Audit Lanes).

| Option | Verdict | Why |
|---|---|---|
| MODIFY `journey-qc` "Seam Trust Anchor Journeys" to add the PR-gate clause | **Rejected — collision** | W5's unmerged delta already rewrites that requirement (topology-agnostic anchor scheduling). Two in-flight MODIFIED deltas to one requirement means whichever archives second silently clobbers the first. The gate is a new obligation with its own lifecycle — a clean ADDED requirement that references the anchors. |
| Home the gate in `e2e-testing` (beside PR-Gated Coverage Floor) | **Rejected** | The mechanism (aggregator `needs`) is already contracted there and is cited, but the *content* — which journeys constitute per-PR seam trust — is anchor doctrine, and journey-qc owns the anchors. Splitting anchor authority across two capabilities invites drift. |
| New `ci-lanes` capability | **Rejected** | The council marks only W3/W4 "(NEW spec)". Both touched capabilities already own the adjacent obligations (journey-qc: anchors + verify lanes; e2e-testing: coverage floor + suite topology). A third owner would fragment CI-lane doctrine across three files. |
| Delta `flow-audit-routines` for lanes | **Rejected — doctrinal** | Its own requirement: "Flow Audits Are Evidence, Never CI Gates". Lanes are blocking-style assertions run on a schedule; flow-audit surfaces stay evidence-only and untouched. |
| Delta `scenario-packs` for pack-spec tags | **Rejected** | W4 is unmerged; its spec already contracts pack scoping. W6's pack-parity tagging is an additive title-tag task gated on W4 impl, governed by the Subsystem Tag Taxonomy requirement (manifest↔tag equality guard) — no pack-side delta needed. |

**Duplicate-owner guard**: the deltas restate neither the anchors' invariants (journey-qc owns them, W5 is rewording them) nor the aggregator mechanism's coverage-floor instance (cited), nor the flow registry vocabulary (cited — `flow-audit-routines` "Flow Registry" owns the six tags; this change reuses the literals and guards equality).

### D2 — The per-PR gate: one sibling job, single Playwright invocation, blocking via aggregator `needs`; branch protection untouched

**The named failure modes this decision exists to kill:** (a) a required-check name change (names are the branch-protection currency — the repo memory's gotcha); (b) a manual settings flip smuggled in as if it were a repo change; (c) a 3× webServer-boot job blowing the PR budget; (d) a flaky blocking gate; (e) a gate whose failure path silently greens — the verified skipped-aggregator hole (Context) and the verified CI retry default, either of which would make the gate's promise fictional.

**Choice**:

1. **New job `seam-anchors` (`name: Seam Anchors`)** in `pr-checks.yml`, byte-mirroring the `E2E Smoke` job shape (L476-531): `needs: [detect-changes, install-deps]`, `timeout-minutes: 20`, per-step `if:` guards on the `code`/`e2e` detect-changes outputs (docs/openspec-only PRs no-op in seconds with the check green), the same Playwright browser cache steps, failure-only report upload with its own artifact name.
2. **One invocation, three specs, retries forced off**: `npx playwright test e2e/active-session-recovery.spec.ts e2e/seam-roster-materialization-handoff.spec.ts e2e/seam-fresh-construction-no-instant-defeat.spec.ts --project=chromium --retries=0` (CI already runs `workers: 1` by config). The `--retries=0` is load-bearing, not stylistic: the config's CI default is `retries: 2` (`playwright.config.ts:46`; `CI=true` on GitHub runners — Context), and a gate that inherits it retries a flaky anchor to green, masking flake inside the very mechanism that exists to expose it. One webServer boot amortized across all 5 tests. The local script's 3-sequential-invocation shape is rejected for CI: each invocation would pay its own boot (`reuseExistingServer: true` never fires on a fresh runner; each self-started server dies with its process) ≈ 3× cost; and the script itself (`npm.cmd`) is unrunnable on ubuntu — the job calls `npx playwright test` directly, the pattern every existing job uses. The jest halves are **not** re-run in this job: they already execute in the unit shards on every PR (Context); duplicating them buys nothing and costs minutes.
3. **Blocking via the aggregator, ratcheted**: add `seam-anchors` to `lint-and-test`'s `needs` array (17 → 18 entries). This is the repo's own spec'd mechanism (`e2e-testing` "PR-Gated Coverage Floor": membership in the aggregator's required `needs` set ⇒ failing the branch-protection required check). Zero branch-protection edits, zero renames, 29 existing check names byte-stable; total 29 → 30. The membership is then **pinned mechanically**: `scripts/qc/validate-openspec-ci-quality.mjs:32` already maintains `requiredAggregatorNeeds` (the 17 current ids, checked as a required subset of the live workflow's `needs` on every `qc:openspec-ci:validate` run — verified in-session, `aggregatorNeeds=17/17 protectedContexts=4 errors=0`); appending `'seam-anchors'` there makes a future aggregator refactor that drops the entry fail the existing gate instead of silently un-gating the anchors. **Aggregator hardening — the soundness half, required and name-stable**: membership alone is NOT blocking authority, because the aggregator's failure path is broken today (Context: a failed fan-out job leaves `Lint and Test` concluded `skipped`, and a condition-skipped required check counts as passing). This group therefore hardens the aggregator without touching its identity: `if: always()` on `lint-and-test` plus replacing the echo step with an explicit result assertion over `${{ toJSON(needs) }}` that exits non-zero — printing every offending job id — unless every needed job's `result` is `success`. Strictness is safe: no job in the workflow carries a job-level `if:` (verified — per-step guards only), so `skipped`/`cancelled`/`failure` results are all defects. Job id, `name: Lint and Test`, and the flow-style `needs: [...]` array shape stay byte-identical — no rename, no repurpose (the hardening makes the job's own comment, "Passes only if every fan-out job passed", true on the red path for the first time), and `parseAggregatorNeeds` in `validate-openspec-ci-quality.mjs` (which regex-reads that flow-style array) keeps parsing it. The hardening closes the latent hole for all 17 pre-existing needs, not just the new 18th.
4. **Staged landing: advisory burn-in → harden → wire → red-path proof**: the job lands **without** the aggregator `needs` entry first; after ≥3 observed green runs on real code-touching PRs (with wall-clock recorded against D3's budget), a follow-up commit adds the one-line `needs` entry, riding the hardened aggregator (D2.3 lands before or with the wiring — wiring into the unhardened aggregator would wire a gate whose failure path is fictional). **Green means first-attempt green**: the job runs `--retries=0`, and each burn-in run's Playwright report must record zero `flaky`/retried results (structurally impossible with the flag in place — the report check exists to catch the flag being dropped). A flaky *required* gate is the worst deliverable this change could ship; the staging makes flake visible while the job is still advisory. (An anchor that flakes during burn-in is an anchor stability bug fixed under journey-qc's own 3×-green obligation — never retried-away, never sliced.) **The wired gate's blocking authority is then proven red-path, never assumed** — all burn-in evidence is green-path by construction: a **draft** PR carrying a deliberate single-anchor failure (the failing assertion lives only on the never-merged draft branch; main's anchors stay untouched) must show, at that head SHA, the seam job concluding `failure` on first attempt, `Lint and Test` concluding `failure` — ran and asserted, never `skipped` — with the hardened step naming `seam-anchors` as the non-success need, and the merge state reporting blocked (`gh pr view --json mergeable,mergeStateStatus` → `BLOCKED`). The draft is closed unmerged. If the merge box is NOT blocked, the one-line `needs` entry is reverted, the gate stays advisory, and delivery STOPs with the finding — the delta's "cannot merge" scenario is an obligation to demonstrate, not an assumption to inherit.
5. **The human-action reality, documented**: making any check independently *required* is a GitHub branch-protection setting (`required_status_checks.contexts`), editable only via Settings → Branches or `gh api --method PUT …/protection` by an admin — **not a repo file**. This design deliberately avoids needing it. If a future change promotes `Seam Anchors` (or any lane) to a named required context, that proposal must call out the manual admin step explicitly, with the rollback being restoration of the current 4-element contexts list (snapshot recorded in group-1 verification). `.github/BRANCH_PROTECTION.md` gains one sentence documenting that the seam gate rides the aggregator, so the doc stays truthful.

**Rejected alternatives**: extending the `E2E Smoke` job with the anchor specs (repurposes a stable job's scope and failure semantics; its name is not protection-pinned but it is the one Playwright job every engineer knows — scope creep there muddies triage and violates the sibling-not-extension rule W4/W5 established for their own lanes); a 5th required context `Seam Anchors` (needs the manual flip + rollback ceremony for zero marginal safety over aggregator membership); `--grep @smoke`-based selection (the anchors deliberately do not and must not carry `@smoke` — journey-qc's anchor requirement bans double-scheduling tags; and the tag's 12-of-28 embedding rate makes it unreliable selection); wiring `verify:qc:seam-anchors` verbatim (Windows-only `npm.cmd`, 3× boots).

### D3 — Quantified wall-clock budget: the seam job must fit the measured slack, and its first CI timings are the acceptance evidence

**Baseline (measured, Context):** PR wall 6m09s–7m41s; critical path ≈ 434s (Prepare Build 237s → Build Test/win 197s) of 461s observed total; E2E Smoke — the closest shape analog — 187–205s job wall when running (2m21s test step; single spec, cold webServer, cached browser).

**Seam job estimate (honestly labeled):** no CI timing exists for these three specs in any invocation shape — they have never run in CI (verified; the one load-bearing unmeasured quantity). Inputs: 3 specs / 5 tests / 1,196 spec lines; two of the five tests are double-load determinism comparisons (two cold navigations each); the roster-handoff test click-drives a full campaign→launch path. Single-invocation estimate: **~3–6 min job wall** (one boot ≈ E2E Smoke's fixed ~60–90s overhead + 5 route-mounted tests). Slack arithmetic: the job starts after `install-deps` (~40s) and runs parallel to the build chain; it extends the workflow only if job wall exceeds ≈ 434s − 40s ≈ **6.5 min**. Expected net effect on PR wall-clock: ~zero; worst-estimate: ~+1 min against the 6.5-min slack ceiling.

**Acceptance is measurement, not the estimate:** burn-in (D2.4) records the job's actual wall on ≥3 real code-touching PR runs *before* the blocking `needs` entry lands — first-attempt-green runs only (`--retries=0`; a run whose report records a retried/flaky result is a burn-in failure, not a slow success), and the deliberately-red draft-PR proof run (D2.4) is excluded from the budget sample. Recorded thresholds: job wall ≤ 6.5 min ⇒ wire the gate; 6.5–10 min ⇒ wire it only with the measured workflow-total regression recorded and a follow-up filed to investigate the slow anchor (an anchor slow enough to move the critical path is a stability/perf finding in its own right — investigated on the anchor side, never "fixed" by slicing the gate); > 10 min or any flake ⇒ STOP, do not wire, report (the change ships the job as advisory and the blocking step becomes a follow-up decision). Docs-only PRs: guard-skipped in seconds (the E2E Smoke skip path, verified at 4s).

### D4 — The subsystem facet lives on the journey catalog, validated closed-set, joined at query time; the qc-registry and lane scripts are deliberately untouched

**Choice**:

1. **Facet**: every entry in `docs/qc/mekstation-journey-scenarios.json` gains `subsystems: string[]` — members drawn from the closed six-tag vocabulary, no duplicates, field required on every entry. **Empty is legal only as an explicit authored `[]`** for journeys whose surface lies outside the campaign-subsystem taxonomy — concretely `mek-build` (construction; the vocabulary is gameplay-subsystem, not module, and forcing a bogus tag is worse than an honest empty). Proposed seed mapping (finalized against each journey's step list at implementation time): `character-build` → `[personnel, experience]`; `mek-build` → `[]`; `combat-1v1`/`combat-4v4` → `[combat]`; `contract-campaign` → `[navigation, economy, combat]`; `campaign-short` → `[navigation, economy, personnel]`; `campaign-long` → `[navigation, economy, personnel, maintenance, experience]`.
2. **Validation**: `journey-qc-core.mjs` gains `allowedSubsystems` (a Set beside `allowedModes`/`allowedTiers`, :51-52) and `validateJourneyCatalog` enforces presence + membership + uniqueness, naming journey id and offending value (the existing issue-format precedent). A jest guard in `scripts/__tests__/` asserts the `.mjs` Set equals the TS `FlowSubsystem` literal set (import the manifest from the test only — no script→e2e import at runtime), completing the three-way lock once W4's manifest guard lands: flow registry = pack manifest = journey facet.
3. **Query**: the QC graph query command gains `--subsystem=<tag>` — resolve the catalog entries whose facet carries the tag, then return their existing graph joins (journey nodes, commands, evidence, gaps). The facet is **joined at query time from the catalog**; `mekstation-qc-validation-graph.json` gains no duplicate field (single home; a copy in the graph JSON is a second place to rot). Unknown tag fails naming the six allowed values.
4. **Deliberately untouched**: `mekstation-qc-registry.json` (its 27 surfaces are an architectural axis — the verified `maintenance-code-health` vs gameplay-`maintenance` name collision is exactly the conflation a shared field would invite); `run-qc-lanes.mjs`/`select-qc-lanes.mjs` (their 9 filter keys filter that registry; with no subsystem field there, a `subsystem` key would be dead, and lane slicing runs through Playwright `--grep` anyway — the scripts cannot synthesize invocations, verified); `src/qc/gameplayUiFlowShell.json` (flows already bind `journeyId`; the facet is derivable through the catalog join — duplicating it onto flows creates a second drift surface for zero query power).

**Rejected**: a non-empty-always rule (breaks on `mek-build` and invites a lie); putting the facet on the qc-registry surfaces (wrong axis, double maintenance, and both lane scripts would need the same new key twice due to their verified duplication).

### D5 — Tags: native `test.describe(title, { tag })`, namespaced `@subsystem:<tag>` literals, bounded initial tenancy

**Choice**: audit specs carry subsystem tags via Playwright's native tag option (the `playtest-sp-smoke.spec.ts:287` precedent) with literals of the form `@subsystem:navigation` … `@subsystem:experience` — derived 1:1 from the closed vocabulary, guard-checked (a tag literal is `'@subsystem:' + FlowSubsystem`).

- **Why namespaced**: the config header documents a live legacy title-tag vocabulary including `@combat`, `@campaign` — a bare `@combat` lane grep would select every legacy-tagged combat spec ever written, not the subsystem-tagged set. `@subsystem:` prefixing makes lane selection exact and future-greppable.
- **Why native tags, not title-embedding**: grep-equivalent selection without polluting human-readable titles, and visibly distinct from the proven-broken docblock convention. **Docblock "tags" are banned as a selection mechanism** (12-of-28 embedding rate verified; all three subsystem stubs claim tags that select nothing) — the spec delta makes the ban explicit so new tags are never planted in comments again.
- **Initial tenancy (bounded)**: the four subsystem validation specs (`@subsystem:economy` on loans, `@subsystem:personnel` on hiring, `@subsystem:economy` on contracts, `@subsystem:experience` on xp-purchase). Pack parity specs join behind the W4 gate (D9) with a guard asserting each parity spec's tags equal its manifest entry's `subsystems`. The **anchors are never tagged** (their chromium-only contract bans double-scheduling tags today and any-project-selecting tags under W5's rewording; lane membership would also re-run them nightly per-lane for no signal — they have their own PR gate). The sweep/walkthrough/deep-play specs stay untagged in this change (they are cross-subsystem by construction).
- **Not project-selecting**: no Playwright project greps `@subsystem:*` — lanes are CLI `--grep` invocations against `--project=chromium`. This keeps the tags outside W5's "project-selecting tag" prohibition for chromium-contract specs by construction, under either topology.

### D7 — Honest reconciliation of "Subsystem Validation Coverage": a machine-validated ledger replaces the 19-spec fiction

**The wound**: a live SHALL demands 19 specs + a CI gate; 3 render-only specs exist and no gate exists anywhere. Two years of OpenSpec discipline on this repo say a false SHALL is worse than a narrow one.

**Choice** (a MODIFIED delta, option (a) of the verified reconciliation menu — fold the 19 into the six-tag facet as detail rows):

1. **`docs/qc/mekstation-subsystem-coverage.json`** (new, beside the other QC registries): one row per fine-grained subsystem — all 19 — each declaring `id`, `facet` (one of the six tags), `coveringSpecs` (paths), `status` ∈ `asserting` | `render-only` | `deferred`, and for `deferred` a `followUpRef` (the Wave-6.1 Track-B/C provenance or a successor). Statuses ship truthful: after this change's mini-journeys land — `hiring`, `loans`, `contract-market`, `XP/leveling` = `asserting`; the Track-B/C remainder = `deferred` with refs; rows whose 2026-05-20 audit rating has drifted (awards — components + `e2e/awards.spec.ts` now exist; unit market — route now exists) get re-verified statuses, not the stale ones.
2. **`scripts/qc/validate-subsystem-coverage.mjs`** + a `scripts/__tests__/` jest wrapper (the established validator pattern): fails on a missing/extra row vs the 19-id set, an invalid facet/status, a dangling `coveringSpecs` path, an `asserting`/`render-only` row whose named spec file lacks the row's `@subsystem:` tag literal, or a `deferred` row without a `followUpRef`. Because the wrapper lives in `scripts/__tests__/`, **it runs in every PR's unit shards** — the first real enforcement this requirement has ever had, delivered through an existing lane with zero new CI jobs.
3. **Regression discipline**: demoting a row's status (`asserting` → anything) requires the PR to name what it grandfathers (the sweep-quarantine precedent from W5). The guard cannot judge assertion depth — status is reviewed data; what it mechanizes is existence, linkage, tag presence, and completeness.
4. **The stale markdown matrix** (`playtest/phase-6/SUBSYSTEM_UI_AUDIT.md`) is demoted to a dated snapshot: a header note names the ledger as the living source and the snapshot date; its stale rows are not silently rewritten (audit documents are point-in-time evidence). A full re-audit is a recorded follow-up.

**Rejected**: leaving the requirement as-is with a "known gap" note (a permanently false SHALL rots trust in every neighboring SHALL); deleting the requirement (the per-subsystem coverage doctrine and the write-capable assertion bar are exactly what the mini-journeys deliver — the doctrine survives, the fictional counts go); building the 16 missing specs now (Wave-6.1 deferred them for real testid/UI-gap reasons; the capstone's scope is the ladder's CI payoff, and the audit that would re-scope them is itself stale — re-audit first, per the follow-up).

### D8 — Nightly sliced lanes: a tag-grep matrix whose lane list is guard-locked to ledger tenancy

**Choice**: one new job in `nightly-validation.yml` — `subsystem-lanes`, `strategy.matrix.subsystem` over the tenant tags, `name: Subsystem Lane (${{ matrix.subsystem }})` — each leg: the E2E-Smoke-pattern setup then `npx playwright test --grep "@subsystem:${{ matrix.subsystem }}" --project=chromium`. `needs: install-deps`; the job id appended to `notify-on-failure.needs` (the W4 recapture-job precedent — failures land on the existing tracking-issue convention, no new notification machinery). `fail-fast: false` so one red lane never cancels the others (per-lane signal is the entire point).

- **Lane list = tags with tenants, mechanically**: Playwright errors out on a grep matching zero tests, so a lane for an untenanted tag is red-by-construction. The lane list ships as the matrix literal in the workflow, and D7's validator cross-checks it: every matrix tag must have ≥1 ledger row with a tagged covering spec, and every tag with tagged `asserting`/`render-only` rows must have a lane. Initial list: `economy`, `personnel`, `experience` (the mini-journeys' tags — `navigation`/`combat`/`maintenance` join when the W4-gated tagging task gives them tenants, D9). The guard reads the workflow YAML (the Long-Campaign-QC-Protection-Gate precedent of validating wiring) — lane drift fails the PR unit lane, not a 3 a.m. cron.
- **Triage against a red blob, stated plainly**: nightly is 10/10 red on 8 pre-existing failures. The lanes' value is precisely that they are the first per-subsystem red/green signal not buried in the 707-test sweep — a red `Subsystem Lane (economy)` beside five green lanes localizes in one glance what currently takes a 48-minute log read. The lanes re-run tagged subsets `e2e-full` already runs — bounded duplication (the tagged population, minutes per lane), accepted for the signal. Lanes are **never** gates; `e2e-full` remains the authoritative sweep; fixing the 8 reds is a named follow-up, not this change.
- **Rejected**: sharding `e2e-full` by subsystem instead (most of the 707 tests carry no subsystem tag — a facet shard would orphan them or mislabel them); a repo script looping tags in one job (single check run = no per-lane UI signal, and it rebuilds exactly the matrix the workflow gives for free); lanes in `pr-checks.yml` (council put slicing in nightly; the PR budget is spent on the anchors).

## Risks

- **[R1] Formatter/validator quote race** (repo memory): after any spec-delta edit — formatter, then `npx openspec validate add-subsystem-lanes-and-ci --strict`, then any touched validator's jest wrapper.
- **[R2] Ledger race at ship time**: W4's and W5's entries may archive off `origin/main` mid-flight → rebase; land the ledger with only then-active entries; `qc:openspec-ci:validate` is the hard gate (the W4/W5 R5 precedent).
- **[R3] Seam job timing is the one unmeasured quantity** — never run in CI. → D3's burn-in measurement with recorded thresholds (≤6.5 min wire / ≤10 min wire-with-recorded-regression + anchor-perf follow-up / >10 min or flake STOP-advisory). The estimate is an input; the measurement is the acceptance.
- **[R4] A flaky anchor becomes a flaky required gate** — the single worst outcome, and the CI config actively invites it: `playwright.config.ts:46` sets `retries: process.env.CI ? 2 : 0`, so on GitHub runners (`CI=true`) the job would inherit **2 retries** — a flaky anchor retry-passes green, burn-in records clean evidence, and a flaky gate gets wired while the spec promises flake is never retried away. → The invocation carries an explicit `--retries=0` (never assumed "inherited" — verified false for CI); burn-in acceptance requires zero `flaky`/retried results in each run's report (catches the flag being dropped); flake during burn-in STOPs the blocking step and files the anchor stability bug (journey-qc's 3×-green obligation owns the fix). The mini-journeys' 3×-green acceptance is local (`CI` unset ⇒ genuinely 0 retries) and pins `--retries=0` explicitly anyway; their nightly/lane executions inherit the CI default `retries: 2` — accepted for non-gating triage lanes (a retry-pass surfaces as `flaky` in the report, which is honest for evidence; only the gate must be retry-free).
- **[R10] The skipped-aggregator hole: `needs` membership alone does not block on failure** — verified live (run 29059132882: `Unit Tests (4/6)` `failure` ⇒ `Lint and Test` `skipped`), and GitHub's documented semantics treat a condition-skipped required check as passing — without intervention the delta's blocking scenario would be fiction. → D2.3's name-stable hardening (`if: always()` + every-need `result == 'success'` assertion) plus D2.4's red-path draft-PR proof (forced anchor failure ⇒ aggregator concluded `failure`, never `skipped` ⇒ merge state `BLOCKED`); an unblocked merge box reverts the wiring and STOPs delivery. The hardening also closes the hole for the 17 pre-existing needs — a strict improvement to the existing gate, landed name-stably.
- **[R5] Same-file merge race on the workflows**: W4's implementation adds a nightly job + `needs` append too (`scenario-pack-recapture`). → Both changes are appends; whichever lands second rebases; the notify `needs` array unions trivially. `pr-checks.yml` is touched by no other in-flight change (verified on this tree).
- **[R6] Tag/ledger/lane drift** (a tag renamed, a spec moved, a lane orphaned): → one validator owns the whole linkage (D7.2 + D8's lane cross-check) and runs per-PR in the unit shards; every failure names the row/tag/lane.
- **[R7] The XP journey's spendable-XP precondition may not exist front-door** → D6's ordered decision rule with a STOP-and-report terminal (a testability gap gets filed; the journey is never faked via store writes).
- **[R8] Lane noise on a chronically red nightly** → lanes are additive signal (per-subsystem localization), `fail-fast: false`, notify-issue already fires nightly; the 8 pre-existing reds are a named follow-up — this change must not be judged by their persistence.
- **[R9] The docs-only guard drifts between the two Playwright jobs** (e2e-smoke edits its filter, seam job forgets) → the seam job copies the exact `if:` expressions; group-1 re-verification records both; a divergence is caught by the check-name/no-op proofs in the workflow group's acceptance.

## Migration Plan

Additive test/QC/workflow surface; zero production files. Rollback = revert the workflow edits (job blocks, the aggregator hardening — the `if: always()` line + the assertion step, independently revertible — and the `needs` entries), the package-script lines, the catalog/ledger/validator additions, the spec tags, and the change's ledger entry; the extended subsystem specs' new assertions revert with their files. The staged gate landing gives an extra safety valve: the blocking `needs` entry is one line, revertible independently of the job and of the hardening (reverting the hardening alone, however, re-opens the pre-existing skipped-aggregator hole — it should outlive the gate). Implementation starts after the W5 implementation merges (ladder sequencing); the W4- and W5-gated groups carry their own STOP checks (D9).
