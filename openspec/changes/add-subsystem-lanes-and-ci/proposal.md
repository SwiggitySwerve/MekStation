# Proposal: add-subsystem-lanes-and-ci

## Why

**W6 of the Seam-First Ladder — the capstone** (council decision `openspec/council-decisions/2026-07-09-ui-audit-testability-architecture.md`): the ladder's lower rungs built the trust primitives — W1 seeded determinism, W2 the three seam trust anchor journeys, W3 the headless invariant net — but nothing yet **cashes them in**. Verified on this base (`claude/subsystem-lanes-and-ci@6e0ed7868`):

- **The PR gate never runs the seam anchors.** `pr-checks.yml`'s only Playwright invocation is one hardcoded file (`npx playwright test e2e/tactical-map-visual-smoke.spec.ts --project=chromium`, line 521 — grep proves no other `playwright test` anywhere in the file). The three anchors pass today only inside nightly's full sweep — and that pass is **invisible**, because `nightly-validation.yml` has been red on all 10 of its last 10 scheduled runs (2026-07-01 → 2026-07-10) on 8 unrelated pre-existing spec failures. The council's per-PR trust contract ("PR gate = headless invariants + the three seam journeys", Phase 3 update 5) is half-delivered: the W3/W2-adjacent jest invariant suites DO run on every PR (ordinary members of the `unit` jest project, split across the 6 unit-test shards — verified by config read and a live 6.8s targeted run), but no contract pins them there, and the anchors' route-mounted halves run nowhere PR-blocking.
- **Per-subsystem triage does not exist.** The journey-qc registry has zero subsystem vocabulary (grep of the 552-line spec and of `journey-qc-core.mjs` returns nothing); the closed six-tag `FlowSubsystem` vocabulary (`e2e/flows/manifest.ts:25-31`) exists but selects nothing; `run-qc-lanes.mjs`/`select-qc-lanes.mjs` filter a different registry (`mekstation-qc-registry.json`, an architectural taxonomy whose `maintenance-code-health` surface collides in name only with the gameplay `maintenance` tag) and can only replay static command arrays — they cannot synthesize a Playwright slice. When nightly goes red, nothing localizes the failure to a subsystem.
- **The coverage holes drifted from the council sketch (verified facts override it).** The council row says take-loan, hire, and XP/ability purchase are "currently zero e2e." Reality: `e2e/playtest-subsystem-loans.spec.ts` and `e2e/playtest-subsystem-hiring.spec.ts` exist but assert **render-visibility only** — the hiring spec's own docblock claims a testid-targeted hire click and a store-mutation assertion its 37-line body never performs; the loans spec self-documents its deferral. `e2e/playtest-subsystem-contracts.spec.ts` is the same class. Only XP/ability purchase is literally zero-e2e — its UI testids (`add-ability-btn`, `owned-spa-{id}`, `PilotAbilitiesPanel.tsx:306/109`) and store math (`purchaseSPA`, jest-proven in `spaAcquisition.test.ts` + `PilotService.spaEditor.test.ts`) both exist; the seam between them has never been crossed by a test.
- **A stale SHALL is lying in the spec ledger.** `openspec/specs/e2e-testing/spec.md`'s "Subsystem Validation Coverage" requirement (from the archived 2026-05-20 Wave 6.1 change) demands "exactly 19 specs at `e2e/playtest-subsystem-*.spec.ts`" and claims "the CI gate MUST fail if any spec is missing." Verified: 3 of 19 specs exist, and **no CI gate exists anywhere** — zero references to `subsystem` or `playtest-subsystem` in any workflow file, package script, or `scripts/`. The companion audit matrix (`playtest/phase-6/SUBSYSTEM_UI_AUDIT.md`) has also drifted stale (it rates Awards BLOCKED/"no renderer" while `src/components/award/` now ships seven components plus `e2e/awards.spec.ts`, and rates Unit Market "no route" while `src/pages/gameplay/campaigns/[id]/acquisitions.tsx` exists). W6 must reconcile this requirement honestly — silence would build a third subsystem facet next to an already-broken second one.

**Dependency reality:** the W4 (`add-scenario-packs`) implementation is in flight on a parallel branch (0/30 tasks on this tree; `e2e/scenario-packs/`, the loaders, and all six pilot packs do not exist here); the W5 (`add-viewport-layout-sweep`) implementation has not started (the 6-project matrix is still live; W5's contract deletes the three responsive projects + `smoke` and installs tag-then-scope). Per the ladder's strict sequencing, W6 implementation starts only after the W5 implementation merges — and every W6 task consuming a W4 or W5 artifact additionally carries a mechanical STOP-on-absence gate (the W2-task-4.0 pattern both W4 and W5 adopted; design D9).

**CI-safety is the capstone's riskiest surface, and the verified branch-protection reality shapes the whole design:** branch protection on `main` requires only **4 named contexts** (`Lint and Test`, `Build Test / win|mac|linux` — live `gh api` snapshot matching `.github/BRANCH_PROTECTION.md`), and `Lint and Test` is an aggregator job whose 17-entry `needs` array (`pr-checks.yml:573-599`) transitively gates the other 25 of today's 29 check runs. Making a check required is a GitHub settings action, **not a repo file** — so this change never needs it: the new seam-anchor job becomes blocking through aggregator-`needs` membership, the exact mechanism the `e2e-testing` "PR-Gated Coverage Floor" requirement already contracts for the coverage job. One verified wrinkle makes hardening mandatory: the aggregator's failure path is unsound today — `lint-and-test` has no `if:` guard, so a failed fan-out job leaves it **skipped** rather than failed (verified live: run 29059132882 — `Unit Tests (4/6)` concluded `failure`, `Lint and Test` concluded `skipped`), and GitHub treats a condition-skipped required check as passing for merge purposes. The change closes this name-stably (`if: always()` + an explicit every-need-succeeded assertion on the aggregator — a latent hole predating this change, fixed for all 17 existing needs too) and proves the blocking chain red-path on a never-merged draft PR before the gate is claimed delivered (design D2, R10). All 29 existing check names stay byte-stable; branch protection is untouched (design D2).

## What Changes

- **Subsystem facet on the journey catalog** (`docs/qc/mekstation-journey-scenarios.json` + `scripts/qc/journey-qc-core.mjs`): every one of the seven required journey entries gains a `subsystems` array drawn from the closed six-tag `FlowSubsystem` vocabulary (`navigation`/`combat`/`economy`/`maintenance`/`personnel`/`experience` — reused verbatim, never a new list: W4's pack manifest already binds the same literals with a guard test, and a third vocabulary would split the taxonomy the moment W4 lands). `validateJourneyCatalog` enforces membership against a closed set (the `allowedModes`/`allowedTiers` precedent, `journey-qc-core.mjs:51-52`); a jest guard asserts the `.mjs` set equals the TS `FlowSubsystem` literals; the QC graph query command gains a `--subsystem=<tag>` mode that joins the catalog facet at query time (the catalog stays the single home of the facet — no duplicate field in the graph JSON, and no edit to `mekstation-qc-registry.json`, whose taxonomy is a different axis; design D4).
- **Subsystem tags on audit specs, via Playwright's native tag API** (`test.describe(title, { tag: ['@subsystem:<tag>'] })` — the `playtest-sp-smoke.spec.ts:287` precedent): namespaced `@subsystem:` literals so lanes never collide with the legacy `@combat`/`@campaign` title-tag vocabulary the Playwright config header documents. Docblock-comment "tags" are proven non-functional (only 12 of 28 `@smoke`-claiming files actually embed the tag anywhere `--grep` can see) and are banned as a selection mechanism. Initial tenants: the four subsystem validation specs; pack parity specs join behind the W4 gate with a manifest↔tag equality guard. Anchors are never tagged (design D5).
- **Four write-flow mini-journeys closing the verified holes** (design D6): extend `playtest-subsystem-loans.spec.ts` (drive `loan-submit` → assert loan row, balance credited by principal, `LoanDisbursement` transaction), `playtest-subsystem-hiring.spec.ts` (click `candidate-hire-{id}` → assert roster +1, hire-cost ledger debit — using its existing `seedHiringHall` **pre-hire** state; the W4 personnel pack's `roster-updated` genesis is post-hire, verified wrong for this journey and deliberately not consumed), `playtest-subsystem-contracts.spec.ts` (drive the accept action — the fourth hole, same render-only-stub class, already named write-capable by the spec), and a new `e2e/playtest-subsystem-xp-purchase.spec.ts` (front-door pilot with spendable XP → `add-ability-btn` → assert `owned-spa-{id}` + XP decremented by exactly the ability's cost, read through the exposed `window.__ZUSTAND_STORES__` surface via `e2e/helpers/store.ts`). Mini-journeys run nightly + in their subsystem lanes + locally; they are **not** PR-gated (the council's PR gate is invariants + anchors, nothing else).
- **The per-PR seam-anchor gate** (design D2/D3): one new `pr-checks.yml` job — `Seam Anchors` — running all three anchor specs in a **single** Playwright invocation (`--project=chromium --retries=0` — the config's CI default is `retries: 2` (`playwright.config.ts:46`), and a gate must never retry a flaky anchor to green; one webServer boot — the 3-invocation shape of the local `verify:qc:seam-anchors` script would triple the boot cost, and that script's `npm.cmd` form cannot run on ubuntu runners at all), mirroring the E2E Smoke job's shape exactly (needs `[detect-changes, install-deps]`, per-step docs-only guards, browser cache, failure report upload). It lands **non-blocking first, first-attempt-green burn-in, then** joins the `lint-and-test` aggregator's `needs` (17→18) — riding a **hardened** aggregator (`if: always()` + per-need result assertion, name-stable; without it a failed fan-out job skips the aggregator and the skipped required check counts as passing — the verified R10 hole) — becoming transitively required with zero branch-protection edits and zero renames, and the wired gate's blocking authority is then **proven red-path**: a deliberately failed anchor on a never-merged draft PR must show the aggregator concluding failure and the merge box blocked, or the wiring is reverted. Check count: 29 → 30; the 29 existing names byte-stable (task-level before/after diff proof). The headless-invariant half of the council's gate is already delivered by the unit shards; the spec delta pins it so it can never be silently excluded.
- **Nightly sliced subsystem lanes** (design D8): a matrixed `Subsystem Lane (<tag>)` job in `nightly-validation.yml` (`needs: install-deps`, appended to `notify-on-failure.needs` — the W4 recapture-job precedent), each leg `npx playwright test --grep "@subsystem:<tag>" --project=chromium`. Lane list = exactly the tags with tagged tenants, guard-checked against the coverage ledger (an empty grep is a Playwright hard error — lanes for untenanted tags are forbidden until tenants exist). Lanes are triage granularity against the verified 10/10-red nightly blob — never gates. The full chromium suite remains nightly's authoritative sweep.
- **Honest reconciliation of the stale coverage SHALL** (design D7): a MODIFIED `e2e-testing` "Subsystem Validation Coverage" delta replaces the false "exactly 19 specs + CI gate" claims with a **machine-validated coverage ledger** (`docs/qc/mekstation-subsystem-coverage.json`): all 19 fine-grained subsystems mapped to their six-tag facet, their covering spec (or explicit `deferred` status with the Wave-6.1 Track-B/C provenance), and their coverage depth (`asserting` | `render-only` | `deferred`). Its validator runs in the PR unit lane via a `scripts/__tests__/` jest wrapper (the established journey-qc validator pattern) — the first real, existing enforcement this requirement has ever had. The stale markdown audit matrix is demoted to a dated snapshot with a pointer.
- **Local lanes**: a parameterized `qc:subsystem-lane` package script (grep runner over `--project=chromium --workers=1`) plus validator wiring; nothing joins `verify:qc`'s chain.
- **Ledger**: `openspec/active-change-ledger.json` gains this change's entry (ships with this declarative PR, per the W1–W5 precedent).

## Capabilities

### Modified Capabilities

- `journey-qc`: MODIFIED — Required Journey Scenario Catalog (the `subsystems` facet joins the per-entry contract); QC Graph Query Command (the `--subsystem` query mode). ADDED — Seam Anchor Pull-Request Gate (the anchors' CI authority: single-invocation retry-free job, aggregator-needs membership on a hardened always-run aggregator, name stability, first-attempt-green burn-in before blocking, red-path-proven blocking authority, the pinned unit-lane invariants). journey-qc owns the anchors and the journey catalog; the gate requirement **cites** the anchors' owning requirement rather than modifying it — deliberately, because W5's unmerged delta already carries a MODIFIED "Seam Trust Anchor Journeys" and two unmerged deltas to one requirement would clobber each other at archive time (design D1).
- `e2e-testing`: MODIFIED — Subsystem Validation Coverage (the honest ledger contract replacing the 19-spec fiction; the write-capable assertion bar kept verbatim and finally delivered for hiring/loans/contract-market/XP). ADDED — Subsystem Tag Taxonomy (native tag API, namespaced literals, closed vocabulary, docblock ban); Nightly Subsystem Audit Lanes. No collision with W5's unmerged e2e-testing deltas (it MODIFIES only Deep-Play Playwright Project Scoping and ADDs three differently-named requirements; design D1).

Not chosen as homes: `flow-audit-routines` is doctrinally disqualified for anything here that gates or asserts (its own "Flow Audits Are Evidence, Never CI Gates" requirement — and this change leaves flow-audit evidence-only, untouched); `scenario-packs` (W4, unmerged) owns pack manifests and parity specs — W6 adds tags to those specs behind a gate but deltas nothing there; no new capability (the council ladder marks only W3/W4 "(NEW spec)").

## Impact

- **New/modified test + QC code** (implementation phase): `docs/qc/mekstation-journey-scenarios.json` (facet data), `scripts/qc/journey-qc-core.mjs` (facet validation + query mode), `docs/qc/mekstation-subsystem-coverage.json` + `scripts/qc/validate-subsystem-coverage.mjs` + `scripts/__tests__/` wrappers/guards (new), the three extended + one new `playtest-subsystem-*` specs, `.github/workflows/pr-checks.yml` (one job + one `needs` entry + the name-stable `lint-and-test` hardening: `if: always()` + result-assertion step), `.github/workflows/nightly-validation.yml` (one matrix job + one `needs` append), `package.json` (lane scripts), `playtest/phase-6/SUBSYSTEM_UI_AUDIT.md` (snapshot header), one ledger entry.
- **Modified production code: none.** No `src/` behavior changes — `src/lib/e2e/storeExposure.ts`, the testid surfaces, and `purchaseSPA` are consumed exactly as shipped. This is the ladder's third zero-production-touch wave.
- **CI surface**: PR check runs 29 → 30 (`Seam Anchors`); all 29 existing names unchanged; branch protection untouched (4 contexts, verified live). Per-PR wall-clock: expected ~zero net (the new job runs parallel, off the 434s Prepare-Build→Build-Test/win critical path; quantified budget + measured-on-real-runs acceptance in design D3). Docs-only PRs stay fast (the job no-ops in seconds under the detect-changes guard). Nightly gains one matrix job of small tagged subsets against a 180-minute timeout.
- **Ordering**: implementation starts after the W5 implementation merges (ladder sequencing). Groups 2–6 depend only on merged surface; the pack-tagging/pack-binding group gates on W4 impl, the topology-dependent proofs and the mobile-touch tenancy review gate on W5 impl (mechanical STOP checks, design D9).
- **Ledger reconciliation at ship time**: this worktree's `active-change-ledger.json` lists `add-scenario-packs` and `add-viewport-layout-sweep` as in-progress; either may archive off `origin/main` mid-flight. The PR rebases and lands the ledger with only the then-active entries (`qc:openspec-ci:validate` is the hard gate — the W4/W5 R5 precedent).
- **Risk**: the riskiest edit is the PR-gate wiring; it is fenced by the burn-in staging, the check-name diff proofs, the name-stable aggregator hardening + red-path draft-PR proof (a merge box that fails to block reverts the wiring), and the explicit no-branch-protection-edit rule (design D2/D3, risks R3/R4/R10). Everything else is additive test/QC surface.

## Non-goals

- **No branch-protection changes.** The required-context list stays exactly its current 4 names. The flip that would make any check independently required is a **manual GitHub settings/API action by a human admin, not a repo file** — this change documents that reality (with the rollback: restore the 4-element `required_status_checks.contexts` list) and deliberately never performs or needs it. Any future promotion of the seam job (or any lane) to a named required context is its own explicit decision with that human step called out.
- **No existing check renamed or repurposed.** `E2E Smoke` keeps its exact name, scope, and single hardcoded spec; the seam job is a sibling, not an extension (the branch-protection-names gotcha: names are the protection currency).
- **Mini-journeys, packs, lanes, and flow audits gate nothing.** The PR gate gains exactly the three seam anchors (per the council). Pack parity specs stay non-gating per `scenario-packs`' own gate rule; flow-audit routines stay evidence-only per their own requirement; nightly lanes are triage. No `verify:qc` chain additions.
- **No new harnesses, no new runners, no new Playwright projects.** Lanes are `--grep` invocations against the chromium project — adding per-subsystem projects would re-introduce the exact multiplication anti-pattern W5 deletes. No edits to `run-qc-lanes.mjs`/`select-qc-lanes.mjs` filter keys and no subsystem field on `mekstation-qc-registry.json` (different taxonomy; the `maintenance-code-health` name collision makes conflation concrete).
- **No vocabulary growth.** The six-tag set is closed; the 19 fine-grained subsystem ids live as ledger rows mapped onto it, not as new tags. Growing either vocabulary requires spec deltas to their owning requirements (`flow-audit-routines` Flow Registry for the tags).
- **No W4/W5 implementation scope, and no edits to their change folders.** The W4 gate group tags pack parity specs only after the W4 implementation lands (additive title tags + a guard — never manifest/loader/payload edits); the W5-deferred mobile-touch tenancy review is a recorded verdict + follow-up recommendation, never a deletion (deleting the project would require a delta to W5's still-unmerged topology requirement).
- **No anchor spec edits.** The anchors run as shipped; a slow or flaky anchor discovered by the gate is an anchor stability bug fixed under journey-qc's own 3×-green obligation — never sliced, softened, or retried-away from the gate side (the gate invocation carries `--retries=0` explicitly, overriding the CI-default `retries: 2`). The red-path proof's deliberate anchor failure lives only on a closed, never-merged draft branch — main's anchors stay untouched.
- **No repo-wide tag retrofit and no nightly-red remediation.** The ~16 specs with non-functional docblock-only tags are recorded as a follow-up (only files this change already touches get real tags); the 8 pre-existing failing nightly specs are a named, separate remediation candidate — this change makes their triage legible (lanes), not green.
- **No golden traces, no store injection, no `test.skip`-on-missing-data.** Mini-journeys seed through the blessed seeder/front-door tier and assert unconditionally (the e2e-testing conditional-skip prohibition binds them).
